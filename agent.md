Ты — senior Python architect и product analyst. Нужно спроектировать программу учета товаров для склада Oriflame.

Цель: создать приложение, которое работает:

1. Локально без интернета.

2. Онлайн через браузер.

3. С синхронизацией между несколькими операторами в разных странах через интернет-сервер.

Опиши оптимальную архитектуру, базу данных, логику синхронизации, интерфейс и этапы разработки MVP.

Функции программы:

1. Складской учет товаров
- Код товара.

- Название товара.

- Категория товара.

- Количество на складе.

- Себестоимость на момент поступления.

- Продажная цена.

- Текущая цена в каталоге.

- Не все поля обязательны.

- Заранее подготовленный справочник: код товара → название товара.

- Возможность редактирования товара.
2. Поступление товара
- Добавление товара на склад по коду.

- Автоподстановка названия из справочника.

- Указание количества.

- Указание себестоимости.

- Указание текущей каталожной цены.

- Указание продажной цены.

- Сохранение истории всех изменений цен.
3. Продажа товара
- Указание кода товара.

- Автоподстановка названия.

- Указание количества.

- Продажная цена может отличаться от стандартной.

- Покупатель: имя, фамилия, номер консультанта — необязательные поля.

- После продажи количество на складе уменьшается.

- Продажа сохраняется в истории.
4. Другие операции
- Списание товара.

- Возврат товара.

- Корректировка остатков.

- Редактирование карточки товара.

- История всех действий оператора.
5. Покупатели
- Профиль покупателя.

- История покупок.

- Что покупал, когда и по какой цене.

- Периодичность покупок.

- Напоминание, что у клиента может заканчиваться товар.

- При поступлении товара показывать список клиентов, которым этот товар может быть интересен на основе истории продаж.
6. Отчеты
- За день.

- За неделю.

- За месяц.

- За произвольный период.

- Продажи.

- Прибыль.

- Остатки.

- Списания.

- Популярные товары.

- Активные покупатели.

- Товары, которые давно не продавались.

- Товары с низким остатком.
7. Пользователи и роли
- Администратор.

- Оператор.

- Просмотр отчетов.

- Учет того, кто и когда сделал операцию.

- Возможность работы нескольких операторов из разных точек мира.
8. Синхронизация
- Локальная работа без интернета.

- Каждая точка может работать автономно.

- После появления интернета данные синхронизируются с центральным сервером.

- Нужно предложить надежную схему разрешения конфликтов.

- Нельзя терять продажи, поступления и изменения.

- Желательно использовать журнал операций/event log.
9. Интерфейс
- Максимально простой для оператора.

- Быстрый поиск по коду или названию.

- Минимум кликов.

- Формы: поступление, продажа, списание, покупатели, отчеты.

- Подсказки и автозаполнение.

- Защита от ошибок: предупреждение при продаже товара, которого нет на складе.
10. Технические пожелания
- Python.

- Локальная база данных, например SQLite.

- Серверная база данных, например PostgreSQL.

- Веб-интерфейс через браузер.

- Желательно предложить стек: FastAPI/Django, SQLAlchemy, SQLite/PostgreSQL, React/Vue/HTMX или другой простой вариант.

- Объяснить, какой стек лучше для MVP и почему.

- Дать структуру проекта.

- Дать пример схемы базы данных.

- Дать пример основных API endpoint’ов.

- Дать план разработки по этапам.
