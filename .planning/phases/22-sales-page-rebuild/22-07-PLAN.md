**Why this is a real fix and not a no-op:** htmx auto-includes the enclosing form's inputs ONLY on
non-GET requests [htmx.org/docs/#parameters]. The card tap is a GET with only `hx-vals`, so it sends
`batch_id`/`code`/`row` and nothing else. For the sale wizard, `pick_url` is `/m/sales/step/batch` and
the swap target `#batch-wrap` is precisely where the four `*_acc[]` hidden inputs live — so the
endpoint reads them as empty and the re-rendered `#batch-wrap` comes back with NO accumulator inputs
at all. A multi-line mobile basket loses its earlier lines when the operator re-taps a batch card on a
later line. This was verified empirically (`code_acc hidden input rendered: False`).

**Correct the record while you are here.** 22-RESEARCH.md Pitfall 3 and 22-UI-SPEC.md Interaction 10
both claim "every sibling wizard already passes `hx-include="closest form"`; the sale wizard is the
outlier." **That is wrong** and the cited lines are those wizards' «Назад»/«Далее» NAVIGATION buttons,
not their batch cards. `batch_card_picker.html` is a SHARED partial, so the card tap is missing
`hx-include` UNIFORMLY across all three consuming wizards. 22-CONTEXT.md's D-11 already carries this
correction — follow D-11, not the two downstream docs. Do not add a comment repeating the "outlier"
claim.

**What `closest form` resolves to differs per consumer — this is why the regression pass is
mandatory, not precautionary:**
- **Sale** (`sale_step_batch.html`): `#sale-wizard-form`, the persistent wizard shell. Sends the four
  `*_acc[]` arrays + `code`. The endpoint already declares all of them → **the actual fix**.
- **Write-off** (`writeoff_step_batch.html` / `writeoff_batch_wrap.html`): the ambient `#writeoff-form`.
  Newly sends `code` + `name`. `mobile_writeoff_step_batch_pick` declares only `batch_id`/`code`;
  FastAPI IGNORES undeclared query params, so `name` is silently dropped → **benign no-op**.

  step wrap. Newly sends `code` + `name` + `batch_qty`. `mobile_correction_batch_pick` declares only
  `batch_id`/`code` → **benign no-op**.

**One subtlety to verify rather than assume:** `hx-vals` sends `code` AND `hx-include` will now also
send the `code` hidden input, so `code` appears TWICE in the query string. FastAPI resolves a
duplicated scalar query param to the **LAST** value (probed against this app's FastAPI version on
2026-07-17). This is benign ONLY because both sources carry the same `{{ code }}` value in all three
wizards. Confirm that holds in each consuming template before shipping; if any consumer's hidden
