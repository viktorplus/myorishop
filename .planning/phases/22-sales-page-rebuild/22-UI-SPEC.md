> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

Built from `22-CONTEXT.md` (locked decisions D-01..D-12), `22-RESEARCH.md` (4 patterns, 7 pitfalls,

   margin-bottom: 4px }` (`style.css:55-59`) stacks the three options one per line with the xs gap —

2. **The live-total line** → `<p id="sale-total" class="num">`. `.num`

   is load-bearing here and not decorative: the figure re-renders on **every keystroke** (D-08), and
   proportional digits would make it jitter horizontally while the operator types. Right-alignment is
   the standard invoice idiom and sits the figure under the basket's money columns. The `<p>` inherits
   its top gap from the table's `margin-top: 16px` rhythm; no margin override is needed.

  search input, the 3 new-customer inputs, the picker buttons, the mobile equivalents
  (`style.css:160-167`, automatic, no new rule)

**Explicitly NOT accent:** «Добавить строку», «Удалить строку», «Убрать» (chip), «Удалить» (mobile
card), and every customer-picker row button stay `button.secondary` (white/`#222`/`#d9d9d9`,
`style.css:188-192`) — the shipped treatment, carried over.

**Destructive reserved for:**
- Validation error **copy** only, via the existing `<p class="error">` (`style.css:234-238`) and
  `.error-block` (`style.css:240-246`) classes — i.e. the D-10 guard message, `errors.basket`,
  `errors.form`, and the per-field new-customer errors.
- `button.danger` (`style.css:194-197`) is **not used** — this phase has no destructive action
  (see Copywriting).

**No new color role is introduced by Phase 22.** Four standing rules apply and are load-bearing here:

1. **Money is NEVER sign-colored, and the live total is NEVER color-coded.** The total renders in the
   default `#222` at every value. It is **not** green when "good", **not** red when large, and — most
   importantly — **not** `.error` red when the «итог неполный» marker is showing. An incomplete total
   is a *typing-in-progress* state, not a fault: the operator is mid-basket on essentially every
   keystroke, so painting it destructive-red would fire the app's strongest signal continuously and
   train the operator to ignore `#b91c1c` everywhere else. The marker is `.muted` (`#6b7280`) — see
   Copywriting and Interaction 5.
2. **The «Розница» label is `.muted` (`#6b7280`), not `.error`** (D-06). A walk-in sale is a valid,
   deliberate outcome, not missing data. This is exactly why D-06 rejects a blank/em-dash cell: `.muted`
   grey says "recorded, and the answer is 'nobody'", where a blank says "we lost it".
3. **No price-cue tint on the total.** The amber/blue `.price-below` / `.price-above` cue
   (`style.css:299-300`) is scoped to ДЦ/ПЦ price **inputs** by `price-cue.js` and must keep working
   untouched on the rebuilt form (Phase 18 regression, `22-RESEARCH.md` Validation map). It does not
   apply to `#sale-total`, and `sale-total.js` must never touch `classList` on a price input —
   that surface belongs to `price-cue.js`.
4. **No new selection tint.** The active radio's mode block gets **no** background highlight. The
   checked radio *is* the state indicator. `#e8effd` is the established search-match/selection token
   (`mark`, `.selected-batch`, `.name-search-list:hover`, `.mobile-card.selected`) and stays scoped to
   those; a fifth meaning would dilute it.

**Color is never the sole carrier of state in this phase** (WCAG 1.4.1). The incomplete total is
signalled by *text* («итог неполный»), not by tint. The selected mode is signalled by the radio's
native checked state, not by tint. The walk-in row is signalled by the *word* «Розница», not by grey
alone. (Contrast: `style.css:289-295` documents the price cue as a known 1.4.1 gap precisely because it
*is* color-only — do not repeat that pattern here.)

### Copy rationale (resolved this phase)

**1. Why the anon option reads «Без покупателя (розница)», not «Аноним».**
`22-CONTEXT.md` D-01 names the option «Аноним» but explicitly assigns *"Exact Russian wording/labels
for the 3 radio options"* to Claude's discretion. Three facts drive the deviation:
  - **D-06 locks the word «Розница»** as what the operator will read in the recent-sales list for
    exactly this outcome. Labelling the *cause* «Аноним» and the *effect* «Розница» gives one concept
    two names on one page, and the operator has to learn the mapping.
  - **«Без покупателя (розница)» already ships**, verbatim, on today's sale form
    (`sale_customer.html:38`) — the operator has read this exact phrase since Phase 4. It is a carry-
    over, not an invention.
  - **«Аноним» is inaccurate.** D-05 stores `customer_id = NULL` — no anonymous *person* is recorded.
    «Без покупателя» describes what actually happens.
The label breaks grammatical parallelism with «Новый»/«Существующий», which is accepted: under the
legend «Покупатель», the group reads *"Покупатель: Новый / Существующий / Без покупателя (розница)"*.
Clarity over symmetry. **The code token stays `anon`** (`_CUSTOMER_MODES = ("new", "existing", "anon")`)
— code is English, copy is Russian.

**2. Why «Предварительный итог», not «Итого».**
`22-CONTEXT.md` D-08/D-09 and `22-RESEARCH.md`'s Architectural Responsibility Map both state the total
is **advisory** — `register_sale` (`services/sales.py:282`) is the sole authority on the charged
amount, and it can still refuse the basket entirely (oversell, below-minimum, missing batch). A bare
«Итого» asserts finality the number does not have. One adjective carries the whole caveat, which is
cheaper than Phase 21's precedent of a separate mandatory caveat line (`finance_tiles.html:29-30`) —
appropriate here because, unlike a report tile, this line is under the operator's eye continuously and
a permanent second line of small print would be noise. The authoritative figure has its own distinct,
already-shipped copy («Продажа оформлена: … на сумму …», `sale_form.html:6`), so the two are never
confusable.

**3. Why the marker reads «итог неполный: проверьте кол-во и цену».**
D-09 gives the concept («итог неполный») but not the string. A bare «итог неполный» states a problem
with no path out. The suffix names the two fields that can cause it — which is exhaustive: the marker
fires only when `qtyToInt()` or `moneyToCents()` returns `null` for a row that has any of
code/qty/price filled (`22-RESEARCH.md` Pattern 2). Leading «·» joins it to the total line as one
sentence rather than orphaning it as a second block.

**4. Why the anon mode still renders a caption when SALE-06 says "no extra fields".**
SALE-06 and ROADMAP criterion 3 say anonymous *"shows no extra fields"* — a constraint on **inputs**,
and it is honored absolutely: the anon block renders **zero** `<input>`, `<select>`, or `<textarea>`
elements (the `customer_id` hidden input is forced empty and is not a field the operator sees or
fills). A muted caption is copy, not a field. It earns its place because it is the only surface that
tells the operator what «Розница» in the recent-sales list means — without it, D-06's label is an
unexplained word — and because an entirely blank block after clicking a radio reads as a loading
failure. Precedent: today's form already ships a `<p class="muted">Без покупателя (розница)</p>`
(`sale_customer.html:38`) for the same purpose.

### The customer header (SALE-03..06)

**1. The mode control is a real radio group with `fieldset` + `legend` semantics — mandatory, not
cosmetic.**
```jinja
<fieldset class="customer-mode"
          hx-get="/sales/customer-mode"
          hx-trigger="change"
          hx-include="#customer-header"
          hx-target="#customer-header"
          hx-swap="outerHTML"
          hx-sync="this:replace">
  <legend>Покупатель</legend>
  <label><input type="radio" name="customer_mode" value="new" {% if mode == "new" %}checked{% endif %}> Новый</label>
  <label><input type="radio" name="customer_mode" value="existing" {% if mode == "existing" %}checked{% endif %}> Существующий</label>
  <label><input type="radio" name="customer_mode" value="anon" {% if mode == "anon" %}checked{% endif %}> Без покупателя (розница)</label>
</fieldset>
```
Three radios sharing `name="customer_mode"` form a native group: arrow keys move between options, only
one is checked, and the `<legend>` supplies the group's accessible name so a screen reader announces
*"Покупатель, Новый, radio 1 of 3"* rather than three unrelated radios. **Wrapping each input in its
`<label>`** (rather than using `for=`/`id=`) is the shipped idiom (`receipt_batch_chooser.html:30-39`)
and makes the whole label text a click target — which matters on mobile. This is not a preference:
Phase 9's UAT found an **unlabeled batch chooser confusing, and the fix was exactly `fieldset`+`legend`**
— now shipped at `receipt_batch_chooser.html:20-21`. Re-deriving a bare `<div>` of radios here would
re-open a defect this project has already paid to close. **Do not** substitute `<select>` (SALE-03 and
ROADMAP criterion 3 say "radio"), and **do not** use `role="radiogroup"` on a `<div>` (native elements
first — nothing here needs ARIA).

**2. The radio's `hx-include="#customer-header"` is MANDATORY (D-03).** htmx sends **only the
triggering element's own value** on a GET; the enclosing form's inputs are auto-included **only on
non-GET** requests (`[CITED: htmx.org/docs/#parameters]`, `22-RESEARCH.md` Pattern 1). Without
`hx-include`, the server cannot know what the operator typed, re-renders an empty block, and **D-03
breaks silently — on the second switch, not the first.** `22-RESEARCH.md` names this *"the single most
likely defect in the phase."* Additionally: `#customer-header` sits **outside** `<form id="sale-form">`
(the basket form must not nest a form), so there is no enclosing form to fall back on even if the
request were a POST.

**3. Mode state survives switching via server-side hidden-input echo, not client-side caching (D-03).**
The server renders the **active** mode's visible fields and re-emits the **two inactive** modes' values
as hidden inputs inside `#customer-header`; the radio's `hx-include` ships them all back on the next
switch. State therefore round-trips indefinitely without ever being re-fetched or reset. This resolves
CONTEXT's *"which mode's HTML stays live in the DOM vs. gets re-fetched"* discretion item: **only the
active mode's HTML is live; the inactive modes persist as hidden values, not as hidden HTML.**
Rejected alternatives (`22-RESEARCH.md` §Alternatives Considered / §Don't Hand-Roll):
  - **`hx-preserve`** — documented as requiring the element in the response anyway (defeating D-01's
    per-mode fetch) and as **lossy for `<input type="text">`**, losing focus and caret position
    `[CITED: htmx.org/attributes/hx-preserve/]`.
  - **A JS/`sessionStorage` cache** — a fourth source of truth for customer state.
  - **Client-side show/hide of all 3 blocks** — trivially satisfies D-03 but is **locked out by D-01**.

**4. Every render path goes through one shared `_customer_context()` builder (D-12 — a verified bug).**
`sale_customer.html` must be rendered from a single context builder called by `sale_new_page`,
`sale_customer_mode`, `sale_customer_create`, **and all four `sale_create` branches**. The builder
resolves `selected` from `customer_id` server-side on **every** path. This is the fix for the
empirically-verified defect (`22-RESEARCH.md` Pitfall 7): today a 422/oversell re-render passes only
`{"customer_id": …}`, so `{% if selected %}` (`sale_customer.html:17`) is false, the chip hides, the
search box reappears — **while the hidden input still submits the id**. Verified: `customer_id survives:
True` / `chip text present: False`. **This is a silent mis-attribution surface: the UI says "no
customer", the database says otherwise.** The rebuilt header must not re-ship it. An id that no longer
resolves degrades to "nothing selected" rather than rendering a chip for a deleted customer (mirroring
`_build_lines`' batch rule). **Warning sign for review:** any `sale_create` branch whose context dict
still hand-writes `"customer_id": customer_id` without `selected` / `mode` / `form`.

display"* discretion item):

- **Directly under the table** is a literal requirement (SALE-02, ROADMAP criterion 2) — it goes
  **above** the existing «Если товара с таким кодом ещё нет…» hint (`sale_form.html:71`), not below.

  security control, not a style note: the browser's money math must be incapable of reaching the
  server. `register_sale` recomputes from `price[]` (`services/sales.py:282`). **Verify at review that
  no plan ever adds a `name=` to this element.**

  `.mobile-actions`, carrying `data-rows="mobile"` so the shared script targets `.mobile-card`

**12. The total is integer-cents string math with ZERO float arithmetic.** `CLAUDE.md` §What NOT to Use
bans floats for money. Note the deliberate divergence from the precedent: `price-cue.js:19` uses
`Math.round(parseFloat(...))` and gets away with it because it only **compares** and the server
re-stamps `data-ref-cents` authoritatively (`sale_row.html:41`). `sale-total.js` **displays a computed
sum** with no server-rendered fallback, so rounding drift would be visible. Parse string → integer
cents (`22-RESEARCH.md` Pattern 2), mirror `format_cents`' comma separator (`core.py:49`), and mirror
`to_cents`' ROUND_HALF_UP tie rule (`core.py:44`). Write via `.textContent` only.

**13. Three recompute triggers, not one — this is where the total silently goes stale.**
`22-RESEARCH.md` Pitfall 2 verifies two update paths that fire **no `input` event**:
  - `input` (delegated on `document`) — the typing path
  - `htmx:afterSettle` (on `document.body`) — the **422/oversell re-render**, which re-echoes the
    operator's values into fresh DOM with no `input` event
  - an explicit `window.recalcSaleTotal()` call appended to **both** delete buttons
    (`sale_row.html:49` desktop, `sale_basket.html:30` mobile) — a plain DOM `.remove()` fires neither
    an `input` **nor** an htmx event

  Guard the call (`if (window.recalcSaleTotal) …`) so the button still works if the script hasn't
  loaded. **Warning signs:** a total that only ever grows; a total unchanged by «Удалить строку».

**14. Row-counting mirrors the server's `non_blank_lines` filter** (`services/sales.py:90-94`): a row
counts if **any** of code/qty/price is non-blank. Using the same filter means «итог неполный» appears
exactly when the server would 422 that row — the marker predicts the server rather than disagreeing
with it. A fully blank row (the always-present empty last row) is skipped silently and must **not**
trigger the marker.

**15. The accept-set divergence is deliberate and bounded** (A2). `to_cents` accepts more than D-08's
"comma-decimal, no space-thousands" characterisation implies — `Decimal(str)` also takes exponents
(`1e3`), PEP-515 underscores (`1_000`), signs, and Unicode digit scripts (fullwidth `１２`, Arabic-Indic
`٣`) `[VERIFIED by execution, 22-RESEARCH.md Pattern 2]`. Byte-exact JS parity is not achievable in a
regex and **must not be attempted**. Accept the strict common subset; everything else falls into D-09's
marker bucket. A false «итог неполный» on `1e3` is harmless (advisory display; no operator types an
exponent into a price field); a **wrong number** would not be. Two consequences the executor must honor:
  - **Qty parity is exact** and must stay exact: server is `qty_text.isascii() and qty_text.isdigit()`
    then `> 0` (`services/sales.py:136-139`); JS `/^[0-9]+$/` + `> 0` is a byte-exact mirror. The
    `isascii()` guard exists to reject `'²'` and other non-ASCII digits that Python's `isdigit()` accepts.
    Write the JS class as `[0-9]`, not `\d` — **not** because `\d` would behave differently (in JavaScript
    `\d` is exactly `[0-9]`, with or without the `u` flag; the Unicode-wide behavior is Python's `re` on
    `str`, not JS), but because `[0-9]` states the ASCII-only contract on its face. A reader comparing this
    line against the server's `isascii()` guard should not have to recall a cross-language regex subtlety
    to see that the mirror holds.
  - **The money regex omits the sign on purpose** — `services/sales.py:157` rejects a negative sale
    price outright, so a negative is correctly "incomplete", not a negative subtotal.
