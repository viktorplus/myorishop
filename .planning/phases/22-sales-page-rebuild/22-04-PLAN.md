Create `app/static/sale-total.js` implementing the advisory running total.

Export four functions and one global:
- `moneyToCents(text)` → integer cents or `null`. Match against `/^(?:\d+(?:[.,]\d+)?|[.,]\d+)$/` on
  the trimmed string; on match, normalise the comma to a dot, split on the dot, and compute
  `whole * 100 + Number(frac.slice(0, 2))` from STRING slices, then apply the ROUND_HALF_UP tie rule
  by incrementing when the third fraction digit is `>= 5` (mirroring `core.py:44`, so `12,505` → 1251).
  Return `null` unless `Number.isSafeInteger(cents)`. **Zero float arithmetic anywhere** — CLAUDE.md
  §What NOT to Use bans floats for money. This is the deliberate divergence from the precedent:
  `price-cue.js:19` may use `Math.round(parseFloat(...))` because it only COMPARES and the server
  re-stamps `data-ref-cents` authoritatively; `sale-total.js` DISPLAYS a computed sum with no
  server-rendered fallback, so rounding drift would be visible. State this in the header comment.
  Omit the sign from the regex on purpose (`services/sales.py:157` rejects a negative sale price, so
  a negative is correctly "incomplete", not a negative subtotal).
- `qtyToInt(text)` → positive integer or `null`. Match `/^[0-9]+$/` then require
  `Number.isSafeInteger(n) && n > 0`. This is a byte-exact mirror of the server's
  `isascii() and isdigit()` + `> 0`. Write the class as `[0-9]`, NOT `\d` — **not** because `\d`
  would behave differently (in JavaScript `\d` IS exactly `[0-9]`, with or without the `u` flag; the
  Unicode-wide behavior is Python's `re` on `str`, not JS) but because `[0-9]` states the ASCII-only
  contract on its face for a reader comparing it against the server's `isascii()` guard. Do NOT ship
  a comment claiming `\d` would match Unicode digits in JS — that is false.
- `formatCents(cents)` → string. Mirror `core.py:49 format_cents` exactly: sign prefix, `Math.trunc`
  of the absolute value over 100, a COMMA separator, and a zero-padded 2-digit remainder.
- `recalcSaleTotal()` → reads `#sale-total`; early-return if absent (the script loads on every page
  via the shell, so most pages have no total). Select rows from `#basket-rows tr` by default, or from
  `#wizard-basket .mobile-card` when `#sale-total`'s `dataset.rows === "mobile"` (the mobile markup
  lands in 22-07; this script must already support it). Per row read
  `[name="code[]"], [name="code_acc[]"]`, `[name="qty[]"], [name="qty_acc[]"]`, and
  `[name="price[]"], [name="price_acc[]"]`; skip rows with no qty or price element (that skips
  `sale_row.html`'s batch-wrap `<tr>`). Mirror the server's `non_blank_lines` filter: a row counts
  only if ANY of code/qty/price is non-blank — so a fully blank row (the always-present empty last
  row) is skipped SILENTLY and must NOT trigger the marker. For counted rows, if `qtyToInt` or
  `moneyToCents` returns `null`, set an `incomplete` flag and skip that row's contribution (D-09);
  otherwise accumulate `cents += q * p` and `units += q`.
  Write results with `.textContent` ONLY — never `innerHTML` (the `customer_picker.html:1-12` house
  rule; also an XSS vector if product names ever entered the string). Toggle
  `#sale-total-warning`'s `hidden` from the `incomplete` flag.

Register THREE triggers — this is the single likeliest silent bug in SALE-02 (22-RESEARCH.md
Pitfall 2 verified both extra paths):
1. `document.addEventListener("input", ...)` filtered to `name` in `qty[]` / `price[]` / `code[]`
   (and their `_acc[]` twins) — the typing path.
2. `document.body.addEventListener("htmx:afterSettle", recalcSaleTotal)` — the 422/oversell

3. `window.recalcSaleTotal = recalcSaleTotal` — exposed so both delete buttons can call it; a plain
   DOM `.remove()` fires neither an input NOR an htmx event.

The file must never touch `classList` on a price input — that surface belongs to `price-cue.js`
(22-UI-SPEC.md Color rule 3).

Header comment must state: the requirement/decision ids (SALE-02 / D-08 / D-09), that the total is
ADVISORY and `register_sale` (`app/services/sales.py:282`) is the sole authority on the charged
amount, the deliberate no-float divergence from `price-cue.js`, and the bounded accept-set

rejected by both: `1 000`, `12abc`, ``, `inf`, `nan`, `12,5,0`) with the rationale that a false

No CDN, no import, no npm — a first-party vendored file (CLAUDE.md offline mandate).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client-computed total → server | MUST NOT exist — the control this plan structurally enforces |
| stored product name → JS string | would be an XSS vector if the total were built via `innerHTML` |
| browser → `POST /sales` | untrusted `qty[]` / `price[]`; the server re-parses them regardless of any client display |

| T-22-03 | Tampering | `#sale-total` client money math reaching persistence | mitigate | Structural: `#sale-total` carries no `name=` and contains no `<input>`/`<select>`/`<textarea>`, so the browser's figure cannot be submitted. `register_sale` (`services/sales.py:282`) recomputes `total_cents += qty * price_cents` from `price[]` on every write. Asserted by `test_sale_total_has_no_name_attribute` (22-02). |
| T-22-02 | Tampering (XSS) | `sale-total.js` DOM writes | mitigate | `.textContent` only — never `innerHTML`, never string-built HTML. Acceptance criteria assert `grep -c innerHTML` == 0. Follows the `customer_picker.html:1-12` house contract. |
| T-22-08 | Tampering | rounding drift making the displayed total disagree with the charged amount | mitigate | String→integer-cents parse with zero float arithmetic, mirroring `to_cents`' ROUND_HALF_UP tie rule.
