<objective>
Land the shared foundation and the only genuinely new server logic for WH-03: register the `"transfer"` operation type across the two runtime type-sets, then build `app/services/transfers.py` with `register_transfer()` (validate → confirm gate → create destination batch → two-row ledger write) and `recent_transfers()`. No migration is needed — `operations.type` is `String(20)` with no CHECK constraint and the destination batch is an ordinary `batches` row (RESEARCH Finding 8).

Purpose: A transfer moves stock between warehouses through the existing single write path (`record_operation`) so the ledger stays append-only, both quantity projections update in one transaction, and the destination inherits the source's frozen `price_cents` (this is *how* cost/price history is preserved). Getting the two type-set registrations right is the phase's one silent-failure trap (Pitfall 1).
Output: A batch-aware, fully unit-tested transfer service and the op-type registration that the Plan 02 route/UI wires to.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| transfer form/API → register_transfer | `code`, `qty`, `batch_id`, `dest_warehouse_id`, `confirm` are untrusted; the source batch may be foreign/exhausted, the dest warehouse inactive or the same as source |
| register_transfer → record_operation | the single sanctioned writer of operation rows + Product/Batch quantity caches |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Tampering / Elevation | source `batch_id` naming another product's batch | mitigate | register_transfer rejects `source is None or source.product_id != product.id` before any write; record_operation re-checks `batch.product_id == product_id` (ledger.py:97-98) |
| T-10-02 | Tampering | destination `warehouse_id` (inactive/deleted, or == source) | mitigate | re-validate `dest in active_warehouses` ids AND reject `dest == source.warehouse_id` with a RU error, server-side, before any write (Pitfall 4, D-02) |
| T-10-03 | Tampering | negative / non-integer / huge `qty` | mitigate | `qty_text.isascii() and qty_text.isdigit()` then `qty > 0`, never bare `int()` (writeoffs precedent) |
| T-10-04 | Repudiation / Tampering | ledger mutation to fake/undo a transfer | mitigate | append-only DB triggers (operations_no_update/operations_no_delete) untouched — transfers only INSERT via record_operation |
| T-10-05 | Tampering | double-submit racing two transfers of the same batch | accept | WAL single-writer + commit=False/one-commit; over-transfer past remaining is allowed-by-design (D-06 warn-but-allow), not a vulnerability |
| T-10-SC | Tampering | npm/pip/cargo installs | accept | zero external packages installed this phase (RESEARCH ## Package Legitimacy Audit — vacuous gate) |

No high-severity threat is open: every mitigation is inherited from the reused record_operation + writeoffs validation shape.
</threat_model>
