<objective>
Deliver LOT-06: a read-only expiry report at `/reports/expiry` listing every open batch (quantity > 0) that has a set expiry date, earliest first, with already-expired batches visually flagged. This is a pure read feature mirroring `/reports/stock` — one new read helper (`expiring_batches()`) beside `open_batches()`, one route mirroring `reports_stock_page`, one page template mirroring `reports_stock.html`, and a landing link. Purpose: Surface batches nearing or past expiry so the operator can act before stock is lost — satisfying ROADMAP criterion 3. Legacy batches (NULL expiry) are excluded automatically by the `is_not(None)` filter, sidestepping the NULL-expiry trap.
Output: A working `/reports/expiry` page reachable from the reports landing, with the expired-marker logic and empty state.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| stored batch/product text → report HTML | product name, warehouse name, batch comment/location rendered into the report table |
| server clock/timezone → expired marker | `today` must be the operator's local date, or batches near midnight are mis-flagged |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-10 | Tampering (XSS) | product/warehouse name, batch comment/location on the report | mitigate | Jinja autoescape only, never `|safe` (reports_stock.html precedent) |
| T-10-11 | Information disclosure | read-only report exposing stock | accept | single local operator, no auth in v1 (CLAUDE.md); the report shows only data the operator owns |
| T-10-12 | Tampering / integrity | report query mutating state | mitigate | expiring_batches is read-only (portable `select()`), no writes; the append-only ledger is untouched |
| T-10-SC | Tampering | npm/pip/cargo installs | accept | zero external packages installed this phase (vacuous gate) |
</threat_model>
