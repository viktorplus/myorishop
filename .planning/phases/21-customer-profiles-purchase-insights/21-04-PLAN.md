<objective>
Ship the CUST-01..05 edit surface: four repeatable contact sections (phone / Telegram / email /
social) plus the address field on `customer_form.html`, the `/customers/contact-row` HTMX endpoint
that appends a blank row, and the form-array binding that saves them.

Purpose: Plan 02 built the service contract. This is where the operator actually records

Output: `contact_row.html`, the `.contact-row` CSS rule, `GET /customers/contact-row`, extended

## Locked-decision reinterpretation — READ THIS FIRST

**D-03 in `21-CONTEXT.md` is internally contradictory and is being followed in its form-array
reading, not literally.** D-03 says both:
- *"an «Добавить строку» button appends a blank input row … row removal is a client-side
  `hx-on:click` with no server round-trip"* (form-array model), **and**
- *"add/delete are per-row HTMX operations"* (server-CRUD model).

These are incompatible. **`21-RESEARCH.md` Pitfall 2 and `21-UI-SPEC.md` Interaction 1 both resolve
it to form-array, and this plan follows that resolution**, because server-CRUD is *unreachable* on
the new-customer form: `POST /customers/{id}/contacts` needs a `customer.id` that does not exist
yet — `routes/customers.py:99-101` renders `/customers/new` with `"customer": None`. The
sale-basket precedent D-03 itself cites (`sale_form.html:74` / `sale_row.html`) is likewise a
form-array flow; "per-row HTMX operations" describes the basket's add-row *UX*, not its persistence.

**Consequence: rows become DB rows ONLY on final submit.** There is no
`POST /customers/{id}/contacts` endpoint and no per-row DELETE endpoint in this phase. A task that
says "POST a new contact row to the server" is wrong.
</objective>
