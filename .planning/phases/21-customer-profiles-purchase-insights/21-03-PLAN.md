    `Select` object (NOT executed). It must be a standalone function so Task 3's portability guard
    can compile it without a session. Shape:
    - selected expression: `func.coalesce(func.sum(-Operation.qty_delta * func.coalesce(Operation.unit_price_cents, 0)), 0)`
      — **coalesce at BOTH levels.** RESEARCH Pitfall 4, "the likeliest first bug in this phase":
      `unit_price_cents` is nullable, `NULL * anything` is `NULL`, and `SUM()` over zero rows returns
      `NULL`. The inner coalesce saves a price-less line from silently contributing nothing; the
      outer one saves a zero-order customer from rendering `None`. Mirrors the shipped
      `returns.py:56` `func.coalesce(func.sum(...), 0)` precedent.
    - `.join(Sale, Operation.sale_id == Sale.id)`
    - `.where(Sale.customer_id == customer_id, Operation.type.in_(("sale", "return")), Operation.created_at >= start_iso, Operation.created_at < end_iso)`

    revenue; a `return` op has `qty_delta > 0` at the SAME frozen `unit_price_cents` → negative
    revenue. The sum nets returned revenue automatically.

    **No `strftime`, no `date_trunc`, no `EXTRACT`, no date function of any kind.** The window is
    computed in Python and passed as bound params; `created_at` is `String(32)` holding
    `utcnow_iso()` output, and ISO-8601 UTC strings sort lexicographically == chronologically

    migration breaks (CLAUDE.md, explicit).

    **`_favorites_stmt(customer_id: str, limit: int)`** — private, returns the `Select` (NOT executed)
    so Task 3's portability guard can compile it. Copy-adapt `top_selling_products`' shape; do not
    invent a new one:
    - `freq = func.count(func.distinct(Operation.sale_id)).label("freq")`
    - `qty = func.coalesce(func.sum(-Operation.qty_delta), 0).label("qty")`
    - `select(Product, freq, qty)`
      `.join(Operation, Operation.product_id == Product.id)`
      `.join(Sale, Operation.sale_id == Sale.id)`
      `.where(Sale.customer_id == customer_id, Operation.type == "sale")`
      `.group_by(Product.id)`
      `.order_by(freq.desc(), qty.desc(), Product.name)`
      `.limit(limit)`

    **Frequency is `count(DISTINCT sale_id)` — orders containing the product — NOT `count(lines)`.**
    This resolves D-04's under-specification (RESEARCH Pitfall 3, accepted by the developer). Nothing
    prevents two `sale` ops for the same product in one sale: **the batch picker writes one op per
    batch**, so a batch-split purchase is a ROUTINE occurrence in this app, not an edge case. "How
    often does this person buy X" means how many shopping trips included X; a two-batch split of one
    purchase is ONE purchase. Total quantity is D-04's secondary column, so per-line volume is not
    lost. Put this reasoning in the docstring — it is the one non-obvious semantic in the file.

    **`Product.name` is a mandatory third sort key.** `freq DESC, qty DESC` alone leaves ties on BOTH
    keys in DB-arbitrary order, which makes any test asserting exact ordering of two equal-scoring
    products flaky.

    **Do NOT filter `Product.deleted_at`** — this is a historical view (`reports.py:30-33` Pitfall 5
    rule). A soft-deleted product a customer genuinely loved still belongs in their ranking. Say so
    in the docstring so nobody "fixes" it.

    **`type == "sale"` only** — mirroring `purchase_history`. This ranks what was BOUGHT, not net
    financial exposure; returns are `spend_totals`' concern. This is a deliberate scope boundary
    (D-04 vs D-06), not an oversight — state it in the docstring.

    **`favorite_products(session, customer_id, limit: int = 10) -> list[dict]`** — public. Executes
    `_favorites_stmt` and returns `[{"product": Product, "freq": int, "qty": int}, ...]`, the house
    `list[dict]` return shape. Default `limit=10` per D-04a. `limit` is an int parameter bound by
    SQLAlchemy — never string-interpolated.

    **`last_order_date(history: list[dict]) -> str | None`** — public, **pure**: it takes the
    already-loaded `purchase_history` result and returns `history[0]["op"].created_at` when non-empty,

    and the detail route already loads it (`routes/customers.py:132`), so row 0 IS the most recent
    order — issuing a 7th query for data already in memory is the thing RESEARCH Pitfall 6 tells us
    not to do. CONTEXT explicitly leaves this computation to Claude's discretion; this is the choice.
    Returning the raw ISO string (not a `date`) is deliberate: the detail page renders it with the
    existing `| local_dt` filter, exactly as `purchase_history.html:22` already does.
