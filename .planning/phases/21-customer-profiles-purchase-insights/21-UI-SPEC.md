## Security Decision: Contact Values Are NOT Clickable

**Stated decision, with rationale — this is a security control, not a cosmetic choice.**

`21-RESEARCH.md` §Security Domain names exactly one realistic threat in this phase: **stored XSS via
the free-form social-profile link (CUST-04)**, the only operator-supplied free-form value that gets
stored and re-rendered.

**The contract:**

> **Every contact value — phone, Telegram handle, email, and social link — renders as
> autoescaped plain text. No `<a href>`. No `tel:`, no `mailto:`, no `https://` anchor.
> On both `customer_detail.html` and `customer_form.html` (the form re-echoes values into
> `value="…"` attributes, which autoescape also covers).**

**Why:**

1. Jinja2 autoescape is on by default and is the codebase's stated house rule
   (`purchase_history.html:1-3`: *"Autoescape only; never |safe"*). Autoescape fully neutralizes the
   HTML-injection vector for a value rendered as **text**.
2. Autoescape does **not** neutralize `javascript:` in an `href`. Rendering
   `<a href="{{ c.value }}">` would escape the quotes but happily produce a working
   `href="javascript:…"` — reopening the exact hole autoescape otherwise closes. Escaping and
   URL-scheme safety are different problems; only one of them is solved for free here.
3. **No requirement asks for clickable contacts.** CUST-01..04 ask that the profile *supports*
   multiple values — i.e. that they can be recorded and read. Clickability is unrequested scope that
   costs a security control to obtain.
4. The app is a local, offline, single-operator tool. The operator can select and copy a value; a
   click-through affordance buys little and would need a server-side scheme allow-list to be safe.

**If a future phase makes any contact value clickable, it MUST first ship a server-side scheme
allow-list** restricted to `http`/`https` (validated in `app/services/customers.py`, per the house
rule that all validation lives in the service layer — `routes/customers.py:1`), reject or render-as-
text anything else, and re-verify. **Do not add `| safe` anywhere in this phase — there is no
legitimate use for it here.**

**Second, related control (same class of bug):** the `kind` query parameter on
`GET /customers/contact-row?kind=phone` is interpolated into the rendered input's `name` attribute,
which makes it untrusted input. The route **must** validate `kind` against the `CONTACT_KINDS`
allow-list and reject unknown values before rendering — the CR-01 rule that `/sales/row` already
follows with `_ROW_ID_RE` (`sales.py:312-320`).

## Interaction & Layout Decisions

Not covered by the fixed template sections above but load-bearing for the executor. Each traces to a
specific `21-CONTEXT.md` decision or `21-RESEARCH.md` finding.

### Contact rows on `customer_form.html` (CUST-01..04)

1. **Form-array pattern, NOT per-row server CRUD.** `21-RESEARCH.md` Pitfall 2 resolves D-03's
   internal contradiction: only the form-array reading works, because the new-customer form has no
   `customer.id` yet (`routes/customers.py:99-101` renders with `"customer": None`), so a
   `POST /customers/{id}/contacts` endpoint would be unreachable there. Rows become DB rows **only**
   on final submit, exactly like the sale basket. This is the single most important interaction
   decision in this phase — a plan task that says "POST a new contact row to the server" is wrong.
2. **Markup shape per kind** — four structurally identical sections, differing only by `kind`:
   ```jinja
   <div class="field">
     <label>Телефоны <span class="muted">(необязательно)</span></label>
     <div id="contacts-phone">
       {# one .contact-row per existing value; if none, exactly ONE blank row #}
     </div>
     <button type="button" class="secondary"
             hx-get="/customers/contact-row?kind=phone"
             hx-target="#contacts-phone"
             hx-swap="beforeend">Добавить телефон</button>
     {% if errors.phone %}<p class="error">{{ errors.phone }}</p>{% endif %}
   </div>
   ```
   Container ids: `#contacts-phone`, `#contacts-telegram`, `#contacts-email`, `#contacts-social`.
3. **New partial `partials/contact_row.html`** — modeled on `sale_row.html`, but a `<div>` not a
   `<tr>`: a contact row has one data column, so the basket's table shape would be pure overhead
   inside a `.stacked-form` (`display: block; max-width: 480px`, `style.css:71-74`).
   ```jinja
   <div class="contact-row">
     <input type="text" name="{{ kind }}[]" value="{{ value or '' }}">
     <button type="button" class="secondary"
             hx-on:click="this.closest('.contact-row').remove()">Удалить</button>
   </div>
   ```
4. **Array input names bind to `Form([], alias="…[]")`** — `phone[]`, `telegram[]`, `email[]`,
   `social[]`, mirroring the shipped `sale_row.html:16` ↔ `sales.py:392-395` contract.
   No `id=` on contact inputs: unlike `sale_row.html` (whose generated per-row ids exist to serve
   its focus hook and `hx-target`s), a contact row has no lookup, no focus hook, and no swap target,
   so per-row ids would be unused ceremony that could collide. The `<label>` labels the **section**,
   not an individual row — which is correct for a repeated N-value group.
5. **Row removal is client-side only.** `hx-on:click="this.closest('.contact-row').remove()"` — no
   server round-trip, exactly the `sale_row.html:49` mechanism (minus its batch-wrapper cleanup,
   which has no analog here: contact rows have no parallel hidden array to keep index-aligned).
6. **Empty state on the form = exactly one blank row per kind** (resolved this phase). Both on
   `/customers/new` and on an edit page for a kind with zero stored values, render one blank
   `.contact-row`. This mirrors `sale_form.html:64-68`'s own else-branch (one blank row when there
   are no lines) and means the operator can type a first phone without clicking "Добавить" first.
   **There is no empty-state *message* on the form** — a blank input is the empty state.
   Consequence the executor must honor: **blank rows are discarded server-side** on save (the
   always-present blank row must never create an empty `CustomerContact`).
7. **Field order on the form** (resolved this phase): the three existing identity fields
   (Имя → Фамилия → Номер консультанта, untouched) → Телефоны → Telegram → Email → Соцсети →
   Адрес → `.form-actions`. Identity first (unchanged from today), then reachability, then location
   — most-specific to least-specific. Address is last because it is the one single-valued contact
   field (D-02) and separates cleanly from the four repeatable groups above it.
8. **Address is a plain `input[type=text]`** (CUST-05, D-02), a standard `.field` in the
   `.stacked-form`. **Not a `<textarea>`** — `21-RESEARCH.md` records that no textarea exists
   anywhere in `app/templates` today, and introducing one here would be a new UI convention for a
   `String(300)` column that `warehouse_form.html` already models as a text input.
9. **Re-echo on validation error:** contact values and address follow the form's existing
   `{% if form %}{{ form.x }}{% elif customer %}{{ customer.x }}{% endif %}` echo convention
   (`customer_form.html:15,22,29`) so a 422 never silently drops what the operator typed. The
   submitted arrays are re-rendered as `.contact-row`s (one per submitted value), not reset to one
   blank row.
10. **Route order is load-bearing.** `GET /customers/contact-row` **must** be declared **above**
    `GET /customers/{customer_id}` in `routes/customers.py`, or the dynamic route swallows it and
    the operator gets a 404-shaped surprise (`routes/customers.py:22-25` already warns about this;
    `21-RESEARCH.md` Open Question 3). Validate `kind` against `CONTACT_KINDS` (see Security).

### Insight blocks on `customer_detail.html` (CUST-06, CUST-07, CUST-08)

11. **Section order** (resolves the CONTEXT.md "Claude's Discretion" item on insights layout):
    ```
    <h1> name                      (existing, untouched)
    <p class="muted"> consultant № (existing, untouched)
    <p class="page-actions"> Изменить (existing, untouched)
    <section id="customer-contacts">  <h2>Контакты</h2>       ← NEW (CUST-01..05)
    <section id="customer-insights">  <h2>Покупки</h2>        ← NEW (CUST-06, CUST-07)
    <section id="customer-favorites"> <h2>Любимые товары</h2> ← NEW (CUST-08)
    <section id="customer-history">   <h2>История покупок</h2>  (existing, MOVES TO LAST)
    ```
    Rationale: identity → how to reach them → what they're worth → what to offer them → the raw
    ledger. The existing purchase-history section keeps its `id="customer-history"`, its include,
    and its position **relative to the page's other content** — it simply ends up last because three
    sections are inserted above it. The insights are summaries *of* the history, so they precede it;
    the history stays as the drill-down detail and retains the «Вернуть» action
    (`purchase_history.html:30-31`) untouched.
12. **Contacts section rendering (CUST-01..05):** one `<p>` per **kind that has at least one value**,
    values joined visibly on the line; kinds with zero values are **omitted entirely** (no
    "Телефон: —" × 4 clutter on a bare profile). Address renders on its own line when present. If
    the customer has no contacts **and** no address, the section body is the single
    `<p class="muted">Контакты не указаны.</p>`. All values are **plain autoescaped text** — see
    the Security Decision above. Do not build a table here: the shape is a short label→values list,
    and no requirement asks for per-contact columns, sorting, or actions.
13. **Last order date (CUST-06)** renders as a single `<p>` line at the top of the insights section,
    above the spend tiles — it is one fact, not a metric tile; giving it a fourth tile would break
    `.metric-grid`'s 3-column layout (`style.css:356-360`) and imply a comparability with the money
    figures that it does not have. Uses the existing `| local_dt` filter (as
    `purchase_history.html:22` already does). Derived from the already-loaded `purchase_history`
    rows (`history[0]`, which is ordered `created_at DESC`) — no 7th query
    (`21-RESEARCH.md` Pitfall 6; CONTEXT leaves the computation to Claude's discretion).
14. **Spend totals (CUST-07) reuse `.metric-grid` + `.metric-tile` + `.tile-label` verbatim**
    (Phase 17, `style.css:351-379`; `finance_tiles.html` is the shape to copy). Three tiles map
    exactly onto the existing `repeat(3, 1fr)` grid — no CSS change. Each tile:
    `<p class="tile-label">` → `<p class="num"><strong>{{ cents | cents }}</strong></p>` →
    `<p class="muted">с {{ period_start | ru_date }}</p>`. The mandatory «С учётом возвратов.» line
    renders once, below the grid.
15. **The period-start captions are not decoration — they resolve Pitfall 7.** D-05 is
    calendar-period-**to-date**, so in Jan/Apr/Jul/Oct the month and quarter tiles show *identical
    numbers* (today, 2026-07-17, is one such date — both windows are
    `2026-06-30T21:00:00+00:00 → 2026-07-17T21:00:00+00:00`). That is correct behaviour, but two
    identical figures side by side read as a bug to the operator. The muted "с 01.07.2026" /
    "с 01.07.2026" captions make the identity self-explanatory. This requires the route to pass the
    three period-start `date`s into the template context alongside the three totals — rendered with
    the existing `| ru_date` filter (dd.mm.yyyy), **not** a new month-name helper (no RU month-name
    localization exists in this codebase, and adding one for a caption is unwarranted).
16. **Money renders only via `{{ x | cents }}`** — the integer-minor-units → display contract.
    Values are stored as `Integer` cents and stay integers until the template
    (`21-RESEARCH.md` Architectural Responsibility Map: "Money rendering | Frontend Server (Jinja2
    `| cents` filter) | Integers stay integers until the template"). **Never** `f"{cents/100:.2f}"`,
    never float math in a template — `CLAUDE.md` bans float money, and `format_cents` (`core.py:49`)
    is the single sanctioned rendering point. It already renders negatives with a leading `-`
    correctly (verified), which is why Decision 1 in Color (no sign-coloring) is safe.
17. **Favorites (CUST-08) reuse `top_selling_rows.html`'s table shape**, extended from 2 columns to
    3: Товар | Покупок, раз (`.num`) | Куплено, шт. (`.num`). Capped at 10 rows (D-04a) with
    **no pagination and no "показать ещё" link** — 10 is the whole list by definition, and adding
    list chrome (`partials/pagination.html`) to a deliberately-capped ranking would imply an 11th
    row exists to page to. Product cell renders `{{ row.product.name }} ({{ row.product.code }})`,
    verbatim from `top_selling_rows.html:25`. Product names are **not** links — consistent with the
    existing top-selling report, which does not link them either.
18. **Favorites empty state** uses `<p class="empty-state muted">Покупок пока нет.</p>`.
    Note the class pairing: `top_selling_rows.html:12` uses `.empty-state muted` (48px vertical
    padding, centered) for its own zero-rows case; this phase reuses that treatment, with
    `purchase_history.html:5`'s copy so both sections agree.
19. **The zero-order profile is a first-class state, not an edge case** (`21-RESEARCH.md` Pitfalls 4
    and 6 both flag it). On a customer with no orders the detail page must render, in full and
    without error: «Последний заказ: —», three tiles reading «0,00», the «С учётом возвратов.» line,
    «Покупок пока нет.» under Любимые товары, and «Покупок пока нет.» under История покупок. **No
    `None` may reach any template**, and no section may be conditionally hidden — a new customer's
    profile is a normal profile with zeros in it, and hiding the blocks would make the operator
    think the feature is broken.
20. **Soft-deleted products stay in the favorites list.** The favorites query deliberately does not
    filter `Product.deleted_at` (`21-RESEARCH.md` Code Examples, citing `sales_profit_report`'s
    documented Pitfall 5 rule): this is a historical view, and a discontinued product a customer
    genuinely loved still belongs in their ranking. The template needs no special treatment for
    them — no `.muted` row, no badge. (`product_rows.html`'s `<tr class="muted">` deleted-row
    treatment is a *list-management* affordance for the products page; it does not apply to a
    historical ranking.)
