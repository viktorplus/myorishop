<objective>
Ship the CUST-06/07/08 read surface and the CUST-01..05 contacts display on
`customer_detail.html`: the Контакты section, the Покупки section (last order date + three spend
tiles), and the Любимые товары ranked table.

Purpose: this is the phase goal's payoff — "shows what they actually buy, so the operator can act
on it". Every number already exists (Plan 03); this plan makes it visible, safely.

Output: three new partials, the restructured detail page, the extended `customer_detail` route
context, `CONTACT_KINDS` registered as a Jinja global, and the phase's e2e + security tests.

## Two failure modes this plan exists to prevent

1. **The `| ru_date` TypeError.** `format_ru_date` (`core.py:56-66`) calls `date.fromisoformat(iso)`
   on a **string**. Passing a Python `date` raises `TypeError`. This is closed structurally: Plan 03's
   `spend_view` already returns `start_iso` as `str`. **The route must render the `spend_view` output
   as-is and must never pass a `date` object into the template.** (Found by `21-PATTERNS.md`; unflagged
   by RESEARCH and UI-SPEC.)
2. **`None` reaching a template.** RESEARCH Pitfall 4 names the zero-order customer "the likeliest
   first bug in this phase". The zero-order profile is a **first-class state, not an edge case**.
</objective>
