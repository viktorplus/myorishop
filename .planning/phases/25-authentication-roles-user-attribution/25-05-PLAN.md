## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator → admin routers | An operator must not reach user-management/warehouses/dictionaries/settings |
| admin form input | user creation/reset input is untrusted (login/role/password) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-05-01 | Elevation of Privilege | admin routers | mitigate | `require_role("administrator")` server-side on warehouses/dictionary/settings/users; test `operator_blocked` proves 403 (ROLE-02/03). |
| T-25-05-02 | Elevation of Privilege | menu-hide mistaken for control | mitigate | The dependency is the boundary; nav-hide (Plan 06) is cosmetic only and separately noted (ROLE-03). |
| T-25-05-03 | Elevation of Privilege | self-lockout / self-deactivate | mitigate | `deactivate_user` refuses the acting admin's own id; own-row deactivate control omitted server-and-client-side. |
| T-25-05-04 | Tampering | user create/reset input | mitigate | All validation in `app/services/users.py` (role allow-list, duplicate login, password required); routes stay thin; CSRF-protected POSTs. |
| T-25-05-05 | Information Disclosure | reset password echo | mitigate | The new password is written server-side and never echoed back into the reset fragment. |
</threat_model>
