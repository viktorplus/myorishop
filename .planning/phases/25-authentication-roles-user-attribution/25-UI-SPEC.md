# Phase 25 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Brownfield note:** v1.0–v2.0 shipped with an established design system. This phase adds the app's FIRST security surfaces (login, first-run setup, user management), role-conditioned navigation, logout, and an author column/filter on two existing read pages. It introduces **no new design tokens** — every value below is carried verbatim from `app/static/style.css` and the prior UI-SPECs (02–06, and the 23-UI-SPEC referenced in `partials/history_rows.html`). Each pattern claim cites the existing template it reuses.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (server-rendered Jinja2 + HTMX 2.0.10 vendored; plain CSS, no framework, no build step, no CDN) |
| Preset | not applicable |
| Component library | none — reuse existing `app/static/style.css` |
| Icon library | none — text labels only (offline-first; no icon/font network dependency) |
| Font | system-ui stack (existing `body` rule): `system-ui, -apple-system, "Segoe UI", sans-serif` |

**shadcn gate:** not applicable — stack is FastAPI + Jinja2 + HTMX, not React/Next.js/Vite; no `components.json`. Registry safety gate: not applicable.

**Assumptions (auto-mode — no `25-CONTEXT.md` exists, no interactive user):**
- **A-UI-1 — auth chrome:** login and setup render inside a NEW standalone `auth_base.html` (no `nav`, no `mobile-tabbar`), because an unauthenticated visitor must not see navigation into protected pages, and the mobile tabbar links only to gated routes. `auth_base.html` loads `/static/style.css` + `/static/htmx.min.js` (both public via the `app.mount` static mount — verified in 25-RESEARCH.md) and carries the CSRF `hx-headers` on `<body>`. It is width-responsive (one template serves desktop and phone) using the existing `.stacked-form` (max-width 480px) centered — no separate mobile login template.
- **A-UI-2 — user management lives under `/settings/users`** (admin-only), reusing the Settings section that ROLE-03 already classifies as admin-only. Confirm the exact router prefix at plan time (25-RESEARCH Open Question 3).
- **A-UI-3 — the author filter is placed in the universal top `.filter-bar`** (next to «Тип операции» / «Сортировать по») on `/history`, so it applies to every type view without fighting the per-type column narrowing (23-UI-SPEC Interaction 8 dropped the «Кто» column from narrowed views; the filter is re-introduced as a top-bar control, not a column filter).
- **A-UI-4 — logout is a POST** (state-changing, CSRF-protected) rendered as a right-aligned nav control showing the current user + «Выйти».

UI language: **Russian** (`base.html lang="ru"`). Stored `role` values stay Latin (`administrator` / `operator`) — only their RU labels appear in the UI (mirrors the `reason_code` Latin-value / RU-label convention, `history_rows.html` `WRITEOFF_REASONS`). Jinja autoescape ON — `display_name`, `login`, and `created_by` are untrusted-at-rest text; **never `|safe`** (T-05-18 precedent, cited in `history_rows.html` line 11 and `mobile_partials/history_cards.html` line 18).

## Interaction Contract

- **Login submit:** `hx-post="/login"` (or a plain POST form) targeting the login form partial (`hx-target` the form wrap, `hx-swap="outerHTML"`). Success → server returns a redirect the browser follows to `/`; for an HX request return **204 + `HX-Redirect: /`** (HTMX performs full navigation) so no fragment is swapped. Bad credentials → **422** with the form re-rendered + `.error-block` (422 is whitelisted for swapping in the `htmx-config`, `base.html` line 18–19 — duplicated into `auth_base.html`). Password field cleared on re-render; login preserved.
- **Guard redirect for unauthenticated / expired sessions (25-RESEARCH Pitfall 3):** HTMX does NOT swap 4xx by default (the app `htmx-config` marks `[45]..` non-swapping/error). So for an HX request the guard returns **401 with an `HX-Redirect` header** (→ `/login` or `/setup`), triggering a full-page navigation; for a non-HX request it returns a **303 redirect**. This is a hard requirement — a naive 303 body on an HX POST would silently vanish.
- **First-run detection:** the app-level guard counts users; zero users → redirect every non-public request to `/setup`. `/setup` POST re-checks `count_users == 0` before creating the admin (closes the race), then sets the session and `HX-Redirect`/303 to `/`.
- **Create user:** `hx-post="/settings/users"` targeting the users-table partial (`hx-swap="outerHTML"` on the table, or `beforeend` a new row). Validation failure → 422 with the create-form re-rendered, inline errors, entered values preserved (duplicate login, empty password, bad role). Success → 200 with the new row appended + a neutral «Пользователь создан.» line + the create-form reset. `hx-disabled-elt="find button"` on submit (existing idiom).
- **Deactivate / reactivate:** `button.danger` «Деактивировать» with native `hx-confirm` (the RU confirm copy above), `hx-post="/settings/users/{id}/deactivate"`, targeting that user's row (`hx-swap="outerHTML"`). The row re-renders with status «Отключён» (`.muted`) and the action flips to a neutral «Активировать». Server re-checks `is_active` on every request, so the deactivated user's still-valid cookie is rejected on their next request (USER-03) — no client action needed. Admin's own row never shows a deactivate control.
- **Reset password:** the «Сбросить пароль» control reveals an in-row «Новый пароль» input + submit (HTMX swap of the row/fragment). `hx-post="/settings/users/{id}/reset-password"`, `hx-disabled-elt="find button"`; success swaps back the row + neutral «Пароль сброшен.». The new hash is written server-side via the auth service; the field is never echoed back.
- **Logout:** `hx-post="/logout"` on the nav control (state-change → POST, CSRF-carried), returning **204 + `HX-Redirect: /login`** for HX or **303** otherwise. `request.session.clear()` server-side.
- **CSRF wiring (AUTH-05, 25-RESEARCH Pattern 5) — design requirement for every state-changing surface:**
  - `<body hx-headers='{"X-CSRF-Token": "{{ csrf_token }}"}'>` on `base.html`, `mobile_base.html`, AND `auth_base.html` — one line covers every `hx-post`/`hx-put`/`hx-delete` on the page (including login, logout, create-user, deactivate, reset).
  - Every non-HTMX `<form method="post">` also carries `<input type="hidden" name="csrf_token" value="{{ csrf_token }}">` (login/setup forms, if submitted as plain forms).
  - `csrf_token` reaches every template via the shared-`templates` context processor (25-RESEARCH Pattern 6) — no route re-passes it.
- **Role-conditioned nav is cosmetic:** the `{% if current_user.role == "administrator" %}` guards in `base.html`/`mobile_base.html` only hide links; the authoritative boundary is the server-side `require_role` dependency on the admin routers (ROLE-02/03). An operator navigating directly to an admin URL gets a **403** (rendered page «Доступ только для администратора.» for a full GET; 403 for HX). Never rely on menu-hide alone.
- **History / Reports author filter:** the «Пользователь» `<select>` uses the existing filter HTMX idiom (`hx-get`, `hx-trigger="change"`, `hx-include` the sibling inputs/selects, `hx-target` the results block, `hx-push-url="true"`) so the filtered view is linkable and back-navigable. Filtering resets pagination to page 1 (existing History behavior). NULL-author (pre-auth) rows are excluded when a user is selected — correct, they predate auth.
- **Author display never trusts the client:** `author_id` is derived server-side from the session at the single write path (`record_operation` / `record_cash_movement`), never from a form field (USER-05 / repudiation mitigation). The History/Reports UI only READS it.
- **Keyboard-first (core value: fast entry):** every form submits on Enter; tab order = visual order (login: Логин → Пароль → Войти). Visible 2px accent focus outline on all interactive elements incl. `<select>` — never `outline:none` without replacement.
- **Buttons during request:** `hx-disabled-elt` on submit/confirm buttons; existing `button:disabled { opacity:.6 }`.
- **No client-side JS beyond vendored HTMX.** No modals, no toasts, no external fonts/icons/images — offline-first. htmx 2.x `hx-on::` double-colon syntax only. Native `hx-confirm` for the deactivate action (already used by `product_form.html`).
- **Manual UAT gates:** first run with zero users lands on `/setup` and cannot reach any other page; after creating the admin, login is required everywhere incl. `/export` and `/backup`; a wrong password shows «Неверный логин или пароль.» and writes no session; a logged-in session survives a browser refresh and ends on «Выйти»; an operator cannot see OR reach (`/settings/users` → 403) admin sections; an admin sees «Настройки» and can create/deactivate/reset users; a deactivated user cannot log in and is bounced on their next request, but their past History rows still show their name; a new sale/receipt records the logged-in user as author and both `/history` and `/reports/sales` can filter by «Пользователь»; pre-auth rows still render («operator», muted) and are simply excluded by a user filter; every state-changing POST without a valid CSRF token is rejected (403).
