<objective>
Surface the operating user on the History page and add a "filter by user" control to both History and the sales/profit Report — reading the `author_id` stamped by Plan 07, tolerating pre-auth NULL-author rows.

Purpose: USER-06 (History shows the operating user + filter by user) and RPT-01 (Reports filterable by operator). Both reuse the app's exact existing filter idioms (customer/category/period). Pre-auth rows (frozen `created_by="operator"`, NULL `author_id`) are shown via `created_by` text and simply excluded when a user filter is active.
Output: author join + filter in the read services, author params on the routes, «Пользователь» selects in the two templates, and filter tests.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client filter input | The `author` query param is untrusted (an unknown id must not error, just match nothing) |
| rendered author names | display_name/login/created_by are untrusted-at-rest text |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-08-01 | Tampering | `author` filter param | mitigate | Applied only as a parameterized `.where(author_id == ...)`; an unknown/absent id simply matches no rows (no raw SQL, no 500) — mirrors the existing type/customer filter guards. |
| T-25-08-02 | Information Disclosure | author names in lists | mitigate | Jinja autoescape only, never `|safe`, on `display_name`/`login`/`created_by` (T-05-18 precedent). |
| T-25-08-03 | Integrity | dropped historical rows | mitigate | LEFT OUTER JOIN on author (never inner) so pre-auth NULL-author rows are never silently dropped from the unfiltered view (Pitfall 2). |
</threat_model>
