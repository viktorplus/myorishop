## Phase Goal

**As a** shop operator, **I want to** maintain product cards (code, name, category, three prices) and find any product in seconds by code or name, **so that** daily receipt and sale entry stays fast and prices are never lost.

*(Same assembled story as Plan 02-01 — ROADMAP Goal line is outcome prose; flagged to orchestrator.)*

<objective>
Vertical slice 3: instant search on /products. Operator types a partial code or name (any case, including Cyrillic) and sees ranked, capped, highlighted results update in place within ~300ms — CAT-03 complete.

Purpose: "find any product in seconds" is the headline of the phase goal; this slice delivers it.
Output: search_products/split_match/search_view in the catalog service, GET /products/search partial endpoint, active-search input on the list page, <mark> highlighting, mark styling.

### Decision Traceability (from 02-CONTEXT.md; continues Phase 1 numbering)

| ID | Locked decision | This plan |
|----|-----------------|-----------|
| D-18 | /products is list + search | implements the search half |
| D-25 | Single input, HTMX partial results, debounce ~300ms, case-insensitive LIKE on code prefix and name substring | implements |
| D-26 | Ranking: exact code > code prefix > name substring; cap 20 rows | implements |
| D-27 | Cyrillic case-insensitivity via name_lc (Python-lowered query; SQLite lower()/LIKE are ASCII-only) | implements read side |
| D-20 | Deleted products hidden from search | enforced in the query |

**Critical constraint (RESEARCH Finding 1, verified on this machine's SQLite 3.50.4):** never use `func.lower(Product.name)` or `ilike` on the name column — SQL lower() does not fold Cyrillic. Query strings are lowered in PYTHON and compared against `name_lc`. `func.lower()` is permitted ONLY on Product.code (ASCII codes, assumption A1).
</objective>
