## Phase Goal

**As a** shop operator, **I want to** maintain product cards (code, name, category, three prices) and find any product in seconds by code or name, **so that** daily receipt and sale entry stays fast and prices are never lost.

*(Story assembled verbatim from ROADMAP.md Phase 2 Goal + Success Criteria 1–4; ROADMAP Goal line is outcome prose, not canonical story format — flagged to orchestrator, same convention as Phase 1.)*

<objective>
Vertical slice 1 of the catalog: operator can CREATE a product card with all CAT-01 fields and SEE it in the /products list. Delivers the full Phase 2 data foundation inside the slice: migration 0002, Product columns + Dictionary model, OPERATION_TYPES extension, the IN-01 deleted-product guard, and the catalog service create path through the single write path.

Purpose: after this plan a real user can add a product with prices and see it listed — the first new capability of Phase 2. Every later plan (edit/history, search, dictionary) builds on the columns, service and templates created here.
Output: migration 0002 + extended models, app/services/catalog.py (create/list), GET/POST /products + /products/new pages in Russian, nav in base.html, RED→GREEN tests/test_catalog.py (create/list scope).

Functionally identical for lookups (unique index on code). The Dictionary model + table are created HERE (not in Plan 02-04) so that `Base.metadata` stays in lockstep with migration head — conftest's `create_all` and Alembic must always agree.

**Plan decision PD-2:** product codes are normalized with `.strip()` at write time (hedge for RESEARCH assumption A1 — codes expected ASCII).
