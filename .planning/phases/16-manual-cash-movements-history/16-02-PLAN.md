## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP form strings ‚Üí cash write | `amount_raw`, `category`, `note`, `confirm` arrive as untrusted client strings; all validation/sign/gate logic runs here before any row is written. |
| HTTP query `bucket` ‚Üí history SQL | The ¬´–¢–∏–ø¬ª filter value is untrusted; it must map through `CASH_BUCKETS.get` (parameterised `.in_`), never string-interpolated into SQL. |
| client `confirm` flag ‚Üí negative-balance override | `confirm == "1"` is an explicit operator override; the would-be balance is recomputed live at write time (never trusted from the client). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-01 | Tampering | amount field (`to_cents` parse) | mitigate | Parse only via `app.core.to_cents` (rejects inf/nan/garbage, Russian comma) then reject `<= 0`; zero/blank/negative/non-integer ‚Üí `(None, {"amount": ...})`, ZERO writes. Asserted by Task 1 rejection tests. |
| T-16-02 | Tampering / Elevation | category allow-list + direction | mitigate | Category must be in `CASH_BUCKETS["withdrawal"]`/`["deposit"]`; a system key ("sale"/"return") or unknown key is rejected server-side (defence-in-depth on top of `record_cash_movement`'s `CASH_CATEGORIES` guard). |

| T-16-04 | Tampering | mandatory-comment bypass (D-04) | mitigate | Enforced in the service for `withdrawal_other`/`deposit_correction`; blank/whitespace note ‚Üí `(None, {"note": ...})`, ZERO writes ‚Äî not reliant on HTML `required`. |
