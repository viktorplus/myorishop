# Phase 16 — UI Design Contract

> Visual and interaction contract for the manual cash-movement forms and the cash-movement history on the «Финансы» page (desktop `/finance` + mobile `/m/finance`). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

> **Scope note — REUSE-HEAVY, not greenfield.** The visual system already exists in `app/static/style.css` (plain CSS, no build step, offline). This contract does **not** invent tokens — it records the existing ones verbatim so the checker can validate against them, then locks the two new forms and the history block by mirroring `writeoff_form` / `history_rows` / `writeoff_oversell`, and resolves three flagged open questions. No new dependencies, no motion system, no icon library, no CDN.

## Component Contracts (this phase)

### A. Two inline forms below the balance (D-06)

Both use the existing `form.stacked-form` shell (max-width 480px, `.field` blocks, `.form-actions`), identical to `writeoff_form.html`. Placement: on the «Финансы» page, directly under the balance, «Снять деньги» first then «Внести деньги», then the history block below (D-07a).

**«Снять деньги» (withdraw) form fields, in order:**

| Field | Control | Contract |
|-------|---------|----------|
| Сумма | `<input type="text" name="amount" inputmode="decimal" placeholder="0,00">` | Money string parsed server-side by `app.core.to_cents` (see resolved Q3). Operator always enters a **positive** value; the service applies the negative sign (D-02a). Error → inline `<p class="error">` under the field. |
| Категория | `<select name="category">` iterating the `withdrawal_*` keys of the `CASH_CATEGORIES` Jinja global | Labels come from the dict (Оплата поставщику / Зарплата / Аренда / Коммунальные / Прочее), never hardcoded. First option empty `<option value="">`. Server re-validates against the allow-list. |
| Комментарий | `<input type="text" name="note">` | Label is neutral «Комментарий» (no «(необязательно)» hint — it is conditionally required; see required-indicator decision below). |
| Submit | `<button type="submit">Снять</button>` inside `.form-actions` | Primary accent button. `hx-post="{{ finance_base }}/withdraw"`, `hx-target="#withdraw-form-wrap"`, `hx-swap="outerHTML"`, `hx-disabled-elt="find button"`. |

**«Внести деньги» (deposit) form fields, in order:**

| Field | Control | Contract |
|-------|---------|----------|
| Сумма | same money input as above, `name="amount"` | Positive only; service applies positive sign; deposits never warn. |
| Основание | `<select name="category">` iterating the `deposit_*` keys of `CASH_CATEGORIES` | Labels: Начальный остаток / Корректировка. Empty first option. |
| Комментарий | `<input type="text" name="note">` | Neutral label (conditionally required for `deposit_correction`). |
| Submit | `<button type="submit">Внести</button>` | Primary accent button. `hx-post="{{ finance_base }}/deposit"`, `hx-target="#deposit-form-wrap"`, `hx-swap="outerHTML"`. No warn branch. |

**Required-comment indicator (D-04 — resolved):** show it **on submit error only**. The comment label stays neutral («Комментарий», no «(необязательно)» hint, since the field is conditionally mandatory and a static "optional" label would be misleading). Server enforces the rule for `withdrawal_other` and `deposit_correction`; on a blank/whitespace comment there, the form re-renders (422) with an inline `<p class="error">Укажите комментарий.</p>` under the comment field, zero writes. This needs **no client-side JS** (no category-change listener) — proportionate and offline-safe. Do **not** add a dynamic `required` toggle.

Success re-render: fresh empty form (200) + the refreshed history rows swapped out-of-band (mirror `writeoff_form`'s `include_oob_rows` pattern).

### B. Negative-balance warn-but-allow (D-05, FIN-05) — withdraw only

Mirror `app/templates/partials/writeoff_oversell.html` exactly — reuse `.error-block` + `.form-actions` + `button.danger`, no new CSS/color. Rendered inside the withdraw form wrap on a `confirm != "1"` withdrawal that would drive the balance below zero (200, zero writes).

```
.error-block:
  <p><strong>Баланс уйдёт в минус</strong></p>
  <p>Текущий баланс {{ balance | cents }}, снимаете {{ amount | cents }}.</p>
  .form-actions:
    <button class="danger" form="withdraw-form"
            hx-post="{{ finance_base }}/withdraw" hx-vals='{"confirm":"1"}'
            hx-target="#withdraw-form-wrap" hx-swap="outerHTML"
            hx-disabled-elt="this">Снять всё равно</button>
    <button type="button" class="secondary"
            hx-on:click="this.closest('...').remove()">Вернуться к форме</button>
```

The dismiss button is client-only (no server call); the underlying form keeps the operator's typed values. Deposits and non-negative withdrawals never render this block.

### C. Cash-movement history block (D-07/D-07a/D-07b)

Below the two forms. Each row shows exactly four things (D-07b): **Когда** (`created_at | local_dt`) · **Тип** (category/bucket label) · **Комментарий** (`note`) · **Сумма** (`amount_cents | cents`, signed, no currency symbol). No product column, no separate manual/auto flag column. `note` and labels render with Jinja autoescape only — **never `|safe`** (untrusted operator text).

«Тип» filter = 4 coarse buckets **Продажа / Возврат / Снятие / Внесение** in a header-row `<select name="bucket">` (plus a «Все типы» empty option), driven by the `CASH_BUCKET_LABELS` Jinja global. Default sort: date descending. Filter state preserved across pages via `extra_qs` (mirror `history_rows.html` / `routes/history.py`).

Empty-state copy (in the rows partial): unfiltered → «Движений пока нет.» ; filtered → «Нет движений по выбранному типу.» (rendered in `.empty-state.muted`, mirroring `history_rows.html`).

## Resolved Open Questions (the real decisions of this spec)

### Q1 — Mobile cash-history pagination style → **cards + «Показать ещё» load-more** (mobile-native)

Desktop cash history uses the numbered `page_window` + `partials/pagination.html` bar (matching `routes/history.py`, Phase 14). **Mobile cash history uses the existing mobile convention: `.mobile-card` stacks + a `has_next` sentinel + a «Показать ещё» load-more button**, mirroring `mobile_partials/history_cards.html` + `history_load_more.html` (Phase 11). Justification: the mobile «Финансы» page is a single vertical scroll (forms then history) and the operator already knows the load-more pattern from `/m/history`; numbered page bars with many small tap targets are worse on a phone. Do **not** mix the two styles. (Resolves RESEARCH Pitfall 7 / Open Question 1 / Assumption A2.)

### Q2 — Shared vs separate partials → **share the two forms, fork the history presentation**

- **Forms (withdraw + deposit): SHARED** desktop↔mobile. Markup is identical `form.stacked-form`; the only difference is the route prefix, passed in as a `finance_base` context var (`/finance` desktop, `/m/finance` mobile) so `hx-post="{{ finance_base }}/withdraw"` resolves correctly on each surface. Wrap ids (`#withdraw-form-wrap`, `#deposit-form-wrap`) are reused — the two pages never render together, so no collision. The shared negative-balance warn partial is likewise parameterized by `finance_base`.
- **History presentation: FORKED**, exactly as the codebase already forks it — desktop `partials/cash_history_rows.html` (table + `#cash-history-rows` swap + numbered `partials/pagination.html`) mirroring `history_rows.html`; mobile `mobile_partials/cash_history_cards.html` + `cash_history_load_more.html` mirroring the mobile history pair. Table-rows vs card-stack are genuinely different layouts, so a shared partial would be forced.

(Resolves RESEARCH Open Question 2.)

### Q3 — Amount input format → **money string parsed via `to_cents`** (not whole integer units)

The amount field is a text input `inputmode="decimal"` with placeholder `0,00`, parsed server-side by `app.core.to_cents` — identical to how every existing money-entry form takes amounts (`sale_form` / `mobile_partials/sale_step_qty_price.html` use `inputmode="decimal" placeholder="0,00"` → `to_cents`; `to_cents` accepts the Russian comma «12,50», rejects `inf`/`nan`/garbage, `ROUND_HALF_UP`). The operator enters a positive money value; the service rejects `<= 0` (zero writes) and applies the sign by direction (D-02a). It is **not** a whole-integer-units field and **not** a raw-cents field. (Resolves RESEARCH Assumption A1 / Open Question — amount format.)
