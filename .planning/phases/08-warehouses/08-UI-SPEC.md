## Spacing Scale

Declared values (must be multiples of 4). Identical to Phases 2–7 — already implemented in `style.css`, no changes needed.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Gap between label and input; inline validation error top margin |
| sm | 8px | Table cell padding (8px 12px); button padding vertical; inline add-row form gaps (`.form-actions`) |
| md | 16px | Gap between form fields (`.field`); button horizontal padding; error-block padding; gap between the delete-confirmation warning row and the table above it |
| lg | 24px | Gap between page sections (not heavily used this phase — single page, single table) |

Control heights derived from scale: inputs/buttons 36px (8px vertical padding + 16px text + borders) — the new `name`/`address` inputs and the inline delete/restore buttons reuse this unchanged.

## Typography

Exactly 4 sizes, 2 weights — identical to Phases 2–7. No other values permitted.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body (inputs, table cells, paragraphs) | 16px | 400 | 1.5 |
| Label (form labels, table headers, muted hints) | 14px | 400 (labels/hints) / 600 (table `th`) | 1.5 |
| Heading (h2) | 20px | 600 | 1.2 | not used this phase — single flat page, no section subheadings |
| Display (h1 page title: «Склады») | 24px | 600 | 1.2 |

- Weights: 400 (regular) and 600 (semibold) only.
- No numeric columns this phase (no money/quantity fields on `Warehouse`) — `.num`/tabular-nums treatment does not apply.
- No italics, no uppercase-tracking, no color-coded rows (unchanged project convention).

Identical to Phases 2–7 — no new colors introduced. Semantic color budget stays at exactly one destructive color.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#fafafa` | Page background (existing); text `#222` |
| Secondary (30%) | `#ffffff` | Table surface, nav bar; borders `#d9d9d9`; table header background `#f0f0f0` |
| Accent (10%) | `#2563eb` | See reserved list below; hover/active shade `#1d4ed8` |
| Destructive | `#b91c1c` | «Удалить» button (`button.danger`), the last-active-warehouse warning block (`.error-block`, text/border `#b91c1c`, background `#fdecec` — reuses the EXACT oversell/price-floor warning treatment, not a new "warning" color per project convention) |

Accent reserved for: the new «Склады» nav link (including its active state, accent + 600 weight when path starts with `/warehouses` — same pattern as every other nav entry), primary submit button («Добавить склад», «Сохранить склад»), input `:focus` border/outline. Nothing else — no new accent usage this phase.

**No color coding for active vs. deleted rows beyond the existing `.muted` treatment:** a soft-deleted warehouse row uses `class="muted"` (existing gray `#6b7280` text tint, already used elsewhere for placeholder/empty-state text) — not a new gray, not a background tint, not a badge. This matches the "no new color role" constraint and keeps the deleted-row distinction legible without inventing a fifth color.

Muted text (`.muted`, `#6b7280`): deleted-row text, the «(необязательно)» marker on the «Адрес» label, empty-state body text (if ever shown).

## Interaction Contract

- **Every write response re-renders `partials/warehouse_rows.html` — never `HX-Redirect`.** Unlike `Product` (which has a separate `/products/{id}/edit` page to redirect to/from), Warehouses is a single settings-style page (D-08); add, edit, delete, AND restore all target `#warehouse-rows` with `hx-swap="outerHTML"`. Do not copy `products.py`'s `HX-Redirect` response shape.
- **Deleted rows stay visible (D-09) — `list_warehouses` never filters by `deleted_at`.** The template alone distinguishes active vs. deleted (muted row + swapped action button). This is a deliberate, documented deviation from every other soft-deletable list in the app (`Product`'s list filters deleted rows out) — do not "fix" it to match Product's convention.
- **Last-active-warehouse delete is warn-but-allow, not a hard block (D-06):** a `confirm != "1"` gate computes the active-warehouse count BEFORE any write; if the target is the last active row, return zero writes and render the inline warning row directly under that warehouse's row (mirrors `register_sale`'s existing oversell/price-floor confirm-gate shape, transposed to a single-row delete — RESEARCH.md Pattern 3). A second click with `confirm=1` performs the delete regardless. This guard governs deletion only — it does not (and per D-07, cannot) guarantee "at least one active warehouse always exists" as an invariant Phase 9 can rely on without its own defensive check.
- **Keyboard-first (core value: fast operator entry) still applies:** the add-row «Название» input is autofocused; Tab order follows visual order (Название → Адрес → Добавить склад button → each row's inputs/buttons in table order); every form submits on Enter; visible 2px accent focus outline on every interactive element — never `outline: none` without replacement.
- **No client-side JS beyond vendored HTMX.** No modals, no toast libraries; the delete confirmation uses the native `hx-confirm` dialog (same as Product's delete) for the FIRST click; the warn-but-allow SECOND click (only reachable after the last-active-warehouse warning renders) is a plain `hx-post` with `hx-vals='{"confirm": "1"}'`, no additional native confirm dialog stacked on top (mirrors the sale-warning precedent, which also does not double-confirm).
- **Manual UAT gates:** deleting a non-last warehouse succeeds immediately (native `hx-confirm` only, no warning row); deleting the LAST active warehouse shows the inline warning and performs zero writes until «Удалить всё равно» is clicked; a restored warehouse reappears in its active (non-muted) styling with a «Удалить» button again; the migration seeds exactly one default warehouse (`Склад по умолчанию`, id `00000000-0000-4000-8000-000000000010` per RESEARCH.md) and `/warehouses` shows it on first load with no setup step required.
