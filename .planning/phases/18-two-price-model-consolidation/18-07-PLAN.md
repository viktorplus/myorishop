<objective>
Add the reference-deviation colour cue (PROD-06 / criterion 3) as a purely ADDITIVE layer — nothing in this plan changes stored data or money math. The cue is one delegated `document.addEventListener('input')` in a new `app/static/price-cue.js` (D-10) that reads `data-ref-cents` off the field and toggles a below/above class; a keystroke round-trip is forbidden because it would destroy focus/caret and the basket `price[]` inputs have no `id` (D-12), and the cue never parses money for submission — `parse_optional_cents` stays the sole authority and the server re-renders the authoritative cue (D-13). Visual form is border + soft tint + text badge (colour alone fails WCAG 1.4.1), with the blue fill deliberately NOT `#e8effd` (the search-match/selection token at 6 sites) (D-14). This is not the app's first hand-written JS (D-11): `hx-on:` appears 42 times and `base.html:6-14` already carries an inline script; the cue holds no client-side state, so Alpine.js's deferred caveat is not triggered.

This plan lands the mechanism (js + css + both base script tags) and wires the PRODUCT CARD. Receipt and sale surfaces are wired in plan 18-08. The reference source is the fixed lookup from plan 18-01 and the `latest_price` already in the card context (`products.py:244`) — no new query for the static card; `product_new?code=` (plan 18-05) resolves the reference for the new-form path.

Purpose: PROD-06 — typed ДЦ/ПЦ deviating from the reference is visually cued.
Output: `price-cue.js`, cue CSS tokens, script tags in both bases, and data-ref-cents on the card's two prices (never min_sale), OOB-safe.
</objective>
