<objective>
Complete the cue by wiring `data-ref-cents` onto the remaining price surfaces — the receipt form (desktop + the mobile wizard) and the sale form (desktop basket + the mobile sale wizard) — reusing the `price-cue.js` listener and CSS from plan 18-07. The reference comes from plan 18-01's `reference_prices_for_code` (ДЦ = consultant_cents, ПЦ = consumer_cents), threaded server-side into each fragment.

Two research findings shape this plan. (1) `receipt_price_inputs.html` is the single source for BOTH the static and OOB receipt-price render paths, so adding one optional `ref_cents` param there (mirroring the file's own `oob` param, 18-PATTERNS.md §Q2) covers both paths at once — which is exactly why Pitfall 2 does not bite the receipt fields. (2) D-20: mobile has NO product card and NO dictionary page — its only two price surfaces are the receipt wizard and the sale wizard. This plan wires exactly those two mobile surfaces; it does NOT build a mobile product card (that would be Phase 19 territory and nothing in PROD-05/06/07 asks for it). This interpretation is stated so `/gsd-verify-work` does not read criterion 1 literally and fail the phase for a page that never existed.

The criterion-3 visual behaviour (yellow/blue/neither) cannot be automated — TestClient runs no JS — so this plan ends with a `checkpoint:human-verify` covering all surfaces on both platforms.

Purpose: PROD-06 — every ДЦ/ПЦ input across receipt and sale is cued against the reference.
Output: `ref_cents` on the receipt partial, data-ref-cents on all receipt + sale price inputs (desktop + mobile), reference threading in the receipt/sale routes, and a human visual sign-off.
</objective>
