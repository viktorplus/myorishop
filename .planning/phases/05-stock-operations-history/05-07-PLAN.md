<objective>
Close three non-blocking reliability warnings from `05-REVIEW.md` (surfaced via `05-VERIFICATION.md`'s Anti-Patterns table) that directly touch the project's stated Core Value — "reliably record ... without losing any data" — even though today's happy-path tests don't exercise them:

- **CR-02:** `GET /returns`'s "origin not found" error partial returns `status_code=404`, but `base.html`'s htmx `responseHandling` config only allow-lists `204`/`422` for swapping — every other 4xx/5xx (including 404) gets `swap:false,error:true`, so the RU error message («Исходная продажа не найдена.») is silently discarded. The operator clicks «Вернуть» on a stale/invalid link and nothing visibly happens.
- **CR-03:** `POST /returns`'s bare `except Exception:` handler re-queries the same SQLAlchemy session (via `_origin_context`) without calling `session.rollback()` first. Per SQLAlchemy semantics, a session whose flush/commit failed is left needing an explicit rollback before further queries — any further query raises `PendingRollbackError`, which is NOT caught anywhere, so an unexpected exception (e.g. a locked/busy SQLite file) could itself crash with an unhandled 500 instead of the graceful error page the `# noqa: BLE001` comment promises.
- **WR-03:** The identical bare-`except Exception` pattern in `app/routes/corrections.py` and `app/routes/writeoffs.py` never calls `session.rollback()` either. It's harmless TODAY only because neither error-context builder happens to re-query the DB — but it is the same systemic gap that caused CR-03, and a future edit to either (e.g. adding a "current stock" hint on save-failure) would silently reintroduce the crash.

Deferred, NOT addressed by this plan (explicitly out of scope, per gap-closure guidance to stay tight and low-risk):
- **WR-01** (history "Показать ещё" pagination control stranding) — cosmetic UX-only bug requiring a genuine hx-swap/hx-target restructure of `history_rows.html`, higher risk of introducing new pagination bugs for a non-blocking cosmetic issue. Deferred to a future UI-polish pass.

Purpose: Harden the three write routes (return/correction/write-off) against the same class of "broken session after an unexpected error" failure, protecting the append-only ledger's reliability guarantee.
Output: `app/routes/returns.py`'s error path uses the sitewide 422 convention and rolls back before re-querying; `corrections.py`/`writeoffs.py` gain the same defensive rollback; two new regression tests in `tests/test_returns.py` prove both fixes.
</objective>
