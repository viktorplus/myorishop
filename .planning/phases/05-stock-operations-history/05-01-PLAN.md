<objective>
Establish the shared foundation for Phase 5: the two RU-label constants every
slice reads, and the Wave-0 RED test contract that fixes the interface for all
four services before any implementation exists.

Add `WRITEOFF_REASONS` (D-02/D-03 — latin `reason_code` → RU label; the exact
grouping keys Phase 6 RPT-03 will report on) and `OPERATION_TYPE_LABELS`
(D-16 — latin op type → RU label for the `/history` «Тип» column) to
`app/models.py`, next to `OPERATION_TYPES`. Register both as Jinja globals so
`history_rows.html`, the write-off dropdown, and the correction form can render
RU text while storing latin codes (project rule: RU UI text / Latin stored codes).

Then create the four Wave-0 test files named exactly to the 05-VALIDATION.md
Requirements→Test Map `-k` selectors, and extend `tests/test_ledger.py` with a
Phase-5 append-only assertion. These files are RED by design (they import
`app.services.writeoffs` / `returns` / `corrections` / `operations`, which land
in Waves 2-5) — the same failing-contract shape Phase 4 used for
`tests/test_sales.py`.

Phase user story (reformatted from the ROADMAP `**Goal:**` capability line —
not invented scope):
**As an** operator, **I want to** record every non-sale stock movement —
write-offs with a reason, sale-linked returns, and stock corrections — and
browse the complete operation history, **so that** my stock counts and audit
trail stay correct and traceable without ever editing quantities directly.

Purpose: one source of truth for reason/type labels + a frozen test contract so
Waves 2-5 are pure GREEN implementation.
Output: two constants, two Jinja globals, four RED test files, one extended test.
</objective>
