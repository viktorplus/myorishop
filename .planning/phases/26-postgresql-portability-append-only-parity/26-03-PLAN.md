<objective>
Wire `settings.database_url` through the engine builder and Alembic env, gate
every SQLite-only side effect by dialect, and stand up the GitHub Actions CI that
runs the whole PostgreSQL proof against a real `postgres:17` service. This is
where SRV-01 and SRV-02 are actually PROVEN.

Purpose: With Plan 01 (config + driver + test scaffold) and Plan 02 (branched
migrations) in place, two consumers still hardcode `sqlite:///`
(`app/db.py::build_engine`, `alembic/env.py`) and there is no CI. This plan makes
both consumers read the single URL source of truth, prevents SQLite-only
statements (`PRAGMA …`, parent-dir mkdir, `render_as_batch`) from firing on PG,
and creates the CI job that applies the full migration history to empty PG and
asserts append-only rejection + Cyrillic parity.

Output: `build_engine_from_url(url)` in app/db.py, a dialect-aware
`alembic/env.py`, and `.github/workflows/ci.yml`.

Scope fence: NO sync logic. The `VACUUM INTO` backup path
(`app/services/backup.py`) stays SQLite/client-only and is OUT OF SCOPE — do NOT
wire `create_backup`/`startup_backup` to a PG engine (RESEARCH Pitfall 4). No
change to backup.py in this phase.
</objective>

    honors the configured URL (sqlite by default, PG when `DATABASE_URL` is set).
    Leave the `APPEND_ONLY_TRIGGERS` test-fixture constant unchanged.
