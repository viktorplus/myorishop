<objective>
Retrofit an in-place `op.get_bind().dialect.name` branch into the two FROZEN
migrations that carry the append-only trigger DDL, so `alembic upgrade head` no
longer crashes at revision 0001 on empty PostgreSQL (SRV-01) and PostgreSQL
rejects UPDATE/DELETE on the two ledger tables exactly as SQLite does (SRV-02).

Purpose: `RAISE(ABORT, …)` is SQLite trigger-body syntax; it is a syntax error on
PostgreSQL. On empty PG the migration chain dies at revision 0001 long before any
later revision runs (RESEARCH Pitfall 1) — so a NEW appended migration cannot fix
it. The fix MUST live inside `0001.upgrade()` and `0013.upgrade()`.

WR-06 TENSION (call out explicitly): these migrations are normally immutable —
replaying the chain on an existing SQLite DB must produce the same history
forever. The ONLY permitted edit is ADDING a PostgreSQL branch. The
SQLite-emitted statements must stay byte-for-behavior identical, so deterministic
SQLite replay is preserved (RESEARCH Anti-Patterns / Open Question 4). Adding a
PG-only branch does not touch the SQLite path — that is the whole point.

Output: dialect-branched trigger DDL in 0001 (operations) and 0013
(cash_movements), with trigger NAMES and message substrings identical across
dialects.

Scope fence: NO sync logic; do not touch the portable `create_table`/index/seed
code (already dual-dialect). Never use `op.batch_alter_table` on the ledger tables
(batch = table rebuild = drops triggers — Pitfall 2).
</objective>
