# Phase 12 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (server-rendered Jinja2 + HTMX 2.0.10 vendored; plain CSS, no framework, no CDN) |
| Preset | not applicable |
| Component library | none — reuse existing `app/static/style.css` (established Phase 2, carried through every phase since) |
| Icon library | none — text labels only (offline-first, no icon dependency) |
| Font | system-ui stack (existing): `system-ui, -apple-system, "Segoe UI", sans-serif` |

**Locked carry-over from Phase 2/3/11 (source: `app/static/style.css`, `02-UI-SPEC.md`, `03-UI-SPEC.md`):** every token below is IDENTICAL to what's already implemented. This phase adds zero new tokens — it wires an already-established autofill pattern (debounced `hx-get` + fill-only-if-empty + OOB swap, per `app/services/receipts.py::lookup_prefill()`/`app/services/catalog.py::search_products()`) onto three more surfaces (receipt desktop, receipt mobile, sales name→code). All styling lives in `app/static/style.css`; new CSS rules only if an existing class truly cannot express the layout (this phase needs at most one small new rule for the sales dropdown list — see Interaction Contract). UI language: Russian (`base.html lang="ru"`).

shadcn is not applicable to this stack (Python/FastAPI/Jinja2/htmx, no React/Next.js/Vite) — the shadcn initialization gate was skipped per the tool-priority contract.

## Typography

Exactly 4 sizes, 2 weights — identical to Phase 2/3/11. No other values permitted.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body (inputs, table cells, paragraphs, dropdown rows, visible code/name readout lines) | 16px | 400 | 1.5 |
| Label (form labels, muted hints, autofill hint text, dropdown "no matches" text) | 14px | 400 | 1.5 |
| Heading (h2 step/section titles — none new this phase) | 20px | 600 | 1.2 |
| Display (h1 page titles — none new this phase) | 24px | 600 | 1.2 |

- Weights: 400 (regular) and 600 (semibold) only. The new visible code/name readout lines (D-12/D-13/D-14) use `<strong>` (600) around the code, 400 around the name — mirrors the existing `<th>`/label-vs-value weight convention, not a new typographic role.
- Dropdown rows (sales name→code search, D-09): body 16px/400, matched substring wrapped in `<mark>` (existing accent-tinted highlight, no new type treatment).
- No italics, no uppercase-tracking treatments.

| Autofill hint — name from dictionary (existing, reused verbatim) | «Название подставлено из справочника — можно изменить.» |
| Autofill hint — receipt, existing product (existing, reused verbatim, `CARD_FILL_HINT`) | «Данные подставлены из карточки товара — новые цены обновят карточку.» |
| Autofill hint — sales, price from batch (existing, reused verbatim) | «Цена подставлена из партии — можно изменить.» |
| Autofill hint — sales, price from product card (existing, reused verbatim) | «Цена подставлена из карточки товара — можно изменить.» |
| **Autofill hint — receipt, unknown code, catalog/dictionary source (NEW, PRICE-04 / D-01)** | «Цена и название подставлены из каталога — можно изменить.» — shown once, under the name field (same slot the existing `hint` context variable already fills for `source=="product"`), covering whichever of price/name actually filled; do not build per-field hint variants (matches D-01's "surface together" framing and avoids splintering the single-hint-per-response convention already established in `receipts.py`) |
| **Mobile receipt step 3 — visible code/name readout (NEW, D-12)** | `«{{ code }} — {{ name }}»` — code in `<strong>` (600 weight), em dash, name in regular text (400 weight); no label, no icon, no card/box wrapper. | **Mobile sales steps 2/3 — visible name readout (NEW, D-13)** | Same format as above: `«{{ code }} — {{ name }}»`, placed immediately below the step indicator (`.mobile-step-indicator`), above the step's own `<h2>` |
| **Mobile transfers — visible name readout (NEW, D-14)** | Same format, same placement, carried from the step where the name first resolves through to `transfers_step_dest.html` |
| **Sales name→code dropdown — no matches (NEW, D-08/D-10)** | «Совпадений не найдено.» — muted (14px, `#6b7280`), shown only when the 3+-char query returns zero rows; below 3 characters, the dropdown area is simply empty (no message, no request fired) |
| Error state (existing pattern, unchanged) | Inline under field, destructive color — no new error copy introduced by this phase (autofill never produces a distinct error state of its own; a failed lookup is silently a 204/no-op, same as the existing pattern) |
| Destructive confirmation | Not applicable — this phase introduces no destructive actions |

Field labels (canonical, unchanged): «Код», «Название», «Закупочная цена», «Цена продажи», «Цена по каталогу». Sales dropdown rows show raw code + name (no labels, table-less list — see Interaction Contract).

## Interaction Contract

- **Established pattern, reused verbatim everywhere in this phase:** debounced `hx-get` (`hx-trigger="input changed delay:300ms"`) on the code/name field, `hx-sync="this:replace"`, server returns `204` (htmx-config already whitelists 204-as-no-swap in `base.html`) when the target field already has operator-typed content, or a partial with `hx-swap-oob="true"` fragments otherwise. No loading spinner — local SQLite, sub-100ms; `hx-indicator` remains optional, unused, per Phase 2 precedent.
- **Receipt catalog-source fill (D-01/D-02/D-04):** the new `source=="catalog"` branch in `/receipts/lookup` must run its price fields through the SAME `fill_fields` "still empty?" computation the `source=="product"` branch already uses (Pitfall 1 in RESEARCH.md) — reuses `receipt_price_inputs.html` unchanged, same `id="{field}-wrap"` targets. `sale` is never included in `fill_fields` for this branch (D-02 hard boundary). The `#batch-chooser` is untouched (D-05).
- **Mobile receipt step 3 (D-06/D-12):** no debounced lookup added to mobile step 1 (explicitly rejected, D-06/D-07) — prices resolved once during the step-1→2 POST are threaded into step 3's `value="{{ cost }}"` etc. via cents-formatted strings (never raw `None` — see RESEARCH Pitfall 4). The new visible code/name line is plain server-rendered text, present unconditionally on step 3 (not just when autofill fired), since the hidden `code`/`name` inputs already carry the data regardless of source.
- **Sales name→code dropdown (D-08/D-09/D-10/D-11):**
  - Trigger: `hx-get="/sales/search-name"` on the `name[]` input, `hx-trigger="input changed delay:300ms"`, fires only once 3+ characters are typed (D-10) — enforced server-side in the new route (`q.strip()` length check), not by modifying `search_products()` (RESEARCH Pitfall 5).
  - Target: a persistent, initially-empty `<div id="name-dropdown-{{ row_id }}">` rendered inside the SAME `<td>` as the name input on first row render. The lookup response does a normal (non-OOB) `hx-target`/`hx-swap="innerHTML"` swap into this div — because the target already exists in the live DOM as part of the row's own markup, this sidesteps the `<template>`-wrapping requirement that applies only to OOB-injected `<td>`/`<tr>` elements (Pattern 2 / RESEARCH Open Question 1 — this is the recommended resolution).
  - Row shape: reuse the `<mark>`-highlighted two-segment rendering already used by `product_rows.html` (via `split_match()`), but as a plain list, not a table — each match is a `<li><button type="button">…</button></li>` (native keyboard/tab focusable, existing 2px accent `:focus-visible` outline applies automatically). List style: `list-style: none; margin: 4px 0 0; padding: 0; border: 1px solid #d9d9d9; background: #ffffff;` (this is the one small new CSS rule this phase may add — all values are on-scale); each `<li> button` full width, `padding: 8px`, hover/focus background `#e8effd` (existing mark tint, not a new color).
  - Selection (D-11): each result button carries an inline `hx-on:click` handler (same zero-external-JS convention already used in `sale_row.html`'s row-delete button) that sets both the code and name input values directly from the clicked row's own data and clears the dropdown div — no re-trigger of `/sales/lookup`.
  - Zero results (3+ chars, no match): render the dropdown div with the muted «Совпадений не найдено.» line instead of an empty div, so the operator gets feedback that the search ran.
  - No dedicated close/dismiss control this phase — the dropdown clears on selection; clicking away without selecting leaves the last result set visible until the next keystroke changes it. This is an accepted minimal-scope tradeoff (zero-JS constraint, no blur-close wiring) — not a blocking gap.
- **Keyboard-first (unchanged core value):** every existing tab order/autofocus/Enter-submit behavior is preserved; the new dropdown's `<button>` rows are natively tabbable, consistent with the rest of the app's zero-custom-ARIA approach.
- **No client-side JS beyond vendored htmx** (2.0.10) plus small inline `hx-on:click` attribute handlers, matching existing precedent (`sale_row.html` row deletion). No modals, no toast libraries, no new client-side state.
