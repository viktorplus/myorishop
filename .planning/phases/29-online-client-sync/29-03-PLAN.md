<objective>
Build the network heart of client sync: one injectable driver reused by the manual
button (Plan 04) and the background tick (Plan 05). This is the correctness core —
the algorithms (wire format, merge, cursor) already exist and are tested in Phases
27-28; the risk here is the two topology decisions D-13 and D-14, plus the
offline-safe contract (SYNC-06 / SRV-03).

Delivered (all in `app/services/sync_client.py`):
- `run_sync_once(session, *, client)` — the ONE driver. Injecting the `httpx.Client`
  is the test seam: tests pass `httpx.ASGITransport(app=main.app)` (real server + real
  merge, in-process) or `httpx.MockTransport` (offline/5xx/401). It (1) collects
  unsynced `Operation` + `CashMovement` rows, (2) collects the D-13 reference closure,
  (3) serializes via `merge.serialize_exchange`, (4) POSTs to `/api/sync/push` with the
  Bearer token, (5) stamps `synced_at` ONLY after `raise_for_status()`, then (6)
  paginates `/api/sync/pull` and applies each page with the D-14 client upsert.
- The D-13 push closure: the FK parents (`product`/`customer`/`batch`/`sale` +
  `warehouse`) the unsynced ledger rows transitively reference — because an offline
  sale/receipt creates local `Sale`/`Batch` rows and pushing operations without their
  parents FK-fails the server's all-or-nothing merge (`test_push_all_or_nothing`).
- The D-14 client reference upsert: insert-if-new AND UPDATE-existing (server wins on
  the client) — the OPPOSITE of `merge._upsert_reference` (which discards existing).
  This is the one genuinely new algorithm in the phase.
- `run_sync_tick()` — fresh Session + `httpx.Client` + `_run_lock` + `run_sync_once`,
  recording the result in a `finally`; the Plan-05 loop calls this.
- `_run_lock = threading.Lock()` — the D-09 single-run guard shared by both callers.

Purpose: SYNC-01 (push+pull), SYNC-06 (offline never blocks), SRV-03 (local work
untouched). Output: the driver + the network half of `tests/test_sync_client.py` +
the ASGITransport/device-token test fixture.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → central server (internet) | Push/pull cross the public network; token + payload in flight |
| server response → client DB merge | A tampered/malicious NDJSON response must be validated before any DB write |
| network failure → local app | An offline/timeout must never block or corrupt local SQLite work |
| local write path → network | Local receipts/sales must never call out (SRV-03) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-04 | Information Disclosure | Bearer `sync_token` transmission | mitigate | Token only in the `Authorization: Bearer` header (never a query string / never logged); read from `.env` (Plan 01). |
| T-29-05 | Tampering | malicious/garbled server pull response | mitigate | `merge.parse_exchange` validates every pulled page (bad money/ids/format rejected) BEFORE any DB touch; each page applied in one owned transaction that rolls back on error (V5). |
| T-29-06 | Tampering (integrity) | failed push falsely marked synced | mitigate | Stamp `synced_at` ONLY after `resp.raise_for_status()` (Pitfall 3); merge idempotency makes a re-push harmless. |
| T-29-10 | Denial of Service / availability | unreachable server hangs the app | mitigate | Strict `httpx.Timeout(connect=3, read=10, write=10, pool=3)` (D-05); any `httpx.HTTPError` caught → `SyncResult`, never raised (SYNC-06). |
| T-29-11 | Tampering | plain-HTTP push over the internet | mitigate | `sync_server_url` MUST be `https://…` for internet deployment (Caddy TLS from Phase 28); documented operational requirement. |
| T-29-12 | Elevation / concurrency | manual click + tick double-sync | mitigate | Module-level `threading.Lock` acquired non-blocking in both paths (D-09). |
</threat_model>
