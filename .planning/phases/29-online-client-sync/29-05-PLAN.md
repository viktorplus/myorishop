<objective>
Complete SYNC-08: the optional interval auto-sync and its Settings control. The
manual button (Plan 04) already works; this makes sync happen on a timer when the
operator opts in, and stop attempting silently when offline.

Delivered:
- `app/main.py` (EXTEND) — the existing `lifespan` gains an asyncio background task
  (D-06: zero-dependency, keeps syncing with the browser tab closed). It is started
  AFTER `startup_backup()` (which stays) and cancelled cleanly on shutdown
  (`task.cancel()` + suppress `CancelledError` — D-08). The loop reads the toggle +
  interval FRESH each tick from `sync_state` (D-08/D-15 — flipping the toggle takes
  effect next tick), and when enabled offloads the blocking `run_sync_tick` off the
  event loop via `anyio.to_thread.run_sync` with a fresh Session per tick (D-07).
  Each tick body is wrapped in a broad `try/except` so being offline just skips a
  tick silently — no log spam, the loop never dies (D-08).
- `app/routes/settings.py` + `app/services/settings.py` + `pages/settings.html`
  (EXTEND) — a «Синхронизация» section on the admin Settings page (D-03): an
  auto-sync on/off checkbox (default OFF), an interval-seconds number input
  (default 300, min 60 / max 3600), and a save button that persists `auto_enabled`
  + `auto_interval_seconds` to the single-row `sync_state` (D-15), clamping the
  interval in the service.

Purpose: SYNC-08 (optional interval auto-sync, silent offline stop, disabled =
manual only). Output: the lifespan loop + the Settings config form + tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| background loop → local DB + network | The timer must not run the sync Session on the event loop or corrupt local work |
| admin Settings form → sync_state config | Interval must be bounded; token must never be displayed |
| offline network → background loop | An offline tick must fail silently and never crash the loop |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-10 | Denial of Service / availability | background loop stalls the app | mitigate | Blocking driver offloaded via `anyio.to_thread.run_sync` (D-07); driver bounded by `httpx.Timeout` (D-05); tick errors swallowed so the loop never dies (D-08). |
| T-29-14 | Denial of Service | too-frequent interval hammering the server | mitigate | Interval clamped 60..3600 s in `save_autosync_config` (D-15); default 300 s. |
| T-29-13 | Access Control | who can change auto-sync config | mitigate | `POST /settings/sync` inherits the settings router's `require_role("administrator")` gate; CSRF via the base body `hx-headers`. |
| T-29-07 | Information Disclosure | server address / token on Settings | mitigate | Only the non-secret `sync_server_url` is displayed read-only; `sync_token` is NEVER rendered (V7). |
| T-29-12 | Concurrency | tick overlapping a manual click | mitigate | `run_sync_tick` acquires the shared `_run_lock` non-blocking (D-09) — a tick during a manual sync silently skips. |
</threat_model>
