<objective>
Put sync in front of the operator. Everything from Plans 01-03 is wired into a
glanceable header surface and a manual button.

Delivered:
- `partials/sync_status.html` (NEW) — the OOB partial, copied from the proven
  `cash_balance.html` idiom (D-02): two id'd spans `#sync-status` (message +
  last-sync line) and `#sync-badge` (count), each carrying `hx-swap-oob="true"`
  when rendered with `oob=True`. The SAME template serves first paint (oob unset)
  and the OOB refresh.
- `POST /sync/run` (NEW, in the existing `app/routes/sync.py`) — a plain `def`
  handler (threadpool, D-04): acquire `_run_lock` non-blocking, run
  `run_sync_once`, format the RU D-12 string, render the partial with `oob=True`,
  and ALWAYS return 200 (never a 5xx on network failure — SYNC-06). It lives on
  the app-level `auth_guard` (it is NOT under the `/api/sync/` token-bypass prefix),
  so a normal logged-in operator can sync (an operator action, not admin-only).
- `base.html` (EXTEND) — the «Синхронизировать» nav link (`hx-post="/sync/run"`,
  `hx-indicator`) + `{% include "partials/sync_status.html" %}` inside `<nav>`, so
  status + badge are visible on every desktop page for both roles (D-01; Settings
  is admin-gated so status must NOT be Settings-only).
- `app/routes/__init__.py` (EXTEND) — a Jinja context processor that injects
  `sync_message`, `last_sync_line`, and `unsynced` into EVERY page render (mirrors
  the existing `_auth_context`), so `base.html`'s first paint has the current
  status + badge without every route passing them.

Purpose: SYNC-01 (the manual trigger), SYNC-06 (visible status + non-blocking
failure), SYNC-07 (the badge). Output: the route + partial + nav wiring + UI tests.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| driver result → rendered HTML partial | Only fixed D-12 strings + integer counts may reach the DOM; never token/exception bytes |
| operator session → `/sync/run` | A logged-in operator (not just admin) may trigger sync; must stay behind the app-level auth_guard |
| network failure → HTMX swap | A failure must return a swappable 200, not a non-swapping 4xx/5xx |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-07 | Information Disclosure | `/sync/run` HTML response | mitigate | Renders only the LOCKED D-12 fixed strings + integer counts; never the token or raw server error bytes (V7, UI-SPEC accessibility note). |
| T-29-13 | Spoofing / Access Control | `/sync/run` exposure | mitigate | Route is NOT under the `/api/sync/` token-bypass prefix, so the app-level `auth_guard` protects it; CSRF carried by the existing body `hx-headers` (AUTH-05). |
| T-29-10 | Denial of Service | non-blocking failure | mitigate | Handler always returns 200 with the OOB partial; driver timeout bounds the request (D-05); page stays interactive (SRV-03). |
| T-29-12 | Concurrency | double-sync from a fast double-click | mitigate | `_run_lock.acquire(blocking=False)` → `locked` partial (`Синхронизация уже выполняется`, D-09). |
</threat_model>
