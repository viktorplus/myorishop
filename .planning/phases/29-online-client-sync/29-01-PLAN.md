<objective>
Lay the data + config + dependency foundation for online client sync. This plan
adds nothing user-visible; it makes the rest of Phase 29 buildable and portable.

Four things land here:
1. Promote `httpx` from a dev-only to a runtime dependency (the sync driver in
   Plan 03 imports it at app runtime; a `uv sync --no-dev` deploy would otherwise
   `ImportError`).
2. Add two `Settings` fields — `sync_server_url` and `sync_token` — resolved from
   `.env` only (the token is a secret; never store it in the synced DB — it would
   leak on a `myorishop.db` copy). Both blank by default so sync is a no-op until
   configured (offline-first, SRV-03).
3. Declare the single-row `SyncState` model (D-10 result columns + D-15 auto-sync
   config columns = five columns total) and two partial indexes on
   `Operation.synced_at` / `CashMovement.synced_at` that keep the D-11 unsynced
   badge query cheap. Declare in `app/models.py` (so `create_all` builds them for
   tests) AND in migration `0020` (so Alembic builds them for real DBs) — kept in
   lockstep (Pitfall 5).
4. Extend `tests/test_pg_parity.py` to prove the new table + partial indexes build
   on PostgreSQL under the one shared migration history (SRV-01 portability holds).

Purpose: everything else in Phase 29 depends on the `sync_state` table, the
partial indexes, and the two config fields.
Output: a runtime httpx dep, two config fields, one model + two indexes, one
portable migration, and a green PG-parity extension.
</objective>
