# Phase 6 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (server-rendered Jinja2 + HTMX 2.0.10 vendored; plain CSS, no framework, no CDN) |
| Preset | not applicable |
| Component library | none — reuse existing `app/static/style.css` (spec scale established in Phase 2, carried through Phases 3–5) |
| Icon library | none — text labels only (offline-first, no icon dependency) |
| Font | system-ui stack (existing): `system-ui, -apple-system, "Segoe UI", sans-serif` |

**shadcn gate:** not applicable — stack is FastAPI + Jinja2 + HTMX (not React/Next.js/Vite); no `components.json` found in the repo. Registry safety gate: not applicable.

**Locked carry-over from Phases 2–5 (source: 02–05-UI-SPEC.md, `app/static/style.css`):** every token below is the SAME as Phases 2–5 — this phase adds screens (`/reports`, `/reports/sales`, `/reports/stock`, `/reports/writeoffs`, `/reports/products`, `/export`, plus two new optional fields on the product form), NOT new tokens. All styling lives in `app/static/style.css`; new CSS rules are added ONLY where existing classes cannot express a layout, and stay strictly on the scale/palette below. UI language: Russian (`base.html lang="ru"`). This phase is 100% read-only over the ledger (RESEARCH.md domain note) — no new write paths, so no new `hx-confirm`/destructive-button patterns beyond what already exists. Jinja autoescape ON — product/customer names render untrusted in every report and CSV row; **never `|safe`** (Phase 4/5 precedent). Money is integer cents rendered via the existing `cents` filter (comma decimal); timestamps via `local_dt`. CSV rows are written with `;` delimiter and `utf-8-sig` encoding per RESEARCH.md D-07/Pitfall 4 — a backend concern, not a template concern, but noted here because it affects what "correct" looks like when the operator opens an exported file.

Declared values (must be multiples of 4). Identical to Phases 2–5 — already implemented in `style.css`.

| Token | Value | Usage |

| sm | 8px | Gaps inside form rows; table cell padding (8px 12px); button vertical padding; nav padding; filter-bar/preset-bar inner gap |
| md | 16px | Default gap between form fields (`.field`); button horizontal padding; error-block padding; gap between preset buttons and the от/по date inputs |

| 2xl | 48px | Empty-state block vertical padding (`.empty-state`) — reused for every "no data for this period" state |

Exceptions: 12px — dense table cell horizontal padding only (existing `th, td { padding: 8px 12px }`, carried from Phase 2). Control heights derived from scale: inputs/buttons/selects/date inputs 36px (8px vertical padding + 16px text + borders) — the от/по `<input type="date">` fields and the `/export` download links (styled as `a.button`) reuse this same height, no new sizing.

Exactly 4 sizes, 2 weights — identical to Phases 2–5. No other values permitted.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body (inputs, date inputs, table cells, paragraphs, CSV download link text) | 16px | 400 | 1.5 |
| Label (form labels, preset-button text, filter labels, table headers, muted meta: «по умолчанию: N», period sub-labels, cost-unknown-lines caveat) | 14px | 400 (labels/hints/caveat) / 600 (table `th`, active preset button) | 1.5 |
| Heading (h2 section titles: «Итоги за период», «По товарам», «Мало на складе», «Топ продаж», «Залежавшиеся товары», «Списания по причинам») | 20px | 600 | 1.2 |
| Display (h1 page titles: «Отчёты», «Продажи и прибыль», «Остатки склада», «Списания», «Топ и залежавшиеся товары», «Экспорт данных») | 24px | 600 | 1.2 |

- Weights: 400 (regular) and 600 (semibold) only.
- Numeric columns (quantities, revenue, cost, profit, stock levels, thresholds, "days since last sale"): 16px, 400, right-aligned, `font-variant-numeric: tabular-nums` (existing `.num` class) — identical treatment across every report table, no exceptions.
- Report summary totals (Продано, Выручка, Себестоимость, Прибыль on `/reports/sales`) use the same body size/weight as table cells — NOT display-sized "dashboard stat" typography. This app has no dashboard-card visual language; a report summary is a labelled row of numbers, consistent with the rest of the UI's plain-table aesthetic.
- No italics, no uppercase-tracking treatments, no color-coded numbers (profit is not shown in green, low stock is not shown in red — see Color section).

## Color

Identical to Phases 2–5 — no new colors introduced. Semantic color budget stays at exactly one (destructive), and this phase introduces zero destructive actions (100% read-only reporting + CSV export).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#fafafa` | Page background (existing); text `#222` |
| Secondary (30%) | `#ffffff` | Tables, form surfaces, nav bar, filter/preset bar; borders `#d9d9d9`; table header background `#f0f0f0` |
| Accent (10%) | `#2563eb` | See reserved list below; hover/active shade `#1d4ed8` |
| Destructive | `#b91c1c` | Not used by any new screen this phase (no destructive actions exist in Phase 6) — retained in the palette for the inline validation-error pattern only (invalid custom date range) |

Accent reserved for: the **active/selected preset button** on the period filter (Сегодня / Неделя / Месяц — see Interaction Contract; this is a new use of the existing `button` accent-filled treatment, not a new color role), the period filter's «Показать» submit button, the `/export` page's three CSV download links (styled `a.button`, matching the existing `a.button` treatment already defined in `style.css`), text links (nav links; active nav link = accent + 600 weight; «Вся история →»/cross-links between report pages), and input/select/date-input `:focus` outline (2px, existing rule). Nothing else.

**Unselected preset buttons use `button.secondary`** (white, `#d9d9d9` border, `#222` text) — the same neutral treatment as every other secondary action in this app (e.g. «Отмена», correction mode toggle). Only the currently-active preset (or none, if the operator has hand-edited the dates) uses the accent-filled `button` style. This is the one net-new visual pattern this phase introduces, and it stays inside the existing accent/secondary button vocabulary — no new button variant.

**No color coding for report data:** low-stock rows are NOT shown in red, profit is NOT shown in green, stale products are NOT highlighted — this app's established convention (Phase 5: "signed quantity... no color coding") extends to every report in this phase. The signal is the text/number itself (e.g. a «Мало» label in the status column, a «Никогда» value in the last-sale column), never a background or text color. This keeps the semantic budget at exactly one (destructive) and avoids inventing a "warning" color mid-project.

Muted text (`.muted`, `#6b7280`): the cost-unknown-lines caveat on `/reports/sales` («N продаж без указанной себестоимости — прибыль не учитывает их»), the «по умолчанию: N» hint beside the two new product-form threshold fields, empty-state body text, date-input placeholder/format hints, «(необязательно)» markers on the threshold fields, and the "last sale: never" muted note.

Success feedback: not applicable this phase — there is no save action to confirm (read-only reports; CSV export is a direct file download, not a form submit with a success message). The one exception, saving the two new threshold fields on the product form, reuses the EXISTING product-save success/error pattern verbatim — no new copy needed there.

## Interaction Contract

- **Period filter, one code path (D-01, mandatory):** preset buttons (`hx-get` or plain links — Claude's discretion, but MUST resolve to the same GET with `from`/`to` query params either way, never a separate "preset mode" vs "custom mode" code path) pre-fill the от/по date inputs and immediately apply the filter; the operator can then hand-edit either date and click «Показать» to re-apply. Every report page that has a period filter (`/reports/sales`, `/reports/writeoffs`, `/reports/products` top-selling half) uses this exact same `partials/period_filter.html`, targeting that page's own results partial — mirrors `/history`'s existing `hx-get` + `hx-target` + `hx-push-url="true"` filter pattern so filtered views stay linkable/back-navigable.
- **Active-preset indication:** the server determines which preset (if any) matches the current `from`/`to` values and renders that preset button with the plain accent `button` class; the other two presets and "no preset matches" (custom dates) render as `button.secondary`. This is computed server-side in the same partial, not via client-side JS.
- **Week/month preset semantics (RESEARCH A1):** «Неделя» = current ISO week (Monday–Sunday, local calendar per `settings.display_tz`); «Месяц» = current calendar month. Both compute `from`/`to` as local dates and pass through the SAME `local_day_bounds_utc()` helper as «Сегодня» (RESEARCH Pattern 1) — this is a backend concern, listed here only because it determines what dates the от/по inputs get pre-filled with.
- **Custom date validation (Security V5, RESEARCH):** invalid or out-of-order dates never reach the query layer uncaught — a malformed `from`/`to` falls back to a safe default (today) or renders the inline error copy above the от/по row, per the Copywriting Contract; zero uncaught exceptions, zero raw 500s shown to the operator.
- **Stock/stale reports (no period filter):** `/reports/stock` and the stale half of `/reports/products` are plain full-page GETs — no HTMX partial swap needed since there's no filter to re-apply in place. Keep them simple; do not add HTMX machinery where a static page load already satisfies the requirement.
- **CSV downloads (`/export`):** plain `<a href="/export/{entity}.csv">` links styled `a.button` — explicitly NOT `hx-get`. HTMX would attempt to swap the CSV response into the DOM as if it were an HTML fragment; a file download needs native browser `Content-Disposition: attachment` handling, which only works with a real navigation/anchor click, not an HTMX-intercepted request. This is a hard interaction rule, not a style preference.
- **Product form threshold fields:** no new interaction pattern — they submit as part of the existing `POST /products` / `POST /products/{id}` form, validated and saved the same way every other optional numeric-ish field on that form already is (per Pitfall 3: an explicit `0` must be distinguishable from empty at the form-parsing layer, not just in the effective-threshold service logic).
- **Keyboard-first (core value: fast entry) still applies to the period filter:** от/по date inputs and «Показать» are reachable by Tab, «Показать» submits on Enter from either date input; visible 2px accent focus outline on every interactive element including the new preset buttons and date inputs — never `outline:none` without replacement.
- **No client-side JS beyond vendored HTMX.** No modals, no toasts, no charts/graphs (every report is a table or a labelled-number summary row, consistent with this app's plain-HTML aesthetic — no charting library, no SVG, no canvas).
- **Manual UAT gates:** a sale placed at 23:30 local time appears in "today's" report, not tomorrow's (D-02 correctness, the phase's core risk); clicking «Неделя» then hand-editing «по» and clicking «Показать» returns the custom range, not the preset range (Pitfall 1); `/reports/sales` shows a visible caveat line whenever any sale in the period has no recorded cost, and total profit does not silently include phantom cost-free profit for those lines (Pitfall 2); a product with `low_stock_threshold = 0` appears in the low-stock list only at exactly zero stock, never falls back to the global default (Pitfall 3); a soft-deleted product still appears in a past-period sales/write-off report but never in the stale/never-sold list (Pitfall 5); double-clicking any downloaded CSV opens with correctly split columns and readable Cyrillic in Excel (Pitfall 4 — this is the one item that needs an actual manual Excel-open check, not just an automated test, per RESEARCH A3).
