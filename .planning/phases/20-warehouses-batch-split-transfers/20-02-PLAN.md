Give WH-02 its dedicated add/edit/delete pages, and relocate WH-03's existing warn-then-confirm
/stock-blocked delete UI (D-02) from the list's inline row rendering onto the new edit page —
mirroring `app/routes/products.py`'s GET `/new`/`/{id}/edit` shape exactly.

Purpose: WH-02 requires "Add/edit/delete warehouse are reached via links that open a dedicated
form." D-01 locks this as Option B: a picker list (unchanged in THIS plan — restructured in
20-03) plus one dedicated edit/delete page and one dedicated add page.
Output: `GET /warehouses/new`, `GET /warehouses/{id}/edit` routes; `POST /warehouses`,
`POST /warehouses/{id}`, `POST /warehouses/{id}/delete` rewritten to redirect-after-POST
(mirroring Products) instead of re-rendering the list partial; two new templates.

This plan intentionally does NOT touch `app/templates/pages/warehouses.html` or
`app/templates/partials/warehouse_rows.html` (the list page/rows partial) — that cosmetic
restructure (new item_count/last_receipt columns, single "Изменить" link) is 20-03's scope.
Because this plan's route changes alone would break every test that asserts the OLD
rows-partial response shape, Task 2 below fixes/removes those tests in THIS plan so the suite
stays green even before 20-03 lands.
</objective>
