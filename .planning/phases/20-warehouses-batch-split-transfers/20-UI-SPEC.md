## Interaction & Layout Decisions

Not covered by the fixed template sections above but load-bearing for the executor. Each traces
to a specific `20-CONTEXT.md` decision or `20-RESEARCH.md` finding.

### Warehouse list (WH-01, WH-02)

1. **Preserve the entire filter/sort/status bar (RESEARCH Pitfall 1).** `warehouse_rows.html`'s
   `.filter-bar` (sort `<select>`), `.filter-row` (name/address `<input>`s, status `<select>`),
   and `partials/pagination.html` include all stay exactly as they are today. D-01's "plain
   picker table" language applies ONLY to the per-row inline edit `<input>`s and the inline
   Сохранить/Удалить/Восстановить buttons — not to the list-management chrome above the table
   (Phase 14, LIST-01..04).
2. **Table grows from 3 columns to 5.** Order: Название, Адрес, Товаров, Последняя приёмка,
   Действия. The `.filter-row` gains two blank `<th>` cells (no filter control) for the two new
   columns, to keep column alignment — WH-01 specifies no filter/sort requirement for these two
   metrics.
3. **Row cells become read-only.** Название/Адрес render as plain text (no `<input>`), Товаров
   renders the integer with `.num`, Последняя приёмка renders `{{ w.last_receipt | ru_date }}`
   or `<span class="muted">—</span>`. Действия renders exactly one link: `<a
   href="/warehouses/{{ w.id }}/edit">Изменить</a>` — no inline Сохранить/Удалить/Восстановить
   button remains in the list (D-01 rejects the two-delete-entry-point Option A).
4. **`colspan` on the stock-blocked/warning inline rows increases from 3 to 5** if any such row
   still renders inside `warehouse_rows.html` — but per Decision 6 below, those two states move
   to `warehouse_form.html` entirely, so `warehouse_rows.html` should have ZERO remaining
   `colspan`-based warning rows after this phase. If a transitional row is kept for any reason,
   update its `colspan` to match the new 5-column table.
5. **Soft-deleted rows** keep the existing `<tr class="muted">` treatment and the
   "Восстановить" action — but per D-01, restore is a list-level action (not gated behind the
   edit page, since a deleted warehouse cannot be "edited" meaningfully) and stays inline as a
   button in the Действия cell for `status=deleted`/`status=all` rows only. This is the ONE
   inline action D-01 does not relocate, because it is not "edit" or "delete" — it is
   restoration of a previously-deleted row back into the active list, and the existing
   Восстановить button has no warn-then-confirm state to relocate.

### Warehouse dedicated add/edit page (WH-02, WH-03)

6. **`warehouse_form.html` structurally mirrors `product_form.html`** (verified template):
   `<h1>` heading (Decision list above) → error block → `<form class="stacked-form"
   method="post" action="...">` with `name`/`address` fields → `.form-actions` (Сохранить
   склад / Отмена) → **destructive zone**, visually separated by the existing 24px
   `.form-actions` margin-top, containing the delete button and both warn states.
7. **Delete interaction (resolves CONTEXT.md's "Claude's Discretion" on redirect vs. in-place):**
   use HTMX in-place swap for the two non-terminal states, full-page redirect only on terminal
   success — because the item-count/last-receipt metrics on `/warehouses` must refresh after a
   successful delete, and Products' delete pattern (`HX-Redirect` unconditionally) does not fit
   here since Warehouses has TWO possible non-terminal responses first:
   - Click "Удалить" → `hx-confirm` (Copywriting Contract) fires → `hx-post
     /warehouses/{id}/delete` targets a new wrapper `<div id="warehouse-delete-wrap">` on the
     edit page, `hx-swap="outerHTML"`.
   - Stock-blocked response: swap renders the `.error-block` message in place; form and delete
     button stay visible; operator can still edit fields.
   - Last-active-warning response: swap renders the existing warning card (heading + body +
     "Удалить всё равно" / "Отмена" buttons) in place, same copy as today, same
     no-stacked-`hx-confirm` rule on "Удалить всё равно".
   - Terminal success (either first-attempt success at zero stock and not last-active, or
     `confirm=1` re-POST success): `Response(status_code=200, headers={"HX-Redirect":
     "/warehouses"})` — mirrors `product_delete`'s pattern exactly for the ONE case that is
     truly terminal.
8. **`GET /warehouses/{id}/edit` for an unknown id → `HTTPException(404)`** (mirrors
   `product_edit`'s pattern per `20-RESEARCH.md` Code Examples — `update_warehouse`'s existing
   `WAREHOUSE_NOT_FOUND_ERROR` only covers the POST path today, the GET route needs its own
   explicit lookup+404).

### Transfer form override fields (XFER-01)

9. **Override fields render unconditionally once a source batch is selected** (adopts
   `20-RESEARCH.md` Assumption A2 as the locked contract — simpler than gating on which
   destination is picked, and D-06 only requires them to be FILLED for the same-warehouse case,
   not merely visible). Placement in `transfer_batch_wrap.html`: batch picker table → destination
   warehouse `<select>` (D-09: now includes the source warehouse as a selectable option) → the
   two new override fields (expiry, then comment) → close of `.field` wrapper. This ordering
   reads as "what → where → how it changes," matching the form's existing top-to-bottom
   code→name→qty→batch flow.
10. **`_dest_warehouses()` no longer excludes the source warehouse (D-09)** — the destination
    `<select>` now always includes it as a normal `<option>`, positioned in whatever order
    `active_warehouses()` already returns (no special "same as source" labeling needed; the
    warehouse's own name in the list is self-explanatory).
11. **D-06 validation error renders in the existing `errors.form` slot** (Copywriting Contract
    above) — no new error UI element. Position in `register_transfer`'s validation order:
    immediately where today's unconditional `SAME_WAREHOUSE_ERROR` check sits, BEFORE the
    oversell check (RESEARCH Pitfall 4) — same-warehouse-blank-overrides must never reach the
    oversell warn screen.
12. **Mobile parity (RESEARCH Pitfall 3, resolved as in-scope):**
    `mobile_partials/transfers_step_dest.html` gets the same two override fields, placed after
    the existing "Количество" field and before `.mobile-actions` — using the SAME copy as
    desktop (Copywriting Contract), rendered as standard `stacked-form` fields (not
    `.mobile-card` radio-style, since these are free-text/date inputs, not a pick-one list).
    `mobile_transfers.py`'s own duplicate `_dest_warehouses()` gets the identical D-09 fix so
    the source warehouse becomes a selectable radio option there too.
13. **Legacy/NULL date-and-comment display in the batch picker is unaffected** — this phase adds
    NEW input fields, it does not change how `batch_picker.html` displays an existing batch's
    expiry/comment (guard pattern at `batch_picker.html:52,55-59` stays as-is).
