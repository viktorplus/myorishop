<objective>
Extend `register_transfer` (the sole write path for XFER-01) so a same-warehouse "split" is a
first-class outcome (D-05), gated by a new required-override validation (D-06), with an
explicit strip-then-ternary override-or-inherit discipline for the new expiry/comment params
(D-07) — and fix the pre-existing D-11 debt where the success return dict echoed the caller's
raw qty string instead of the actual transferred integer.

Purpose: XFER-01 requires "transferring part of a batch whose moved portion has a different
expiry date or condition... creates a new destination batch and moves only that portion into
it, leaving the source batch's remaining quantity and attributes unchanged." D-05 locks the
"one unified form" design: no new route, no new service — this plan extends the existing
`register_transfer` service function only.
Output: `register_transfer` gains two new optional keyword params (`new_expiry`, `new_comment`)
and its return dict gains a `"qty"` key. This is the interface every downstream plan (desktop
routes, mobile routes, templates) depends on — it must land first (wave 1).
</objective>
