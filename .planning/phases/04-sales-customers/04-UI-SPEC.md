# Phase 4 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (server-rendered Jinja2 + HTMX 2.0.10 vendored; plain CSS, no framework, no CDN) |
| Preset | not applicable |
| Component library | none — reuse existing `app/static/style.css` (spec scale established in Phase 2, carried through Phase 3) |
| Icon library | none — text labels only (offline-first, no icon dependency) |

**shadcn gate:** not applicable — stack is FastAPI + Jinja2 + HTMX (not React/Next.js/Vite); no `components.json`. Registry safety gate: not applicable.

**Locked carry-over from Phases 2–3 (source: 02-UI-SPEC.md, 03-UI-SPEC.md, `app/static/style.css`):** every token below is the SAME as Phases 2–3 — this phase adds screens (sale basket, customer CRUD, customer history), NOT new tokens. ## Spacing Scale

Declared values (must be multiples of 4). Identical to Phases 2–3 — already implemented in `style.css`.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Gap between label and input; error/hint text top margin; chip inner gap |
| sm | 8px | Gaps inside form rows; table cell padding (8px 12px); button vertical padding; nav padding; chip padding |
| md | 16px | Default gap between form fields (`.field`); button horizontal padding; error/warning-block padding; gap between customer search and basket |
| lg | 24px | Gap between page sections (basket → «Последние продажи»; customer detail → «История покупок»); `.form-actions` top margin |

Exceptions: 12px — dense table cell horizontal padding only (existing `th, td { padding: 8px 12px }`; multiple of 4, carried from Phase 2). Control heights derived from scale: inputs/buttons 36px (8px vertical padding + 16px text + borders). Basket-line inputs reuse the same 36px control height.

## Typography

Exactly 4 sizes, 2 weights — identical to Phases 2–3. No other values permitted.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body (inputs, table cells, paragraphs, chip text) | 16px | 400 | 1.5 |
| Label (form labels, table headers, muted meta: dates/hints/«(необязательно)») | 14px | 400 (labels/hints) / 600 (table `th`) | 1.5 |
| Heading (h2 section titles: «Последние продажи», «История покупок», «Новый покупатель») | 20px | 600 | 1.2 |
| Display (h1 page titles: «Продажа», «Покупатели») | 24px | 600 | 1.2 |

- Weights: 400 (regular) and 600 (semibold) only.
- Numeric columns (quantity, unit price, sum, profit): 16px, 400, right-aligned, `font-variant-numeric: tabular-nums` (existing `.num` class).
- No italics, no uppercase-tracking treatments.

## Color

Identical to Phases 2–3 — no new colors introduced. Semantic color budget stays at exactly one (destructive).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#fafafa` | Page background (existing); text `#222` |
| Secondary (30%) | `#ffffff` | Tables, form surfaces, nav bar, customer chip; borders `#d9d9d9`; table header background `#f0f0f0` |
| Accent (10%) | `#2563eb` | See reserved list below; hover/active shade `#1d4ed8` |
| Destructive | `#b91c1c` | Inline validation errors, error blocks (bg `#fdecec`), the oversell warning block, and the «Продать всё равно» confirm button (`button.danger`, existing) |

Accent reserved for: primary submit buttons («Оформить продажу», «Добавить покупателя», «Сохранить покупателя»), the «Добавить строку» add-line button, text links (nav links; active nav link = accent + 600 weight), input `:focus` outline (2px, existing rule), and the `<mark>` match highlight in the customer search picker (accent-tinted `#e8effd`, text unchanged — existing rule). Nothing else. Secondary/neutral actions («Отмена», «Убрать покупателя», «Удалить строку», «Вернуться к корзине») use the neutral `button.secondary` style (white, `#d9d9d9` border, `#222` text) or a plain text link.

**Oversell warning uses destructive, not a new "warning" color:** the oversell state is a blocking, must-confirm condition — it reuses the existing `.error-block` treatment (`#b91c1c` on `#fdecec`) and the existing `button.danger` for «Продать всё равно». This keeps the semantic budget at exactly one and needs no CSS additions for the warning itself.

Muted text (`.muted`, `#6b7280`): sale/purchase timestamps, «(необязательно)» markers, field hints, profit «неизвестна», walk-in «Без покупателя» note, empty-state body text. Tint of the dominant neutral, not a fifth color role.

Success feedback: neutral — plain `#222` text on the page surface («Продажа оформлена…»), no green (Phase 2/3 convention).

## Interaction Contract

- **Add a basket line («Добавить строку»):** `hx-get="/sales/row"` returning a fresh `sale_row.html`, `hx-target="#basket-rows"`, `hx-swap="beforeend"`. Each row's inputs are array-named (`code[]`, `name[]`, `qty[]`, `price[]`) so the multi-line basket maps directly to `list[str] = Form([])` (RESEARCH: repeated form fields over indexed names). Focus moves to the new row's «Код» via an `hx-on::load` hook on the swapped row.
- **Per-line code lookup (D-10 / SAL-02):** `hx-get="/sales/lookup"` on each line's Код input, `hx-trigger="input changed delay:300ms"`, `hx-sync="this:replace"`, `hx-include` scoped to that row's fields. Server decides: product match → fragment filling that row's Название + pre-filling Цена продажи from `sale_cents` (editable); dictionary-only match → name; nothing / operator already typed → 204 no swap. Swap-time guard (`hx-on::htmx:before-swap` / `oob-before-swap` → `shouldSwap = false` when the operator typed a value) replicated from Phase 3 `receipt_form.html` — a lookup response never clobbers typed input.
- **Finalize (D-02/D-03, core value — fast entry):** `hx-post="/sales"` targeting the whole form partial (`hx-target="#sale-form-wrap" hx-swap="outerHTML"`). Success → 200 with a fresh empty basket + neutral success line + `hx-swap-oob` recent-sales refresh; validation failure → 422 with the basket re-rendered, entered values and lines preserved, inline errors (422 swap already whitelisted in `base.html` htmx-config). Empty basket → 422 with the empty-basket error, zero writes. Focus: explicit `hx-on::htmx:load` on the fresh form calling `document.getElementById('code').focus()` (do NOT rely on `autofocus` inside swapped content — RESEARCH Pitfall 4). Keep `autofocus` on the first Код for initial page load.
- **Oversell two-step (SAL-04/D-08/D-09):** finalize runs a server-side aggregate stock check (sum requested qty per `product_id` across all lines) BEFORE any write. If any product oversells and `confirm != "1"`, return `sale_oversell.html` with ZERO writes. The «Продать всё равно» button re-POSTs the same basket via `hx-post="/sales" hx-vals='{"confirm":"1"}'` (form association, no re-serialization of the basket) — `hx-vals` merges `confirm=1` into the existing fields. On `confirm=1` the service skips the block and writes; stock may go negative. «Вернуться к корзине» dismisses the warning and restores the editable basket (client-side swap back, no write).
- **Customer autocomplete (D-05):** `hx-get="/sales/customer-search"` on the header search input, `hx-trigger="input changed delay:300ms"`, target a picker rows partial; Cyrillic folding done in Python against `search_lc` (Phase 2 rule); `<mark>` highlights the match. Selecting a row swaps in the selected-customer chip carrying a hidden `customer_id` into the sale form.
- **Customer inline quick-create (D-05):** the «Новый покупатель» sub-region gathers Имя/Фамилия/Номер консультанта; a button (NOT a nested `<form>` — the basket form must not be nested) `hx-post="/sales/customer"` with `hx-include` of those three inputs; the response swaps in the selected-customer chip (hidden `customer_id`) into the sale form's customer slot. «Имя» required → inline «Укажите имя покупателя.»
- **Customers list search (CST-01):** `hx-get` on the search input, `hx-trigger="input changed delay:300ms, keyup[key=='Enter']"`, target = results tbody partial; Cyrillic-safe via `search_lc`. No spinner needed at local latency.
- **Keyboard-first (core value: fast entry):** every form submits on Enter; tab order = visual order (within a line: Код → Название → Кол-во → Цена; then next line / «Добавить строку» / «Оформить продажу»); visible 2px accent focus outline on all interactive elements — never `outline:none` without replacement.
- **Buttons during request:** `hx-disabled-elt` on submit/confirm buttons; existing `button:disabled { opacity:.6 }`.
- **Money inputs:** `type="text" inputmode="decimal"`, comma and dot accepted (server `to_cents`); quantity `type="text" inputmode="numeric"`, positive integer only. Display money via the `cents`/`format_cents` filter (comma decimal separator). Never `float()` on prices.
- **No client-side JS beyond vendored HTMX.** No modals, no toasts; htmx 2.x `hx-on::` double-colon syntax only. Native `hx-confirm` is available if a destructive customer action is ever added (not required this phase).
- **Manual UAT gates (from RESEARCH validation map):** after finalize, focus lands on the first «Код» without a click; per-line lookup never overwrites a typed value; an oversell warning appears with zero committed `sale` ops until «Продать всё равно» is pressed; purchase history renders the frozen `unit_price_cents`, not the current card price.
