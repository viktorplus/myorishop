# Phase 23 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

Built from `23-CONTEXT.md` (locked decisions D-01..D-10), `23-RESEARCH.md` (7 patterns/pitfalls,
3 assumptions, 2 open questions), `REQUIREMENTS.md` (DASH-01..05, HIST-01..04), `ROADMAP.md`
(Phase 23, 5 success criteria), and the project's already-shipped design system
(`app/static/style.css`, established Phase 2, extended through Phase 22). Run in `--auto` mode
(STATE.md: full-auto ON): no interactive design questions were asked — every value below either
reuses an existing project-wide token/pattern (marked **"reused, unchanged"**) or resolves a
`23-CONTEXT.md`/`23-RESEARCH.md` discretion item or open question with a concrete, prescriptive
answer (marked **"resolved this phase"**).

**This phase introduces ZERO new CSS rules and ZERO new color/spacing/type tokens.** Every surface
below is built from `style.css`'s existing vocabulary — the sixth consecutive phase to do so
(Phases 19/20 shipped none, Phase 21 shipped one rule, Phase 22 shipped none). The two new shapes
this phase needs (a 4-tile metric grid, a type-narrowed table) are both direct reuses of
`.metric-grid`/`.metric-tile` (Phase 17) and `history_rows.html`'s existing filter-row idiom
(Phase 14), respectively.

**This phase rebuilds two stub pages and extends two shipped ones.** `pages/home.html` (17 lines,
walking-skeleton) and `mobile_pages/home.html` (static tile grid) are rebuilt into real dashboards.
`app/routes/history.py` / `partials/history_rows.html` and their mobile counterparts are extended
in place — same route, same swap-on-filter-change mechanism, new params. `pages/catalogs.html`
gains one small new form (D-02). No page in this phase is deleted or replaced by a different route.

|------|-------|-------|

| Secondary (30%) | `#ffffff` | Table / tile / card surfaces; borders `#d9d9d9`; `th` fill `#f0f0f0` |
| Accent (10%) | `#2563eb` | See "Accent reserved for" below |

**Accent reserved for** (explicit list — every plain `<a>` is accent by the global `a { color:
#2563eb }` rule, `style.css:38-40`; nothing below is a new color role):
- «Сохранить» — the catalog form's submit button (D-02), styled as the default `button`/`.button`
  accent fill, mirroring every other primary form submit in the app.
- «Указать номер и дату закрытия» — the dashboard's empty-catalog-state link to `/catalogs` (plain
  `<a>`, accent by default).
- «Подробнее» — the recent-operations feed's per-row link to `/history?type=…&product=…` (plain
  `<a>`, same neutral-link idiom as `recent_sales.html`'s «Вернуть» and `history_rows.html`'s
  «Действие» column — a text link, never `.button`).
- Keyboard `:focus-visible` outline ring on every new control (catalog fields, the relocated «Тип»
  select, the new История date/category/customer filters) — automatic (`style.css:160-167`).

**Explicitly NOT accent:** the pagination buttons stay `button.secondary` (white/`#222`/`#d9d9d9`,
unchanged, reused verbatim). The active-page indicator stays `.current-page` (`#e8effd` tint,
unchanged) — not a new selection color, the same token `.selected-batch`/`.mobile-card.selected`
already use.

**Destructive reserved for:** validation error copy only — the catalog form's date-format error
(`<p class="error">`), and History's malformed date-range fallback message, both via the existing
`.error`/`.error-block` classes. **This phase has no `button.danger` and no delete action anywhere**
(read-only rebuild + one small settings-style edit — see Copywriting, Destructive confirmation).

**No new color role is introduced.** Two standing rules apply and are load-bearing here:
1. **Money is NEVER sign-colored.** Every figure in the 4 dashboard tiles (revenue, profit,
   expense, valuation) renders in default `#222`, exactly like `finance_tiles.html`'s tiles today —
   including a negative net-profit figure, which is still plain text, not red.
2. **Color is never the sole carrier of state (WCAG 1.4.1).** An expired/closing-soon catalog is
   signalled by *text* («закрыт»), not by a red tint on the countdown line — see Copywriting.

## Interaction & Layout Decisions

Not covered by the fixed template sections above but load-bearing for the executor. Each traces to
a specific `23-CONTEXT.md` decision or `23-RESEARCH.md` finding.

### Dashboard layout (DASH-01..05)

**1. Page order, top to bottom** (resolves DASH-01..05's relative placement, not specified in
CONTEXT):
```
<h1>Главная</h1>
<p class="muted">{{ weekday }}, {{ date }} · {{ time }}</p>                  <!-- DASH-01 -->
<p>{{ catalog line, or the empty-state link }}</p>                          <!-- DASH-02 -->
<div class="metric-grid">…4 tiles: Сегодня / Неделя / Месяц / Склад…</div>  <!-- DASH-03/04 -->
<h2>Последние операции</h2>
<table>…10-column feed, see Interaction 4…</table>                          <!-- DASH-05 -->
```
DASH-01/02 are **context lines** (single facts, `<p>`, no tile chrome) — distinct from DASH-03/04's
**aggregated metrics** (`.metric-grid` tiles). Mixing a bare fact into a tile grid designed for
money aggregates would misrepresent it as a computed figure.

**2. Empty-catalog state does not block the rest of the page (CONTEXT, explicit).** `dashboard_view()`
composes DASH-01/03/04/05 unconditionally; DASH-02's line is the only piece that branches on whether
`get_active_catalog()` returns a row. A missing/never-configured catalog renders the empty-state
link and nothing else changes.

**3. The 4-tile metric grid — exact composition** (resolves CONTEXT's undecided "how period totals
and stock valuation share the grid"):
```
Tile 1 «Сегодня»   Tile 2 «Неделя»    Tile 3 «Месяц»     Tile 4 «Склад»
Выручка: X          Выручка: X         Выручка: X          Товаров в наличии: N
Прибыль: X           Прибыль: X         Прибыль: X          Стоимость по закупке: X
Расход: X            Расход: X          Расход: X           Стоимость по продаже: X
                                                             на текущий момент [muted]
```
Each of the 3 period tiles is `<div class="metric-tile"><p class="tile-label">{{ period }}</p>` +
3 `<p class="num">` lines. `{{ period }}` money values are never sign-colored (Color rule 1).
«Прибыль» is **net** profit — `gross_profit_cents + cash_expense_total(...)` — the identical D-08
formula from `_metrics_context`, addition never subtraction (the sign-convention gotcha is
pre-existing, not new to this phase). Week/month boundaries reuse `_resolve_period`'s Monday-start/
calendar-month convention exactly (`23-RESEARCH.md` Anti-Patterns) — the dashboard and Финансы must
never disagree on what "this week" means for the same data.

**4. Recent-operations feed — one fixed 10-column table, per-row muted dashes for irrelevant cells**
(resolves DASH-05's "columns adapt per operation type" for a *single mixed-type* list — as opposed
to HIST-01's per-type *narrowing*, see Interaction 5):
```
Когда | Тип | Код | Товар | Срок | Кол-во | Себестоимость | Прибыль | Покупатель | Действие
```
This is the DASH-05-literal column list (`type, code, name, expiry, quantity, cost, profit,
customer`) plus the universal `Когда`/`Действие` bookends every list table in this app already
carries. Each row populates only what its type has; everything else renders the existing
`<span class="muted">—</span>` idiom (`history_rows.html`'s established pattern for missing
price/cost). Concretely, per type:
- **receipt**: Срок/Кол-во/Себестоимость populate; Прибыль/Покупатель show `—`.
- **sale, return**: all 8 data columns populate. Покупатель reuses `recent_sales.html`'s exact
  «Розница» muted label for `customer is None` (never blank, never `None`) — see D-06 there.
- **writeoff**: Срок/Кол-во/Себестоимость populate; Прибыль/Покупатель `—`.
- **correction**: Кол-во populates; Срок/Себестоимость/Прибыль/Покупатель `—` (a recount has no
  cost delta).
- **transfer**: Срок/Кол-во populate; Себестоимость/Прибыль/Покупатель `—`. Each of the two rows a
  transfer writes shows **only its own side's data** via its own `Batch` join (`23-RESEARCH.md`
  Pitfall 6) — no synthesized "from → to" single row.
- **Тип** column always renders `OPERATION_TYPE_LABELS[op.type]` (existing dict, unchanged).
- **Действие** is always «Подробнее», linking into `/history` pre-filtered to that row's type +
  product — this is the feed's only interactive element, and it is how DASH-05 connects to HIST-01
  (`23-RESEARCH.md` D-09: "no new History-side plumbing needed").

**5. Mobile dashboard — supplement, never replace the navigation tile grid** (resolves
`23-RESEARCH.md` Open Question 1 / Pitfall 1 — **must not be decided differently by a plan**).
`mobile_pages/home.html`'s existing 10-tile nav grid is the *only* way to reach `/m/sales`,
`/m/receipts`, `/m/writeoff`, etc. — MOB-01's tab bar does not ship until Phase 24. The rebuild:
```
<h1>MyOriShop</h1>
<div class="mobile-tile-grid">…existing 10 nav tiles, UNCHANGED, UNTOUCHED, first…</div>
<h2>Показатели</h2>
<p class="muted">{{ weekday }}, {{ date }} · {{ time }}</p>
<p>{{ catalog line }}</p>
<div class="metric-grid">…same 4 tiles, 2-col via .mobile-shell .metric-grid…</div>
<h2>Последние операции</h2>
…{{ feed as .mobile-card list, one card per row, mirroring history_cards.html's card shape }}…
```
Nav grid stays **first and untouched** — zero regression risk to existing muscle memory or to every
other mobile page's reachability. Dashboard content is **added below it**, in mobile's own
card/accordion layout (D-10: "own simpler layout", never a squeezed copy of the desktop grid). Each
feed card follows `history_cards.html`'s existing shape: muted meta line (when · type), bold
name+code line, a `.num` line for qty/cost/price/profit (whichever apply), and a «Подробнее» link —
no new card variant is introduced.

### The catalog form (DASH-02, D-01/D-02)

**6. Fully manual, two independent fields, on the existing `/catalogs` page** (D-01/D-02, locked —
not re-litigated here). Placed **above** the existing `catalog_rows.html` list, its own small
inline form (not `.stacked-form` — two short fields, matching the inline-row shape `form` already
uses by default, `style.css:62-68`):
```jinja
<h2>Активный каталог</h2>
<form hx-post="/catalogs/active" hx-target="this" hx-swap="outerHTML">
  <div class="field">
    <label for="active-number">Номер каталога</label>
    <input type="text" id="active-number" name="number" value="{{ active.number or '' }}">
  </div>
  <div class="field">
    <label for="active-close-date">Дата закрытия</label>
    <input type="date" id="active-close-date" name="close_date" value="{{ active.close_date or '' }}">
  </div>
  <button type="submit">Сохранить</button>
  {% if error %}<p class="error">{{ error }}</p>{% endif %}
</form>
```
Both fields are independently optional (an operator may set only the number before knowing the
close date, or vice versa) — no cross-field validation. `close_date` goes through
`date.fromisoformat` server-side (ASVS V5, mirrors every other date field in this codebase); a
malformed value degrades to the D-02 error copy, never a raw 500. No success banner is added — the
updated countdown line on `/` and the re-rendered fields here **are** the confirmation, matching
this app's existing minimal-feedback convention for simple settings-style edits.

### History type-first rebuild (HIST-01..04)

**7. The «Тип» select moves from the table's `filter-row` to the page's top `.filter-bar`, becoming
the first, most prominent control** (resolves CONTEXT's "type selection is a refinement, not a
gate" into a concrete layout — **Claude's discretion, resolved this phase**). This is the literal
reading of "type-first": the control that picks the type sits above and outside the column-specific
filters that then narrow *within* that type, rather than being buried as one of several equal
`<th>` filters. No new element is introduced — it is the exact same `<select name="type">`
(`OPERATION_TYPE_LABELS` options, unchanged, all 9 types still listed — see Interaction 11) with a
new position and a widened swap responsibility (rows **and** columns, not rows alone):
```
.filter-bar:  [ Тип ▾ ]   [ Сортировать по ▾ ]   [ Сегодня ][Неделя][Месяц]  [ с __ ] [ по __ ]
<table>
  <thead>
    <tr>…type-appropriate data columns, see Interaction 5-of-column-sets below…</tr>
    <tr class="filter-row">…Товар / Категория / (Покупатель, sale+return only)…</tr>
  </thead>
  <tbody>…rows…</tbody>
</table>
{% include "partials/pagination.html" %}
```
The date-range preset bar + от/по inputs are the **entire, unmodified** `period_filter.html`
partial (D-05: date range is universal — visible for every type, including the untyped generic
view). `hx-include="#history-rows input, #history-rows select"` (the existing include list) already
covers every field in this reorganized layout — no new `hx-include` scope is needed since the whole
bar+table stays inside `#history-rows`.

**8. Per-type column sets** (extends `23-RESEARCH.md` Pattern 3 with the universal bookends and the
generic-view fallback — this is the authoritative column list for the executor):

|------|---------------------|
| *(no type selected — generic, D-04, UNCHANGED)* | Когда · Тип · Код · Товар · Кол-во · Цена · Себестоимость · Причина · Кто · Действие |
| `receipt` | Когда · Код · Товар · Срок · Кол-во · Себестоимость · Действие |
| `sale` | Когда · Код · Товар · Срок · Кол-во · Цена · Себестоимость · Прибыль · Покупатель · Действие |
| `return` | Когда · Код · Товар · Срок · Кол-во · Цена · Себестоимость · Прибыль · Покупатель · Действие |
| `writeoff` | Когда · Код · Товар · Срок · Кол-во · Себестоимость · Причина · Действие |
| `correction` | Когда · Код · Товар · Срок · Кол-во · Причина · Действие |

**9. Date-range filter — reuse `period_filter.html` verbatim, universal across every type
(D-05).** Malformed от/до degrades exactly like `reports.py::_resolve_period` — fall back to

**10. Category filter — free-text, reused verbatim from `product_rows.html`'s idiom, visible for

**11. Customer filter — free-text, same debounced idiom, visible ONLY when `type_filter` is `sale`
or `return`** (D-05, locked). **Resolved this phase (CONTEXT's own discretion item): hidden
entirely for every other type**, not shown-disabled — an absent control cannot be mistaken for "no
customers exist," which a greyed-out one risks implying. Server-side, the filter is a text match
resolved through `search_customers(session, q)` (the existing SALE-04 service, reused per
`23-RESEARCH.md` Don't-Hand-Roll) rather than a new autocomplete-picker widget — a filter is a
narrower interaction than SALE-04's create/select flow and does not need its own picker component.

**12. Audit types (`price_change`/`product_created`/`product_edited`) stay in the «Тип» dropdown,
unchanged, all 9 options** (resolves `23-RESEARCH.md` Open Question 2 / Pitfall 5 — **do not narrow
the dropdown to the 6 stock-affecting types**, that would regress a currently-working filter the
operator uses today to audit price changes). Selecting one of the 3 renders the **existing** generic
10-column view unmodified — they simply don't get a narrowed column set, matching D-06's explicit
scoping of "type-first browsing" to the 6 stock-affecting types only.

position in the top filter-bar). If the planner adds type-specific sort options (e.g. by quantity

**14. Pagination — reuse `partials/pagination.html` unchanged on desktop; migrate mobile History
onto the SAME partial, retiring `history_load_more.html`** (D-10, HIST-04). This is the only
mobile-navigation-shape change in this phase (Pitfall 1 does not apply here — `pagination.html`'s
buttons are page-turning controls, not the app's primary nav). `mobile_pages/history.html`'s
filter fields become stacked `.field` divs (Тип, Категория, Покупатель-conditional, с/по) above
`#history-cards`, mirroring the existing single-field layout there today — same idiom, more fields.
`#history-cards` and the pagination bar remain DOM siblings (never nested), preserving the
established CR-01 precedent that a filter swap can never destroy the pagination control.

### Guardrails (ROADMAP criterion 5 — regression only)

**15. Do not rename `#history-rows`, `#history-cards`, or `#return-slot`.** `history_rows.html`'s
`/history` URL/query-param contract (D-03, locked) and the sale-linked return entry point
(`hx-get="/returns?...` hx-target="#return-slot"`) both depend on these ids surviving this rebuild
untouched.

**16. Do not touch `join`→`outerjoin` on the `Operation → Sale → Customer` chain anywhere in this
phase.** Both DASH-05's feed and HIST-02's customer filter walk-in-safety depend on the same double
`outerjoin` `recent_sales` already established (`23-RESEARCH.md` Pitfall 4) — an inner `join` here
would silently drop every walk-in sale/return and every non-sale/return type from the feed.
