## Trust Boundaries

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-23-04 | Tampering | `category` filter | mitigate | Never string-interpolated into SQL; candidate resolution happens in Python against the small `category_options()` list, then applied via a parameterized `.in_()` — same discipline as the existing `type_filter in OPERATION_TYPES` allow-list check |
| T-23-05 | Tampering | `customer` filter | mitigate | Resolved via the existing `search_customers()` bounded query (parameterized `.contains()`), never raw string interpolation; result narrowed further via `.in_(ids)` |
| T-23-06 | Tampering | `customer` filter cross-type leakage | mitigate | Server-side guard: `customer` is applied ONLY when `type_filter in ("sale","return")`, regardless of what the caller passes — defence-in-depth even if a route forgets to hide the control (D-05) |
| T-23-07 | Information Disclosure | `Operation -> Sale -> Customer` outerjoin chain | mitigate | Both hops stay `.outerjoin()` (never `.join()`) so walk-in/non-sale rows are never silently dropped from a filtered result set — a `.join()` regression here would be a data-loss bug, not just a display bug |
</threat_model>
