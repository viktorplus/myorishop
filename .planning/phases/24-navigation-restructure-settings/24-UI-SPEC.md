## Spacing Scale

Declared values (must be multiples of 4) — confirmed from `app/static/style.css` header comment (line 2) and verified against usage throughout the file:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Label-to-input gap, `.contact-row`/`.batch-picker` fine spacing |
| sm | 8px | Compact element spacing (`.form` inline gap, `.preset-bar`, `.pagination`, `.contact-row`) |
| md | 16px | Default element spacing (`nav` gap, `.field` bottom margin, `.form-actions`/`.filter-bar`/`.metric-grid` gap, table cell padding) |
| lg | 24px | Section padding (`.form` bottom margin, `.form-actions` top margin, `.mobile-actions` top margin, `.mobile-card` bottom margin) |
| xl | 32px | Layout gaps (`.container` bottom padding, `nav` bottom margin) |
| 2xl | 48px | Major section breaks (`.empty-state` vertical padding) |

Exceptions:
- **44px minimum touch target on mobile** (`.mobile-back`, `.mobile-card`, `.mobile-actions button/a.button` — `app/static/style.css:317,330,349`). This phase's new mobile tab bar items MUST also honor this exception — each `nav.mobile-tabbar a` gets `min-height: 44px` even though 44 is not a multiple of 4/8.
- No other exceptions needed for this phase.

## Typography

Confirmed 4 sizes / 2 weights already declared in `app/static/style.css` — no new sizes or weights for this phase:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label / muted / small | 14px | 400 | 1.5 (inherited from body) |
| Heading (h2) | 20px | 600 | 1.2 |
| Heading (h1) | 24px | 600 | 1.2 |

New elements in this phase map onto these exactly:
- Toolbar group labels ("Действия" / "Справочники") → Label role (14px/400, `.muted` color).
- Settings-page summary lines (warehouse count, last-backup date) → Label role (14px/400, `.muted` color) — matches how `.tile-label`/`.muted` are used elsewhere for secondary metadata.
- Mobile tab bar labels → Body-adjacent small text, 14px/400 (matches `.mobile-step-indicator`'s existing 14px pattern), 600 weight only when the tab is active (mirrors `nav a.active`'s `font-weight: 600`).
- "← Назад к отчётам" back-links → Body role (16px/400), plain `<a>`, no new size.

## Color

Confirmed 60/30/10 split, read verbatim from `app/static/style.css`:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#fafafa` | Page background (`body`) |
| Secondary (30%) | `#ffffff` | Cards, nav bar, table surfaces, buttons' hover-inverse (`button.secondary`), mobile tiles/cards |
| Accent (10%) | `#2563eb` | Links, primary buttons, active nav/tab state, focus outline, selected-row/search-match tint background (`#e8effd`) |
| Destructive | `#b91c1c` | Destructive actions only (`button.danger`, `a.link-danger`, `.error`/`.error-block`) |

Accent (`#2563eb`) reserved for, explicitly, in this phase's new elements:
- The new mobile tab bar's **active tab** indicator (font-weight 600 + `border-bottom: 3px solid #2563eb` — shape-plus-color, never color alone, matching this project's existing WCAG-1.4.1-aware convention seen in `a.link-danger`'s retained underline and the `price-cue` compliance note).
- Every `<a>`/`a.button` inside the new Товары toolbar and the Настройки hub (inherits the existing global `a { color: #2563eb }` / `a.button` treatment — no override).
- Focus outline on any new focusable element (`outline: 2px solid #2563eb`, already global via `a:focus-visible`).

Not introduced or extended in this phase: no new semantic color role. The existing 4th role (`price-cue` amber, `#b45309`/`#fef9e7`/`#eff6ff`) is unrelated to this phase's scope and must not be reused or reinterpreted for nav/settings UI.

## Visual Hierarchy

Primary visual anchors for this phase's key screens (added per UI-checker Dimension 2 recommendation):

- **Товары page** — the always-visible toolbar (D-05) is the primary anchor below the page title: it never competes with the product list for attention since it's structurally separate (own `.toolbar` block) and always expanded, never a click-to-reveal element.
- **Настройки hub** — the 3-row link list is a flat hierarchy by design (no single dominant row); each row's `a.button` is the visual anchor within its own row, with the `.muted` summary subordinate underneath.
- **Mobile tab bar** — the active tab's `border-bottom: 3px solid #2563eb` + `font-weight: 600` is the sole navigational focal point on every mobile screen (sticky top, always visible), telling the operator where they are at a glance.

## New Components (layout composition only — no new interactive JS/CSS component)

### Товары toolbar (D-01/D-02/D-03/D-04/D-05 — desktop `/products`, mirrored on mobile `/m/products` per D-11)

```html
<div class="toolbar">
  <div class="toolbar-group">
    <span class="muted">Действия</span>
    <div class="form-actions">
      <a class="button" href="/receipts/new">Приход</a>
      <a class="button" href="/writeoff">Списание</a>
    </div>
  </div>
  <div class="toolbar-group">
    <span class="muted">Справочники</span>
    <div class="form-actions">
      <a class="button" href="/categories">Категории</a>
      <a class="button" href="/dictionary">Справочник</a>
      <a class="button" href="/catalogs">Каталоги</a>
    </div>
  </div>
</div>
```

- Always visible, never behind a click (D-05) — no `<details>`, no JS toggle.
- New CSS (2 rules only): `.toolbar { display: flex; gap: 32px; margin-bottom: 24px; flex-wrap: wrap; }` and `.toolbar-group { display: flex; flex-direction: column; gap: 8px; }`. Everything inside (`.form-actions`, `.button`, `.muted`) is 100% existing, unmodified.
- Use `<div>`, never `<nav>` — `<nav>` is the primary top-nav landmark and some report tests assert `"<nav" not in response.text` on chrome-less HX-partial responses (Pitfall 5 in RESEARCH.md).
- Mobile variant: same markup/classes, rendered inside `.mobile-shell`; toolbar buttons still get the 44px touch-target minimum via the existing `a.button` sizing (padding 8px 16px already clears 44px with 16px font-size — verify at execution, no new mobile-specific override expected).

### Настройки hub (D-06 — new `/settings` page)

Plain list of 3 rows, each a link plus an inline or subtitled summary (exact visual treatment — inline vs. subtitle — is Claude's discretion per CONTEXT.md; **recommend subtitle**, i.e. summary text on its own line below the link in `.muted` 14px, for scanability and consistency with how `.tile-label` + body text stack in `.metric-tile`):

```html
<h1>Настройки</h1>
<div class="field">
  <a class="button" href="/warehouses">Склады</a>
  <p class="muted">{{ warehouse_count }} складов</p>
</div>
<div class="field">
  <a class="button" href="/backup">Резервные копии</a>
  <p class="muted">{% if last_backup_iso %}Последняя копия: {{ last_backup_iso | local_dt }}{% else %}Резервных копий пока нет{% endif %}</p>
</div>
<div class="field">
  <a class="button" href="/finance/report">Экспорт кассы</a>
</div>
```

- Reuses `.field` (16px bottom margin, existing) as the row wrapper — no new spacing rule needed.
- `a.button` styling for the 3 links matches every other primary navigational button in the app (e.g. `catalog_detail.html`'s "Открыть PDF").

### Mobile top tab bar (D-09/D-10/D-11 — replaces the 10-tile home grid)

```html
<nav class="mobile-tabbar">
  <a href="/m/"{% if request.url.path == "/m/" %} class="active"{% endif %}>Главная</a>
  <a href="/m/products"{% if request.url.path.startswith("/m/products") %} class="active"{% endif %}>Товары</a>
  <a href="/m/sales"{% if request.url.path.startswith("/m/sales") %} class="active"{% endif %}>Продажи</a>
  <a href="/m/customers"{% if request.url.path.startswith("/m/customers") %} class="active"{% endif %}>Покупатели</a>
  <a href="/m/history"{% if request.url.path.startswith("/m/history") %} class="active"{% endif %}>История</a>
  <a href="/m/reports"{% if request.url.path.startswith("/m/reports") %} class="active"{% endif %}>Отчёты</a>
  <a href="/m/finance"{% if request.url.path.startswith("/m/finance") %} class="active"{% endif %}>Финансы</a>
</nav>
```

- **Docked at the TOP of the viewport** (D-09, deliberate deviation from bottom-tab convention — confirmed operator choice, not an oversight). Implement as `position: sticky; top: 0;` on `nav.mobile-tabbar`, placed as a sibling immediately before `<main class="mobile-shell">` in `mobile_base.html` (full-viewport width, NOT constrained inside the 480px `.mobile-shell` — matches how desktop's `nav` is full-bleed via its own negative-margin trick).
- New CSS (composition only, all existing tokens): `nav.mobile-tabbar { display: flex; background: #ffffff; border-bottom: 1px solid #d9d9d9; position: sticky; top: 0; z-index: 1; }` and `nav.mobile-tabbar a { flex: 1; min-height: 44px; display: flex; align-items: center; justify-content: center; font-size: 14px; text-align: center; padding: 4px; }`.
- **Active state must not rely on color alone** (project-wide WCAG 1.4.1 convention already established for `a.link-danger` and flagged as a gap for `price-cue`): `nav.mobile-tabbar a.active { font-weight: 600; border-bottom: 3px solid #2563eb; }` — weight + border shape, not just the inherited accent link color.
- Use `<nav class="mobile-tabbar">` — a second, legitimately distinct `<nav>` landmark (HTML permits multiple `<nav>` elements; this is NOT the same landmark as desktop's `<nav>`, and no existing test asserts against `mobile_base.html` the way `tests/test_reports.py:316,459` does against desktop chrome-less fragments).
- Coexists with, does not replace, the existing per-page `{% block back %}`/`{% block step_indicator %}` blocks in `mobile_base.html` — the tab bar is global top-level chrome (switches sections); the back-link block remains the in-flow "return to previous step" affordance for wizards (receipts, transfers, etc.), unchanged.
- No icons — text-only labels, consistent with the rest of the app (see Design System table above).
- Settings has **no mobile tab and no mobile entry point at all** (D-12) — do not add an 8th tab, do not add a "Ещё"/"More" overflow tab as a workaround; Настройки-hosted pages are confirmed desktop-only by the operator.
