## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Operator browser -> `GET /settings` | Same-origin, no auth boundary, zero query/form params accepted |
| Operator browser -> `GET /backup` (now also serving 3 embedded CSV links) | Same-origin; the CSV endpoints themselves are unchanged (V12 pattern: no filename/path/Form/Query param accepted, per `app/routes/export.py`'s own docstring) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-02a | Information Disclosure | `/export` page route left registered but unlinked | accept | Same data exposure as before this phase (no auth boundary changes); noted in 24-RESEARCH.md Security Domain as an accepted, pre-existing condition |
| T-24-02b | Tampering | New `GET /settings` route | accept | Zero query/form params accepted — nothing for an attacker to tamper with; `settings_summary` only reads existing service data |
| T-24-02c | Denial of Service | `settings_summary` calls `list_warehouses`/`list_backups` on every `/settings` GET | accept | Both are existing, already-used-elsewhere queries with no new cost profile; no new unbounded loop introduced |

No V5 input-validation surface is introduced by this plan (no new query/form params).
