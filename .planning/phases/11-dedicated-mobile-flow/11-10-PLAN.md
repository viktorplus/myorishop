<objective>
Close the two gaps from `11-UAT.md` Test 4 (both already root-caused via `/gsd-debug` — do NOT re-diagnose):

**Gap A (blocker):** Batch-selection cards on the mobile Sale/Write-off/Correction/Transfer wizards render as `<button class="mobile-card">`. `app/static/style.css`'s `.mobile-card` rule overrides the generic `button` rule's `background` to white but never overrides its `color: #ffffff` — so card text (price/expiry/quantity/comment) is white-on-white (or white-on-selected-tint), readable only on `:hover` (where `button:hover` coincidentally darkens the background). Root cause confirmed in `.mobile-batch-card-white-on-white` debug session: `.mobile-card` is missing the `color` counterpart that the codebase's own `button.secondary` pattern already establishes.

**Gap B (major):** In the mobile Sale wizard, the qty-price step's «Назад» button is wired identically to the batch step's «Назад» button — both `hx-post="/m/sales/step/product" hx-vals='{"back": "1"}'`, which unconditionally re-renders step 1. When a batch step (step 2) was shown for the current line, clicking «Назад» from step 3 (qty-price) skips straight over it to step 1, losing the batch cards with no way back short of re-typing the code. Root cause confirmed in `.mobile-sale-back-nav-loses-batch-step` debug session: the qty-price «Назад» control was copy-pasted from the batch step without adjusting its target — a correctly-wired alternative (`GET /m/sales/step/batch`, which already re-runs `open_batches` fresh) exists in the codebase but is never targeted by it.

Purpose: make the batch-selection step usable on a real phone (currently invisible text) and make the Sale wizard's back navigation non-destructive (currently a dead end requiring the code to be re-typed).
Output: a CSS fix (scoped to the actual broken element, not a blanket `.mobile-card` color change) + a route/template fix that distinguishes "batch step was shown" from "dictionary-only match, no batch step" so «Назад» targets the correct previous step either way.
</objective>

Batch cards are the only `.mobile-card` usage rendered as a `<button>` (via `batch_card_picker.html` and `transfers_step_batch.html`); every other `.mobile-card` usage in the codebase is a `<div>`, `<label>`, or `<a>` (history cards, receipts batch chooser, corrections mode picker, write-off reason rows, search results) and already renders with its own correct color (inherited `#222` for div/label, the existing `a { color: #2563eb }` accent for the search-result link) — a blanket `.mobile-card { color: ... }` rule would silently change those unrelated, un-reported-as-broken surfaces too, so keep the selector scoped to `button.mobile-card`.

Add, in order:

3. `button.mobile-card.selected:hover { background: #e8effd; }` — needed because rule 2's `button.mobile-card:hover` background would otherwise win (equal-or-higher specificity, later source order) over the plain `.mobile-card.selected` background when a selected card is hovered, incorrectly flashing it white; this rule keeps the selection tint intact while hovering a selected card.

Precede the three rules with a short comment (matching the file's existing comment style, e.g. the `/* Batch picker (Phase 9)... */` comment above) explaining this closes 11-UAT.md Test 4's white-on-white gap and is intentionally scoped to `button.mobile-card` only.
