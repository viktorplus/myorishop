<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser → `POST /m/receipts` | Untrusted `warehouse_id` and `batch_choice` (either an existing batch id or the literal `"new"`) reach `register_receipt` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-11-07 | Tampering | `batch_choice` posted value | mitigate | `register_receipt`'s existing service-layer validation already rejects an unknown/foreign batch id exactly as it does for the desktop form — this plan introduces no new trust boundary, it calls the same unmodified function |
| T-11-08 | Tampering | `warehouse_id` posted value | mitigate | `register_receipt` validates against active warehouses server-side, unchanged from desktop |
| T-11-09 | Information Disclosure | Product/batch comment and location text rendered on the confirmation step | mitigate | Jinja autoescape only, `\| safe` never applied |
| T-11-CSRF | Tampering | CSRF | accept | Unchanged app-wide posture |

</threat_model>
