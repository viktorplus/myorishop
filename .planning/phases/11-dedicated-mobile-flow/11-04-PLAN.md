<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser → `GET /m/sales/step/batch` | Untrusted client-supplied `batch_id` must be re-validated against the looked-up product before being trusted |
| Browser → `POST /m/sales` | Untrusted accumulated basket arrays (`code[]`/`qty[]`/`price[]`/`batch_id[]`) reach `register_sale` — same shape and same validation boundary as the desktop basket POST |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-11-10 | Tampering / Elevation of Privilege | `/m/sales/step/batch` posted `batch_id` | mitigate | Re-validate `candidate.product_id == product.id` before trusting the pick (T-09-08 precedent), copied verbatim from `app/routes/sales.py::sale_batch_pick` |
| T-11-11 | Tampering | Final basket arrays posted to `POST /m/sales` | mitigate | `register_sale`'s own service-layer validation (unchanged) is the backstop, exactly as it is for the desktop basket — this plan adds a client-side accumulation step, not a new trust boundary |
| T-11-12 | Repudiation | Price-floor / oversell bypass via a client that never re-POSTs `confirm=1` | mitigate | Zero Operation rows are written until `register_sale` returns without `oversell`/`below_minimum` set — verified by acceptance criteria requiring a zero-row assertion before the confirm re-post |
| T-11-CSRF | Tampering | CSRF | accept | Unchanged app-wide posture |

</threat_model>
