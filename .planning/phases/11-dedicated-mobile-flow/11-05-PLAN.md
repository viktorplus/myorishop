## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser → `POST /m/writeoff/step/batch-pick` | Untrusted client-supplied `batch_id` must be re-validated against the looked-up product |
| Browser → `POST /m/writeoff` | Untrusted `reason_code` reaches `register_writeoff` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-11-13 | Tampering / Elevation of Privilege | `/m/writeoff/step/batch-pick` posted `batch_id` | mitigate | Re-validate `candidate.product_id == product.id` (T-09-08 precedent) |
| T-11-14 | Tampering | `reason_code` posted value | mitigate | `register_writeoff`'s existing service-layer allow-list validation against `WRITEOFF_REASONS` is unchanged — this plan's `<select>`-replacement is a rendering choice only, the server never trusts the radio's visible label |
| T-11-15 | Repudiation | Over-removal bypass via a client that never re-POSTs `confirm=1` | mitigate | Zero Operation rows are written until `register_writeoff` returns without `oversell` set |
| T-11-CSRF | Tampering | CSRF | accept | Unchanged app-wide posture |

</threat_model>
