<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser → `GET /m/returns?sale_id=&product_id=&origin_op_id=` | Untrusted query params used to resolve a specific sale Operation for a return |
| Browser → `POST /m/returns` | Untrusted `origin_op_id`/`qty` reach `register_return` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-11-22 | Tampering / Elevation of Privilege | `origin_op_id`/`sale_id`/`product_id` query params | mitigate | `_resolve_origin` only accepts an Operation whose `type == "sale"` AND whose `sale_id` is non-null before treating it as a valid return origin — identical guard to desktop's `_resolve_origin`, replicated verbatim, so a tampered id pointing at a non-sale Operation (e.g. a `writeoff`) is rejected |
| T-11-23 | Tampering | `qty` posted value exceeding `returnable_qty` | mitigate | `register_return`'s existing service-layer cap (unchanged) rejects an over-return with zero writes, exactly as desktop |
| T-11-24 | Information Disclosure | Batch comment/product name rendered in `return_confirm.html` | mitigate | Jinja autoescape only, `\| safe` never applied |
| T-11-CSRF | Tampering | CSRF | accept | Unchanged app-wide posture |

</threat_model>
