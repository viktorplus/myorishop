# Phase 11 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (server-rendered Jinja2 + HTMX 2.0.10 vendored; plain CSS, no framework, no CDN) |
| Preset | not applicable |
| Component library | none — extends existing `app/static/style.css` (scale established in Phase 2, carried unchanged through Phases 3–10) |
| Icon library | none — text labels only (offline-first, no icon dependency). A bare `←` character is used as plain text, not an icon asset. |
| Font | system-ui stack (existing, unchanged): `system-ui, -apple-system, "Segoe UI", sans-serif` |

**shadcn gate:** not applicable — stack is FastAPI + Jinja2 + HTMX (not React/Next.js/Vite); confirmed no `components.json`, `package.json`, or `tailwind.config.*` in the repo. Registry safety gate: not applicable.

**Token carry-over (source: 07/08/09-UI-SPEC.md, `app/static/style.css`):** every base token (spacing/type/color) below is IDENTICAL to Phases 2–10. This phase is purely additive per ROADMAP criterion 4 — it adds a NEW namespace of mobile-only screens/templates/CSS rules and does NOT modify any existing desktop template, class, or color. UI language: Russian, sentence case, unchanged.

**New CSS this phase (mobile-only, additive):** a small set of new classes scoped to `/m/...` screens, reusing existing token VALUES only (no new colors, no new font sizes/weights, no new spacing values except the one documented touch-target exception below). See Spacing/Typography/Color sections for the exact rules.

**Phone-width breakpoint (CONTEXT discretion, resolved):** `600px` viewport width is the design threshold for "phone-width" — below 600px the operator is in scope for the D-02 auto-redirect into `/m/...`; at 600px and above they land on the unchanged desktop pages. 600px is chosen because it comfortably excludes small tablets (768px+) while covering all common phone portrait widths (360–599px), and gives the mobile screens themselves a single fixed design target: **360–599px viewport width, portrait.** The exact redirect MECHANISM (server-side UA sniff vs. client-side `matchMedia` + redirect) is an implementation decision, not a design-contract concern — flag for `RESEARCH.md`/planner.

## Spacing Scale

Declared values (must be multiples of 4). Identical to Phases 2–10 — already implemented in `style.css`.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-free inline gaps; `.error` top margin |
| sm | 8px | Compact spacing; table cell padding; gap between wizard step label and its input |
| md | 16px | Default `.field` spacing; mobile-tile grid gap; wizard screen top/side padding |
| lg | 24px | Section padding; `.form-actions` top margin; gap between a mobile batch/basket card and the one below it |
| xl | 32px | Page top padding under nav (desktop, unchanged) |
| 2xl | 48px | Major section breaks; `.empty-state` vertical padding |
| 3xl | 64px | not used this phase |

**Exception — 44px minimum touch target (new this phase, mobile-only):** every tappable element on `/m/...` screens (home tile, batch/basket card, wizard primary/secondary button, «← Главная» back link tap area, «Назад» step-back button) has a minimum height of 44px, even where its visual content would fit in less. This is the one deviation from the strict 4/8/16/24/32/48 ramp, scoped exclusively to mobile interactive elements (per the agent contract's stated allowance for "icon-only touch targets at 44px" — generalized here to ALL mobile tap targets, since the whole phase is thumb-operated). Desktop control heights (36px inputs/selects, existing button padding) are unchanged.

## Typography

Exactly 4 sizes, 2 weights — identical to Phases 2–10. No new sizes or weights are introduced for mobile; mobile screens are laid out with MORE vertical space between the same-sized text, never smaller text.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body (inputs, card text, wizard field values, batch/basket card lines) | 16px | 400 | 1.5 |
| Label (form labels, muted hints, step indicator, card secondary lines) | 14px | 400 (labels/hints) / 600 (card `<strong>` emphasis, e.g. product name on a card) | 1.5 |
| Heading (mobile screen title, h2 equivalent) | 20px | 600 | 1.2 |
| Display (h1 — mobile home title «MyOriShop» and each screen's top-level `<h1>`, e.g. «Продажа») | 24px | 600 | 1.2 |

- Home tile label uses Body size (16px) at weight 600 — still within the declared 2-weight system (400/600), no new weight.
- Numeric values on batch/basket cards (price, quantity) use the existing `.num` treatment (right-aligned where tabular, tabular-nums) — same convention as desktop tables.
- Money via the existing `| cents` filter; expiry via the existing `| ru_date` filter (`dd.mm.yyyy`). No new formatting.
- No italics, no uppercase, no color-coded text (unchanged convention).

## Color

Identical palette to Phases 2–10 — **no new color values introduced.** Mobile screens reuse the exact same roles; nothing gets a bespoke "mobile" color.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#fafafa` | Page background (`/m/...` screens, unchanged from desktop); default text `#222` |
| Secondary (30%) | `#ffffff` | Home tiles, batch/basket cards, wizard screen surfaces, borders `#d9d9d9` |
| Accent (10%) | `#2563eb` | See reserved list below; hover/active `#1d4ed8`; selection tint `#e8effd` (existing `mark`/`.selected-batch` tint, reused verbatim for the mobile selected-batch card) |
| Destructive | `#b91c1c` | «…всё равно» danger buttons (oversell/over-removal/price-floor warnings, same copy as desktop); `.error-block` background `#fdecec` |

**Accent reserved for (mobile additions, appended to the existing desktop list — no new elements beyond these):** the wizard's primary CTA button on each step («Далее», «Оформить продажу», «Списать», etc.), the «← Главная» and «Назад» text links (native `<a>`/link-style-`<button>` color, unchanged global `a { color: #2563eb }` rule), the selected-batch/selected-basket-line card highlight tint `#e8effd`, and input/select/radio/card `:focus-visible` outline. **Home tiles do NOT use accent fill** — they are secondary-surface (`#ffffff`) cards with the existing `1px solid #d9d9d9` border, same visual weight as `button.secondary`, so the 60/30/10 ratio holds even with 8 tiles on screen at once (tiles are surface, not accent).

Muted text (`.muted`, `#6b7280`, unchanged): step indicator («Шаг 2 из 4»), batch/basket card secondary lines (expiry «без срока», comment placeholder «—»), empty-state body copy, auto-select note.

**No new semantic colors:** every mobile warning (oversell, over-removal, price-floor) reuses the EXACT `.error-block` treatment already established — one destructive color, no separate "warning" yellow, no new "success" green for the post-submit confirmation screen (success copy uses default body text, not a color role).

## Interaction Contract

- **Auto-redirect (D-02):** landing on `/` (or any desktop entry point) from a viewport under 600px sends the operator into `/m/`. Desktop viewports (≥600px) are unaffected — this is the ONE new routing behavior; every existing desktop route continues to render exactly as before at desktop widths (ROADMAP criterion 4).
- **No persistent nav on mobile (D-03/D-04):** every `/m/...` screen is single-purpose. The only two navigation controls on any screen are the top «← Главная» exit link (always visible, always goes to `/m/`) and, mid-wizard, the bottom «Назад» step-back button. There is no tab bar, no hamburger, no breadcrumb trail beyond the step indicator text.
- **One action per screen (D-05):** no screen combines more than one "decision" — quantity+price and destination+quantity are treated as a single combined decision (consistent with how the desktop form already groups them), not two. Reason/note and mode/value are the only other same-screen groupings, justified the same way.
- **Batch cards, not tables (D-07):** every batch-selection surface across Sale/Write-off/Return/Correction/Transfer uses the SAME full-width tappable card component (`.mobile-card`), never a table, never a truncated summary. Tapping selects; the selected card gets the `#e8effd` tint; the wizard advances to the next step automatically on tap (no separate "confirm selection" sub-step) UNLESS auto-selected as the sole batch (D-06 note shown instead).
- **Guardrails unchanged (D-06):** min-price and oversell/over-removal warnings fire from the SAME service-layer checks as desktop, at the equivalent wizard step (price-floor at the "Количество и цена" step for Sale; oversell/over-removal at the confirmation step for all). Warn-but-allow is zero-write until the danger button is tapped — identical guarantee to desktop.
- **Save-and-next ergonomics preserved (mobile-adapted):** desktop's "swap whole form, refocus code field" pattern becomes, on mobile, "show a success screen with «Добавить ещё» / «На главную»" — the operator explicitly chooses to continue rather than the form silently resetting, since a silent reset is harder to notice on a phone screen than on a desktop form.
- **Keyboard/accessibility:** all mobile inputs keep `inputmode` hints (numeric/decimal) already used by desktop forms; every tappable card/tile/button meets the 44px touch-target minimum; the existing global `:focus-visible` outline rule (`2px solid #2563eb`) applies unchanged to mobile controls for any operator using an external keyboard.
- **No new JS beyond vendored HTMX idioms already in the repo** — wizard step transitions use the same `hx-get`/`hx-target`/`hx-swap` patterns as desktop's save-and-next forms. No modals, no toasts, no client-side routing framework.
- **Manual UAT gates (flag for planner/executor):** (1) a phone-width landing on `/` redirects into `/m/`; (2) a desktop-width landing on `/` is unaffected; (3) every one of the 8 home tiles reaches its mapped screen; (4) a 2+-batch product shows the full 4-field card list with no truncation at every batch-selection point (sale/write-off/return/correction/transfer); (5) the price-floor and oversell/over-removal warnings on mobile use the exact same copy and zero-write guarantee as desktop; (6) desktop pages (category page, batch picker, transfer form, expiry report, and all pre-existing pages) are pixel-for-pixel unchanged at desktop widths after this phase ships.
