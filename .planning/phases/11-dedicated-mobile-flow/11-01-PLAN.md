<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser (phone-width client) → `app/templates/base.html` | Untrusted viewport width read client-side only; nothing is sent to or trusted by the server |
| Client-supplied `batch_id`/`code` echoed into `batch_card_picker.html`'s `hx-vals` | The values rendered into `hx-vals` on THIS pass originate from the server's own `batches` query, not client input — but the batch_id value the NEXT request sends back is untrusted and must be re-validated by the consuming route (Plans 03-07), not by this partial |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-11-01 | Tampering | `app/templates/mobile_partials/batch_card_picker.html` hidden batch_id echo | mitigate | This plan only RENDERS the value; every consuming route built in Plans 03-07 MUST re-validate `candidate.product_id == product.id` before trusting a posted batch_id, exactly like the existing `sales.py`/`writeoffs.py`/`transfers.py`/`corrections.py` batch-pick endpoints (T-09-08 precedent) — enforced as an acceptance criterion in those plans, not here |
| T-11-02 | Information Disclosure | Batch `comment`/`location` rendered in `batch_card_picker.html` | mitigate | Jinja autoescape only, `\| safe` never applied — verified by this plan's own acceptance criteria (grep for absence of `\| safe`) |
| T-11-03 | Tampering | Inline redirect `<script>` in `app/templates/base.html` | accept | The script reads only `window.location.pathname` and `matchMedia` (no user input, no echoed server data); zero injection surface |
| T-11-SC | Tampering | npm/pip/cargo installs | accept | Zero new packages this plan — Package Legitimacy Gate is not applicable per RESEARCH.md's "Package Legitimacy Audit" section |

</threat_model>
