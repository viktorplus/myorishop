# Phase 17 — UI Design Contract

> Visual and interaction contract for the READ-ONLY financial analytics added to the «Финансы» section: three dashboard metric tiles + a light period selector on `/finance` and `/m/finance`, and a new period cash-flow report sub-page `/finance/report` (+ mobile) with a period-scoped CSV export. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

> **Scope note — REUSE-HEAVY, not greenfield.** The visual system already exists in `app/static/style.css` (plain CSS, no build step, offline). This contract does **not** invent tokens — it records the existing ones verbatim (identical to the Phase 16 contract) so the checker can validate against them, then locks the three new surfaces (metric tiles, the `/finance/report` page + results partial, the CSV export button) by mirroring existing patterns (`cash_balance` / `reports_sales` / `period_filter` / `sales_report_results`). No new dependencies, no motion system, no icon library, no CDN. The single genuinely-new CSS is a `.metric-tile` / `.metric-grid` block that reuses only existing token values (white surface, `#d9d9d9` border, 4px radius, 16px padding, the 2-col mobile grid already proven by `.mobile-tile-grid`).

## Component Contracts (this phase)

### A. Dashboard metric tiles + light period selector (D-04 / D-04b / FIN-10/11/12)

Placement: on `/finance` and `/m/finance`, in a new «Показатели» section **above** the existing balance/forms/history (which stay untouched, D-04). The three tiles render in a `.metric-grid` (a new class: desktop `grid-template-columns: repeat(3, 1fr)`, mobile falls back to the existing 2-col `.mobile-tile-grid` rhythm or single column — planner discretion, mirror `.mobile-tile-grid`). Each `.metric-tile` is a white surface (`#ffffff`), `1px solid #d9d9d9`, `border-radius: 4px`, `padding: 16px` — visually identical to `.mobile-tile` / `.customer-chip`, no new token.

Each tile contains, top-to-bottom: an `h2`-or-`.tile-label` heading (14px/16px), the figure `<p class="num"><strong>{{ … | cents }}</strong></p>`, then an optional `.muted` caveat line.

| Tile | Heading | Value | Caveat line (`.muted`, 14px) | Period-linked? |
|------|---------|-------|------------------------------|----------------|
| Gross profit (FIN-10) | Валовая прибыль | `sales_profit_report(...).totals.profit_cents \| cents` | If `cost_unknown_count > 0`: «{n} продаж без себестоимости — не учтены.» | YES — follows the light period selector |
| Net profit (FIN-11) | Чистая прибыль | `(profit_cents + cash_expense_total) \| cents` | **MANDATORY (D-01b):** «Денежный поток: валовая прибыль минус снятия и возвраты за период. Это не бухгалтерская прибыль.» | YES — follows the light period selector |
| Stock valuation (FIN-12) | Стоимость склада | two figures: «По закупке: {cost_value \| cents}» and «По продаже: {sale_value \| cents}» | If `cost_unknown_count`/`sale_unknown_count > 0`: «{n} товаров без цены закупки / продажи — не учтены.» (two separate `.muted` lines as needed) | **NO — point-in-time (D-02b), ignores the selector** |

**Net-profit caveat is a HARD requirement, not decoration (D-01b / RESEARCH Pitfall 2).** It must be visibly attached to the net tile (a `.muted` line directly under the number is sufficient; a `title=` tooltip alone is NOT — it is not discoverable on touch/mobile). Copy must convey "cash-outflow view, not accounting profit".

**Light period selector (D-04b):** reuse `partials/period_filter.html` with a **distinct** `period_target` (e.g. `#finance-metrics`) and `period_action` pointing at a dedicated tiles endpoint (e.g. `/finance/metrics` desktop, `/m/finance/metrics` mobile — planner discretion per RESEARCH Discrepancy 5/Assumption A1). The gross + net tiles re-render on period change; the stock tile shows a small `.muted` note «на текущий момент» so the operator sees it is deliberately period-independent (the "clear UX cue they are independent" required by D-04b). The two period controls (this light one vs the full one on `/finance/report`) must never share a `period_target`.

**Zero / empty states (tiles):**
- Gross profit with no sales in period → renders «0,00» (not blank), matching `format_cents(0)`.
- Net profit with no movements → «0,00».
- Stock valuation with an empty warehouse → «По закупке: 0,00 / По продаже: 0,00».
- All-null-price stock → «0,00» plus the unknown-count caveat (never a blank or "None").

### B. `/finance/report` cash-flow report page (D-04a / D-05 / FIN-08)

A near-verbatim structural clone of `pages/reports_sales.html` + `partials/sales_report_results.html`, but **newly assembled** for mobile (RESEARCH Discrepancy 1 — no `/m/reports/sales` precedent exists; build `/m/finance/report` from `period_filter.html` + the `finance_base` prefix, not by copying a mobile template that does not exist).

Page structure (mirror `reports_sales.html`):
```
<h1>Движения кассы за период</h1>
{% with period_action = "/finance/report", period_target = "#cashflow-results" %}
  {% include "partials/period_filter.html" %}
{% endwith %}
<a class="button" href="/finance/report.csv?from={{ from_date }}&to={{ to_date }}">Скачать CSV</a>
<div id="cashflow-results">{% include "partials/cash_flow_report.html" %}</div>
```

Route contract: plain GET → full page (chrome + filter + results); `HX-Request` header → the results partial only (`cash_flow_report.html`), swapped into `#cashflow-results` (identical branch to `/reports/sales`). Mobile uses `period_action="/m/finance/report"`, `finance_base="/m/finance"`, and its own results partial (or a shared one — planner discretion, D-04c).

**Results partial (`cash_flow_report.html`) contract (D-05):** two labelled sections whose numbers MUST agree with the net-profit tile definition (D-01), rendered as tables mirroring `sales_report_results.html` (`table` → `thead` → `tbody`, `.num` right-aligned money cells):

- **«Приход»** — income rows: «Продажи» (sum of `sale` credits) and «Внесения» (sum of `deposit_*` credits), each with a signed total. Category labels come from `CASH_CATEGORIES` / `CASH_BUCKET_LABELS` (never hardcoded).
- **«Расход»** — expense rows grouped per `withdrawal_*` category (label from `CASH_CATEGORIES`: Оплата поставщику / Зарплата / Аренда / Коммунальные / Прочее) plus a «Возвраты» row (the `return` bucket). These are exactly the categories that feed `cash_expense_total` (D-01a), so the report's expense subtotal equals the value subtracted in the net-profit tile — the report and the tile can never disagree.
- A closing subtotal/итог row per section (and optionally a net line) using `.num` + `<strong>`.

Row grouping (single «Продажи» line vs per-category expansion; whether returns net visually against income) = planner discretion within D-05, provided the totals reconcile with D-01.

**Empty / zero state (report):** no movements in the period → `<p class="empty-state muted">За выбранный период движений не было.</p>` (mirrors `sales_report_results.html` line 10). Load failure (`report is none`, e.g. bad dates that slipped past validation) → the existing `.error-block` pattern with «Не удалось загрузить отчёт. Проверьте даты и попробуйте ещё раз.»

**Pagination:** start WITHOUT it (single-operator period counts are small, RESEARCH Open Q2). If a real period exceeds `LIST_PAGE_SIZE` (20) movement rows and a raw movement list is shown, add the shared `page_window` + `partials/pagination.html` bar (accent `.current-page`) exactly as `routes/history.py` does — do not hand-roll.

### C. CSV export button (D-03 / FIN-09)

A plain `<a class="button" href="/finance/report.csv?from=…&to=…">Скачать CSV</a>` on `/finance/report` (and mobile). **Must be a plain `<a href>` download — NOT `hx-get`** (RESEARCH anti-pattern: htmx would swap the CSV text into the DOM; pinned by `test_web_export_page`). The `from`/`to` query params echo the currently-resolved period (`from_date`/`to_date` isoformat) so the export scope always matches the on-screen report. Accent styling (it is a legitimate primary export action; added to the accent reserved list above). The download itself (BOM-once, `;` delimiter, `_csv_safe`, `CASH_CATEGORIES` label, `format_cents` amount, `note or ""` for NULL notes) is a service-layer concern (D-03a/D-03c, RESEARCH Pattern 3) — not visual, but the button label and its plain-link nature are the UI contract.

### D. Untouched surfaces (regression guard)

The Phase 15–16 balance (`cash_balance.html`), the withdraw/deposit forms (`withdraw_form.html`/`deposit_form.html`), and the cash-movement history block (`cash_history_rows.html` desktop / `cash_history_cards.html` + load-more mobile) stay **byte-for-byte unchanged** except the optional opportunistic `{{ mv.note or "" }}` NULL-note cosmetic fix noted in RESEARCH Discrepancy 6 (a Phase-16 carry-over, not a Phase-17 requirement). Adding the tiles must not alter their markup, ids, or hx-targets.

## Resolved Open Questions / Assumptions

### Q1 — Metric-tile visual → **new `.metric-tile`/`.metric-grid` reusing existing tokens only**
No tile/card component exists for desktop (only `.mobile-tile` for the home grid). Rather than shoehorn the metrics into a bare table, add a minimal `.metric-grid` (grid, 16px gap) + `.metric-tile` (white `#ffffff`, `1px solid #d9d9d9`, `4px` radius, `16px` padding) — every value is an existing token; no new color, size, or spacing is introduced. On mobile, reuse the `.mobile-tile-grid` 2-col rhythm (or single column). Assumption: this matches the codebase's "reuse the token scale, add only structural CSS" discipline (same rationale as `.customer-chip` / `.batch-picker`). Risk if wrong: LOW — purely structural, no token drift.

### Q2 — Two period controls independence (D-04b) → **distinct `period_target` + dedicated tiles endpoint + explicit "на текущий момент" cue on the stock tile**
The dashboard selector targets `#finance-metrics` via `/finance/metrics`; the report selector targets `#cashflow-results` via `/finance/report`. They never share a param or target (RESEARCH Pitfall 3). The stock tile carries «на текущий момент» so it visibly does not participate in either period. Assumption A1 from RESEARCH (dedicated `/finance/metrics` partial endpoint) is adopted as the cleanest; planner may instead add `from`/`to` to `/finance` provided independence holds.

### Q3 — Net-profit caveat placement → **visible `.muted` line under the number, not a `title=` tooltip alone**
D-01b says "label/tooltip". A tooltip is not discoverable on touch/mobile (the app has a first-class mobile surface), so the caveat is rendered as an always-visible `.muted` line. This satisfies "surface in UI, do not hide" unambiguously across desktop and mobile.

### Q4 — Sign coloring of metric numbers → **none**
Negative net profit (or a negative period cash flow) renders as plain `format_cents` in default text color. Red is reserved for validation errors only; coloring a legitimate negative number red would misread as an error state.
