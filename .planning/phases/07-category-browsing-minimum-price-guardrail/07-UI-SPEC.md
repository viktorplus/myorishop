# Phase 7 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (server-rendered Jinja2 + HTMX 2.0.10 vendored; plain CSS, no framework, no CDN) |
| Preset | not applicable |
| Component library | none — reuse existing `app/static/style.css` (spec scale established in Phase 2, carried through Phases 3–6) |
| Icon library | none — text labels only (offline-first, no icon dependency) |
| Font | system-ui stack (existing): `system-ui, -apple-system, "Segoe UI", sans-serif` |

**shadcn gate:** not applicable — stack is FastAPI + Jinja2 + HTMX (not React/Next.js/Vite); no `components.json` in the repo. Registry safety gate: not applicable.

**Locked carry-over from Phases 2–6 (source: 02–06-UI-SPEC.md, `app/static/style.css`):** every token below is the SAME as Phases 2–6 — this phase adds one new screen (`/categories`), one new optional field on the product form, and one new warning partial on the sale flow, NOT new tokens. All styling lives in `app/static/style.css`; new CSS is added ONLY where existing classes cannot express a layout. UI language: Russian (`base.html lang="ru"`). Product names are untrusted stored input rendered on the new category page exactly like every other catalog view: autoescape only, never `|safe`. Money is integer cents rendered via the existing `cents` filter (comma decimal) — the new `min_sale_cents` field follows the identical `to_cents()`/`inputmode="decimal"`/`placeholder="0,00"` pattern as `cost`/`sale`/`catalog` (CONTEXT D-07).

## Spacing Scale

Declared values (must be multiples of 4). Identical to Phases 2–6 — already implemented in `style.css`.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Gap between label and input; error text top margin; muted "(необязательно)" gap next to a label |
| sm | 8px | Table cell padding (8px 12px); button padding vertical; form row gaps |
| md | 16px | Default gap between form fields (`.field`); button horizontal padding; error-block padding; gap between two stacked warning blocks (oversell + price-floor) |
| lg | 24px | Gap between page sections (e.g. one category group's table → the next category group's heading; form → price history) |
| xl | 32px | Page top padding under nav |
| 2xl | 48px | Empty-state block vertical padding (`.empty-state`) |
| 3xl | 64px | not used this phase |

Exceptions: 12px — dense table cell horizontal padding only (existing `th, td { padding: 8px 12px }`, carried from Phase 2). Control heights derived from scale: inputs/buttons 36px (8px vertical padding + 16px text + borders) — the new `min_sale_cents` input reuses this unchanged.

## Typography

Exactly 4 sizes, 2 weights — identical to Phases 2–6. No other values permitted.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body (inputs, table cells, paragraphs) | 16px | 400 | 1.5 |
| Label (form labels, table headers, muted hints, "(необязательно)" markers) | 14px | 400 (labels/hints) / 600 (table `th`) | 1.5 |
| Heading (h2: each category group heading on `/categories`, e.g. «Косметика», «Без категории») | 20px | 600 | 1.2 |
| Display (h1 page title: «Товары на складе») | 24px | 600 | 1.2 |

- Weights: 400 (regular) and 600 (semibold) only.
- Numeric columns (Остаток, Закупочная, Продажа, Каталог) on `/categories`: 16px, 400, right-aligned, `font-variant-numeric: tabular-nums` (existing `.num` class) — identical treatment to every other catalog/report table.
- No italics, no uppercase-tracking, no color-coded numbers (unchanged project convention).

## Color

Identical to Phases 2–6 — no new colors introduced. Semantic color budget stays at exactly one destructive color; the price-floor warning reuses the EXISTING oversell warning's visual treatment verbatim (CONTEXT D-11 — "same `.error-block` / `button.danger` styling"), not a new "warning" color.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#fafafa` | Page background (existing); text `#222` |
| Secondary (30%) | `#ffffff` | Tables, form surfaces, nav bar; borders `#d9d9d9`; table header background `#f0f0f0` |
| Accent (10%) | `#2563eb` | See reserved list below; hover/active shade `#1d4ed8` |
| Destructive | `#b91c1c` | Error-block text/background (`#fdecec` bg) for BOTH the oversell warning and the new price-floor warning; `button.danger` («Продать всё равно» on both warning blocks); existing "Удалить товар" (unaffected by this phase) |

Accent reserved for: the new `/categories` nav link (including its active state, accent + 600 weight when path starts with `/categories` — same pattern as every other nav entry), text links (including the «Изменить» edit link in the category table's Действия column), input `:focus` border/outline. Nothing else — no new accent usage this phase.

**No color coding for category groups or table rows:** category headings are plain h2 text, not tinted or badge-styled; the "Без категории" bucket uses the identical h2/table treatment as every named category, distinguished only by its text label and its position (sorted last) — never a different color (CONTEXT D-04, project convention "no color coding for report/browsing data" from Phase 6 carried forward).

Muted text (`.muted`, `#6b7280`): the «(необязательно)» marker on the new «Минимальная цена продажи» label, the em-dash placeholder for unset Закупочная/Продажа/Каталог cells (existing pattern from `product_rows.html`), empty-state body text.

## Interaction Contract

- **`/categories` is a plain full-page GET.** No HTMX partial swap needed — CAT-01's scope is read-only browsing with no filter/search control this phase (unlike `/products`), so there is nothing to re-apply in place. Keep it simple; do not add HTMX machinery where a static page load already satisfies the requirement (same principle Phase 6 applied to `/reports/stock`).
- **Price-floor check runs server-side, before any DB write** — identical warn-but-allow contract to the existing oversell check (CONTEXT D-11/D-08/D-09): a `confirm != "1"` gate produces zero writes and returns the basket form intact with the warning block(s) rendered above it; `confirm=1` on resubmit performs the write. This is a hard interaction rule carried from `register_sale`'s existing oversell precedent, not a new pattern.
- **Boundary is strict less-than** (CONTEXT D-10): `line_price_cents < product.min_sale_cents` triggers the warning; equal-to-minimum passes silently — mirrors the existing oversell boundary (`requested > quantity`, hitting the limit exactly is fine).
- **`min_sale_cents` is checked with `is not None`, never a bare `or`** (CONTEXT D-06) — an explicit `0` must stay meaningful and never trigger the warning if unset. This is a backend correctness rule surfaced here because it directly determines what the operator sees: a product with no minimum configured must NEVER show the warning under any entered price, including `0,00`.
- **Money input:** `type="text" inputmode="decimal"`, accepts comma and dot (existing `to_cents()`), displayed via the existing `cents` filter — no new parsing rule for the new field.
- **Keyboard-first (core value: fast operator entry) still applies:** the new product-form field is reachable by Tab in document order (after «Цена продажи», before «Цена по каталогу»); the new warning block's two buttons are reachable by Tab and operable via Enter/Space, consistent with `sale_oversell.html`'s existing buttons; visible 2px accent focus outline on every interactive element — never `outline: none` without replacement.
- **No client-side JS beyond vendored HTMX.** No modals, no toast libraries; the "Вернуться к корзине" dismiss uses the existing `hx-on:click` DOM-removal pattern already proven in `sale_oversell.html` — no new JS pattern introduced.
- **Manual UAT gates:** a product with `min_sale_cents` left unset never shows the warning at any entered price, including `0,00` (CONTEXT D-06/success criterion 4); a product with `min_sale_cents` explicitly set to `0` is respected — selling at a negative-impossible price is moot, but the field must round-trip as `0`, not `NULL`, through save/reload (success criterion 4); a submission that trips BOTH oversell and the price floor shows both blocks stacked and resolves both with one «Продать всё равно» click (CONTEXT D-11); a product with `category IS NULL` or empty string appears under «Без категории», never silently dropped from `/categories` (CONTEXT D-04/success criterion 1); a soft-deleted product never appears on `/categories` (CONTEXT D-05).
