<objective>
Fix the mobile Корректировка (stock correction) wizard's context/navigation gaps: add the
visible code/name/warehouse header to its 3 hidden-input-only steps (Партия, Режим, Значение —
D-01/D-02/D-03), and fix all 3 of its "Назад" links so each returns to its own immediate
predecessor step with state intact (D-05), including step 2's own "Назад" per the
open_question_resolution decided for this phase (see below). This plan also creates the shared
`_wizard_header.html` partial that Plan 13-02 (write-off migration) depends on and reuses verbatim.

**Logged decision (open_question_resolution, superseding 13-CONTEXT.md D-05's ambiguity):**
13-RESEARCH.md's Open Question asked whether corrections' step-2 ("Партия"→"Товар") "Назад"
needs the same hx-get/hx-post+fragment fix as steps 3/4, since receipts' own step-2 "Назад" is
also a plain link and D-06 called receipts "already correct." The orchestrator resolved this:
Success Criterion #2 is a uniformity requirement across every step of every wizard with no
carve-out for "already correct" steps that happen to be plain links. This plan therefore converts
corrections' step-2 "Назад" to `hx-get` + a fragment response too (not left as a plain link).
Plans 13-03/13-04 apply the identical fix to receipts' and transfers' own step-2 "Назад" links for
the same reason (discovered via direct file verification during planning — both are plain links
today, same as corrections was).

Purpose: Phase 13 Success Criteria #1 and #2 (ROADMAP.md) — visible context and consistent
back-navigation across every mobile wizard. Corrections is the first of the 5 wizards to be fixed
and establishes the shared header partial the rest of the phase reuses.
Output: `_wizard_header.html` (new shared partial), corrections' 3 intermediate step templates
updated, `mobile_pages/corrections.html` refactored around a new `corrections_step_product.html`
fragment, `app/routes/mobile_corrections.py` threading `name`/`warehouse_name` and serving step 1
as either a full page or a bare htmx fragment.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Operator's browser → mobile_corrections.py routes | Untrusted form/query fields: `code`, `batch_id`, `name`, `batch_qty`, `mode` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Tampering / Information Disclosure | `_wizard_header.html` rendering `{{ name }}`/`{{ warehouse_name }}` | mitigate | Jinja2 autoescaping only (default) — never apply `\|safe` to `name`/`warehouse_name`; both are untrusted stored product/warehouse text (13-RESEARCH.md Security Domain, "Known Threat Patterns"). |
| T-13-02 | Tampering | `warehouse_name` resolved from client-carried `batch_id` in step/mode, step/value, mobile_correction_create | mitigate | Re-validate `candidate.product_id == product.id` before using the batch's `warehouse_id` for display — never trust `batch_id` directly (mirrors the established `_pick_batch`-style ownership check used everywhere else in this codebase). |
| T-13-03 | Tampering | `code` query param on the new HX-Request branch of `GET /m/corrections` | accept | `code` only pre-fills a text input; it is re-validated identically to manual entry by the existing `select(Product)...` lookups downstream — no new trust boundary introduced. |
</threat_model>
