# Phase 13 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase type:** fix-and-consistency. This phase adds **zero new visual tokens** — it applies the design system already shipped in Phase 2/Phase 11 (`app/static/style.css`) uniformly across 5 existing mobile wizards. Every value below is a citation of an existing rule, not a new decision, except where marked "(new, this phase)".

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — server-rendered Jinja2 + HTMX, no component framework, no build step (project is FastAPI/Jinja2/htmx per `CLAUDE.md`; not React/Next/Vite, so the shadcn gate does not apply) |
| Preset | not applicable |
| Component library | none — hand-written semantic HTML + `app/static/style.css` utility classes |
| Icon library | none — this project is text-only (no icon font/SVG sprite in use); do not introduce one for this phase's quick-action buttons (D-08/D-09) — use plain text button labels ("Продать" / "Принять") matching every other mobile action button |
| Font | `system-ui, -apple-system, "Segoe UI", sans-serif` (`app/static/style.css` line 5) |

## Spacing Scale

Declared values (must be multiples of 4) — unchanged from the existing scale in `app/static/style.css`:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon/inline gaps (e.g. `label` margin-bottom) |
| sm | 8px | Compact element spacing (`.mobile-actions` button gap uses 16, not 8 — see below) |
| md | 16px | Default element spacing, `.mobile-shell` padding, `.mobile-actions` gap |
| lg | 24px | `.mobile-card` bottom margin, section padding |
| xl | 32px | Layout gaps (`.container` bottom padding) |
| 2xl | 48px | Major section breaks (`.empty-state` padding) |
| 3xl | 64px | Page-level spacing (unused on mobile so far) |

Exceptions:
- **44px minimum touch target** — already established (`.mobile-back`, `.mobile-card`, `.mobile-actions button/a.button` all set `min-height: 44px`). The new quick-action buttons on `search_product_detail.html` (D-08/D-09) and every newly-added "Назад" button in write-off/corrections MUST use the same `.mobile-actions` wrapper so they inherit this 44px minimum automatically — do not hand-roll a smaller touch target.

## Typography

Unchanged from the existing mobile/global scale — 4 sizes, 2 weights:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Display (h1) | 24px | 600 | 1.2 |
| Heading (h2) | 20px | 600 | 1.2 |
| Body (p, `.mobile-card` text, buttons) | 16px | 400 | 1.5 |
| Label / muted / step-indicator (`label`, `.muted`, `.mobile-step-indicator`, `.error`) | 14px | 400 | 1.5 |

New visible code/name/warehouse lines (D-01/D-02) MUST use **Body size (16px/400)** with the code wrapped in `<strong>` (weight 600 inline, not a new weight token — `<strong>` inherits the existing bold rendering already used identically in `sale_step_batch.html`/`transfers_step_dest.html`). Do not introduce a third weight or a fifth size for this text.

## Color

Unchanged from the existing palette (`app/static/style.css` header comment: "accent #2563eb, destructive #b91c1c"):

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#fafafa` (body background) | Page background behind `.mobile-shell` |
| Secondary (30%) | `#ffffff` with `#d9d9d9` 1px border | `.mobile-card`, `.mobile-tile`, inputs, nav surface |
| Accent (10%) | `#2563eb` (buttons/links/focus); tint `#e8effd` (selection/search-match) | See "Accent reserved for" below |
| Destructive | `#b91c1c` | `.error`, `.error-block`, oversell/over-transfer/over-removal warning partials (`*_warning.html`), `button.danger` |

Accent reserved for:
- Primary button background/border (`button`, `a.button` default — includes new "Продать"/"Принять" quick-action buttons, which are **primary** actions, not secondary)
- Links (`a`)
- `:focus-visible` outline (2px solid `#2563eb`, offset 1px)
- Selected-state tint `#e8effd` on `.mobile-card.selected` / `.batch-picker tr.selected-batch` / `mark` — this is the ONLY new-to-this-phase reuse: the corrections `mode` radio cards (`corrections_step_mode.html`) already apply `.selected`, no new tint needed

No new color role is introduced by this phase. Do not invent a distinct color for the warehouse line (D-02) or the step indicator's "Корзина" text (D-07) — both use existing `.muted` (`#6b7280`) / body-text treatment as specified per-element below.

## Phase-Specific Interaction Contracts

These four contracts are the actual deliverable of this phase (UI-02 through UI-05). Each restates a `13-CONTEXT.md` decision in prescriptive, copy-paste-ready form so planner/executor do not need to re-derive markup shape.

### Contract A — Visible code/name/warehouse (UI-02, D-01/D-02/D-03)

On every intermediate step template listed below, insert immediately after the step indicator (and after any hidden-input carry-forward block, matching `sale_step_batch.html`'s existing position):

```html
<p><strong>{{ code }}</strong>{% if name %} — {{ name }}{% endif %}</p>
{% if warehouse_name %}<p>Склад: {{ warehouse_name }}</p>{% endif %}
```

- The warehouse line only appears once a batch is selected (batch → warehouse). Steps before batch selection (e.g. corrections/write-off step "Товар") never render it.
- Apply to: `corrections_step_batch.html`, `corrections_step_mode.html`, `corrections_step_value.html`, `writeoff_step_qty.html`, `writeoff_step_reason.html` — currently hidden-input-only, per D-03.
- `writeoff_step_batch.html` already shows code+name partially — align its exact markup to the format above (verify em-dash + `{% if name %}` guard match).
- **Route-level gap to note for planning:** the write-off route (`app/routes/mobile_writeoff.py`) currently does NOT thread `name` as a hidden/carried field past step 1 (`mobile_writeoff_step_qty`/`mobile_writeoff_step_reason` only take `code`/`batch_id`/`qty` — no `name` param). Making the name visible on steps 2-4 requires the route to also accept and re-look-up or carry `name` forward — this is an implementation task the plan must include, not just a template edit.
- Warehouse name is available via the picked `Batch.warehouse_id` → look up the warehouse name (mirrors how `sale_step_batch.html`'s sibling `transfers_step_dest.html` already has `warehouses` in context) — the exact service call is an executor/planner concern, not a design-contract concern.

### Contract B — Consistent "Назад" (UI-03, D-04/D-05/D-06)

Target end state for every wizard step's "Назад" control:

```html
<button type="button" class="secondary"
        hx-post="{{ previous_step_url }}"
        hx-vals="{{ {...carried_fields...} | tojson }}"   {# or hx-include="closest form" #}
        hx-target="#wizard-step-or-shell-id" hx-swap="innerHTML-or-outerHTML-matching-sibling-pattern">Назад</button>
```

- **Sale, receipts, transfers:** already correct (D-06) — no changes.
- **Write-off:** must migrate from full-page `<form method="post">` + `onclick="history.back()"` to the receipts persistent-shell + `hx-post` fragment-swap pattern (D-04). Reference implementation to copy: `mobile_pages/receipts.html` (shell with `#wizard-step`) + `receipts_step_details.html`/`receipts_step_confirm.html` (fragments with `hx-post ... hx-include="closest form"` Назад buttons). This is a structural change to `app/routes/mobile_writeoff.py` and all 3 write-off step templates — not a one-line swap.
- **Corrections:** already fragment-based, but its 3 "Назад" links (`<a class="mobile-back" href="/m/corrections">`) always jump to wizard start, discarding entered state. Fix (D-05): replace with `hx-post` back to the immediately-previous step's endpoint + `hx-include="closest form"`, same technique as receipts. Do not change corrections' forward-navigation shape, only "Назад".
- Button visual treatment unchanged either way: `class="secondary"` (white surface, `#d9d9d9` border, per existing `button.secondary` rule) inside `.mobile-actions`, always the FIRST button in that flex row (existing convention across every wizard).

### Contract C — Sale basket step indicator (UI-04, D-07)

In `sale_basket.html`, immediately before the `<h2>Корзина</h2>` line, add:

```html
<p class="mobile-step-indicator">Корзина</p>
```

Same CSS class, same position-in-DOM convention as every numbered step (`<p class="mobile-step-indicator">Шаг N из 3</p>` in `sale_step_product.html`/`sale_step_batch.html`) — only the text differs. Do not attempt to compute a numeric step for the basket.

### Contract D — Search → wizard quick actions (UI-05, D-08/D-09)

In `search_product_detail.html`, add two plain (non-HTMX) links styled as primary buttons, following the existing plain-navigation convention already documented in that file's header comment ("reached by a plain top-level link click, not HTMX"):

```html
<div class="mobile-actions">
  <a class="button" href="/m/sales?code={{ product.code }}">Продать</a>
  <a class="button" href="/m/receipts?code={{ product.code }}">Принять</a>
</div>
```

- Both buttons render **always**, regardless of stock level (D-09) — zero-stock products remain sellable via the existing oversell-warning pattern; never conditionally hide "Продать" on `stock_rows` being empty.
- `?code=...` query param convention (D-08): `/m/sales` and `/m/receipts` GET handlers (`mobile_sales.py::mobile_sales_page`, `mobile_receipts.py::mobile_receipt_new`) currently hardcode `"code": ""` in their context — both must be extended to accept an optional `code: str = ""` query parameter and pass it through unchanged, landing the operator on each wizard's normal step 1 with the code field pre-filled (not a step-skip, not a new "resume" entry point).
- Button treatment: default `.button`/`button` primary accent styling (not `.secondary`) inside `.mobile-actions` — these are the primary path into a wizard from search, same visual weight as e.g. `sale_basket.html`'s "Оформить продажу".
