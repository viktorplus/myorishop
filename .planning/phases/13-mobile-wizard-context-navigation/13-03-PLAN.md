<objective>
Fix the receipts wizard's step 2 ("Партия") "Назад" control, which today is a plain
`<a class="button secondary" href="/m/receipts">` full-page link that silently discards the typed
code (verified directly in `receipts_step_batch.html` line 53 during planning — 13-CONTEXT.md's
D-06 called receipts "already correct" but this specific transition was not actually audited at
that granularity). Per the open_question_resolution logged in Plan 13-01, this plan converts it to
the same `hx-get` + fragment pattern used everywhere else in this codebase, closing the last
plain-link exception for receipts. This plan also adds the `?code=` GET query-param pre-fill to
`GET /m/receipts` that Plan 13-05's search quick-action link depends on (D-08) — bundled here
because both changes touch the exact same route function.

**Revision (post-13-VALIDATION.md):** step 2 (`receipts_step_batch.html`) was also found to be
hidden-input-only for `name`, with no visible warehouse line at all — it never received the
Contract A header Plan 13-01 gave corrections. Task 3 below closes that UI-02 gap directly, reusing
the `_wizard_header.html` partial Plan 13-01 created. This plan therefore now depends on 13-01 and
moves to wave 2 as a result.

Purpose: Phase 13 Success Criteria #1 (visible code/name/warehouse, UI-02) and #2 (uniform
hx-get/hx-post Назад pattern, no carve-outs), plus UI-05's route-side prerequisite (a "Принять"
quick-action link from search is meaningless until `/m/receipts` actually consumes `?code=`).
Output: `receipts_step_product.html` (new extracted step-1 fragment), `mobile_pages/receipts.html`
refactored around it, `mobile_receipts.py`'s `mobile_receipt_new` serving step 1 as page-or-fragment
with an optional `code` pre-fill, `receipts_step_batch.html`'s "Назад" converted to `hx-get`, and
(Task 3) the same visible code/name/warehouse header every other wizard's Партия step now shows.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Operator's browser → GET /m/receipts | Untrusted query param: `code` |
| Operator's browser → POST /m/receipts/step/batch | Untrusted form field: `warehouse_id` (used for `warehouse_name` display lookup, Task 3) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-07 | Tampering | `?code=` query-param pre-fill on GET /m/receipts | accept | `code` only pre-fills a text input; it is re-validated identically to manual entry by the existing `lookup_prefill`/`select(Product)` calls in `mobile_receipt_step_batch` downstream — no new trust boundary (13-RESEARCH.md Security Domain, "Known Threat Patterns"). |
| T-13-11 | Tampering / Information Disclosure | `warehouse_name` resolved from client-submitted `warehouse_id` in `mobile_receipt_step_batch` (Task 3) | accept | `_warehouse_names(session).get(selected)` is a plain dict lookup restricted to `active_warehouses(session)` — an unrecognized/tampered `warehouse_id` simply yields `None` (the header's `{% if warehouse_name %}` guard omits the line), never an error or a lookup against inactive/other data; same trust level as the existing `code` query param (T-13-07). |
</threat_model>
