<objective>
Make the central server authoritative over mutable reference data (SYNC-05): teach
`apply_merge` to upsert reference rows insert-if-new with server-wins-on-existing
(row-level), apply them in FK-dependency order before the ledger, honor inline
`deleted_at` tombstones without ever resurrecting/deleting a server row from client
input, and resolve a cross-device `Product.code` collision by renaming the incoming
loser (keeping its UUID) while the incumbent keeps the clean code — reporting the rename
for later admin reconciliation. This closes the FK-completeness edge left open in Plan 02:
a missing parent now rejects the whole batch all-or-nothing.

Purpose: Clients push their two ledgers up and pull server-authoritative reference data
down (SYNC-01 topology). So reference rows only travel UP as NEW FK targets their ledger
needs; edits to already-shared rows are owned by the server. Row-level server-wins is
deterministic (no half-merged rows) and matches SYNC-05's literal wording. The one edge
that must never lose data is a duplicate `Product.code` on two devices: rejecting the
loser would orphan its ledger rows, so we rename it and keep its UUID.

Output: the reference-upsert stage inside `apply_merge` plus `_upsert_reference`,
`_resolve_code_collisions`, and `_suffix_code` in `app/services/merge.py`, populating
`MergeReport.reference_inserted` / `reference_server_wins` / `conflicts`; and the SYNC-05
test dimensions in `tests/test_merge.py`.

Design decisions realized here (from 27-RESEARCH.md — recommended defaults, no CONTEXT.md):
- DD-1 (Per-table conflict): insert-if-new + server-wins-on-existing, ROW-level, for every
  mutable table (products, warehouses, customers, dictionary, batches, sales). Client input
  never overwrites/deletes/resurrects an existing server reference row. (SYNC-05, RESEARCH Decision 1)
- DD-2 (Duplicate Product.code): RENAME the incoming loser with a deterministic UUID-derived
  suffix, KEEP its UUID; incumbent keeps the clean code; report in MergeReport.conflicts.
  (SYNC-05, RESEARCH Decision 2)
- DD-1b (Tombstones): inline `deleted_at` only (Product/Warehouse have the column); honor
  incoming deleted_at on a NEW row; never resurrect/delete a server row from client input.
  (SYNC-05, RESEARCH Decision 3)

Scope fence: NO field-level merge, NO client-wins/LWW, NO separate tombstone record type.
NO `dictionary`/`catalog_prices`/`active_catalog` collision-rename (RESEARCH Open Q3: they
flow down, not up — if a duplicate global `code` appears, skip-and-report, do not rename;
default is to exclude them from the push). NO user provisioning / author_id nulling
(Phase 28, Open Q2). NO commit inside apply_merge. NO Alembic migration.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| device reference edits → server master data | A malicious/racing device could try to overwrite, re-price, resurrect, delete, or shadow a server reference row; server-authoritative resolution is the gate. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-27-01 | Tampering | client overwriting server reference data (e.g. re-pricing a product, resurrecting a deleted one) | mitigate | Row-level server-wins-on-existing: an existing UUID is discarded, never field-merged, never `deleted_at`→NULL, never DELETEd from client input (`_upsert_reference`; grep gate: no session.delete/update). Proven by test_server_wins_on_existing_reference + test_tombstone_inline. |
| T-27-02 | Spoofing / Tampering | cross-device `Product.code` hijack (duplicate code to shadow a server product) | mitigate | The incumbent (server) row keeps the clean code; the incoming loser is renamed with a deterministic UUID suffix and reported — it can never displace or masquerade as the server's product. Backstop: partial unique index `uq_products_code_active`. Proven by test_product_code_collision_renamed. |
| T-27-05 | Tampering | SQL injection via reference field values (name/code/etc.) | mitigate | Portable ORM/Core, bound parameters only; the code probe uses `Product.code == row["code"]` bound comparison, never string-interpolated SQL. |
| T-27-07 | Tampering | idempotency abuse on the collision path (non-deterministic rename breaking merge-twice) | mitigate | `_suffix_code` is deterministic in the UUID only (no random/time) so re-merge renames identically and stays a no-op (test_collision_rename_is_deterministic). |
