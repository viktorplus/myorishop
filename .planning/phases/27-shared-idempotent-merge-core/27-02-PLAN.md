<objective>
Land the SYNC-02 + SYNC-03 core: a portable, idempotent ledger-append inside a pure
`apply_merge`, followed by a ledger-derived recompute. This is where "re-sync the same
data twice changes nothing" and "counts stay correct after a merge" actually become
true. Reference-row upserts and conflict resolution are Plan 03 — this plan assumes the
referenced products/warehouses/batches/sales already exist (tests seed them via the
existing fixtures) and focuses on the two append-only ledgers.

Purpose: The whole milestone's "never lose data / never double-count" guarantee reduces
to: insert ledger rows exactly once by UUID, verbatim, then recompute stock and cash
from the ledger. The recompute already exists (`ledger.rebuild_stock`) but commits; we
extract a non-committing `recompute_derived` so the merge stays one caller-owned
transaction. Idempotency is a PORTABLE pre-select set-difference (SQLAlchemy's
`on_conflict_do_nothing` is dialect-specific and project-forbidden).

Output: `recompute_derived(session)` in `app/services/ledger.py` (rebuild_stock delegates
to it then commits — zero behavior change for existing callers); `apply_merge(session,
batch, *, server_now)` + `_insert_new` in `app/services/merge.py` handling the
operations + cash_movements append with MergeReport counters; and the SYNC-02/SYNC-03
test dimensions in `tests/test_merge.py`.

Design decisions realized here (from 27-RESEARCH.md — recommended defaults, no CONTEXT.md):
- DD-5 (Idempotency): UUID primary key is the idempotency key; replay is a no-op via
  pre-select existing UUIDs → set-difference → bulk insert; `UNIQUE(device_id, seq)` is
  the loud DB backstop. (SYNC-02, RESEARCH Decision-note / Pattern 1)
- DD-3 (Derived state): reuse the existing ledger recompute (extract non-committing
  `recompute_derived`); cash balance is already a live SUM — nothing stored to recompute.
  Full recompute (not incremental) keeps the free invariant self-check. (SYNC-03, RESEARCH Pattern 4)

Scope fence: NO reference-row insert-if-new, NO server-wins resolution, NO Product.code
collision rename (all Plan 03). NO `synced_at` write (Phase 28). NO commit inside
apply_merge. NO route/HTTP/file code. NO Alembic migration.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| device-authored ledger rows → server DB | Merged operations/cash rows are attacker-influenceable; they are inserted verbatim but every derived figure is recomputed from the ledger, never trusted from the wire. |
| engine ↔ transaction policy | apply_merge is pure and never commits; atomicity is the caller's single transaction. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-27-04 | Tampering / Repudiation | forged ledger rows to inflate stock/cash | mitigate | Rows insert verbatim, but Product/Batch.quantity are recomputed via `recompute_derived` and cash is a live SUM — a forged cache value cannot survive; the invariant assert (ValueError) rejects an internally inconsistent batch; append-only triggers block any post-hoc edit; `created_by`/`author_id` origin is preserved (attribution not re-minted). |
| T-27-07 | Tampering | idempotency-key / seq forgery to duplicate or double-count | mitigate | Insert-if-new by UUID PK makes replay a no-op (0 inserts, identical snapshot); `UNIQUE(device_id, seq)` is the loud DB backstop against a forged seq collision. |
| T-27-06 | Integrity | half-applied batch (interrupted merge) | mitigate | `apply_merge` never commits mid-batch; the caller wraps the whole call in one transaction (all-or-nothing) — proven by test_bad_record_rolls_back (OFF-05 groundwork). |
| T-27-05 | Tampering | SQL injection via record fields | mitigate | Portable ORM/Core with bound parameters only (`insert(model)`, `select(...).in_(...)`); NO `sqlalchemy.dialects`, NO raw/f-stringed SQL (grep gate enforces). |
