<objective>
Prove the ONE merge engine is truly portable — its idempotency set-difference and its
Product.code collision resolution run identically on PostgreSQL (the server) and SQLite
(the client) — and wire that proof into CI. This is the SYNC-04 "one engine, never two
divergent implementations" guarantee made real on both dialects, and the SYNC-02/SYNC-05
portability backstop against an accidental dialect-specific construct.

Purpose: The engine deliberately avoids `on_conflict_do_nothing` (dialect-specific) in
favor of a portable pre-select set-difference; the only way to KNOW it stayed portable is
to re-run the idempotency + collision core on a real PostgreSQL instance. Phase 26 already
stood up a `postgres:17` CI service and the `test_pg_parity.py` skipif harness — this plan
reuses that exact pattern for a thin merge slice, so both transports (Phase 28/30) inherit
a portability-proven engine.

Output: `tests/test_merge_pg.py` (skipif-guarded PG slice reusing the Phase 26 harness)
and an added step in `.github/workflows/ci.yml`'s `pg-parity` job that runs it against the
live postgres:17 service.

Scope fence: NO new engine code (Plans 01–03 are the engine; this plan only proves it on
PG + wires CI). NO changes to `app/services/merge.py` or `app/services/ledger.py`. NO new
CI job — extend the existing `pg-parity` job. NO Alembic migration.
</objective>
