<objective>
Open the token-authenticated sync surface: a narrowly-scoped `auth_guard` bypass for the
`/api/sync/` prefix, a `require_device` Bearer dependency that guards that tree per-route,
and `POST /api/sync/push` as a thin caller of the already-complete Phase 27 merge engine.

Purpose: ROADMAP Success Criterion 2 — the server exposes sync endpoints that a client
authenticates to with a per-device token, and a request without a valid token is rejected
(SYNC-09).
Output: the guard bypass, the Bearer dependency, an in-process rate limiter, the push route
wired into `app/main.py`, a device-token test fixture, and `tests/test_sync_api.py`.

**This plan contains the single highest-consequence line of the phase.** The bypass prefix
must be exactly `/api/sync/`. A bare `/api/` would silently un-authenticate any future
`/api/...` route. Both negative directions are mandatory tests, not optional extras.

**No merge semantics are implemented here.** `parse_exchange` and `apply_merge` already
validate, deduplicate, resolve conflicts and recompute derived state. If any task in this
plan starts describing merge, idempotency or conflict rules, it is in the wrong phase.
</objective>

Handler: `def sync_push(request: Request, payload: bytes = Body(..., media_type="application/x-ndjson"), device: DeviceToken = Depends(require_device), session: Session = Depends(get_session))`
decorated `@router.post("/api/sync/push")`. Plain `def`, NOT `async def` — CLAUDE.md's
locked sync-session rule means FastAPI runs it in the threadpool; reading the body via a
`Body(...)` parameter instead of `await request.body()` is what keeps the handler synchronous.

Handler body, in this order:
1. Rate limit: `check_rate_limit(device.token_prefix)`; on False raise `HTTPException(429, RATE_LIMITED_ERROR)`.
2. Size cap: reject with `HTTPException(413, PAYLOAD_TOO_LARGE_ERROR)` when the
   `content-length` header exceeds `MAX_PUSH_BYTES`, and again when `len(payload)` exceeds
   it (a missing or lying `Content-Length` must not defeat the cap).
3. Decode `payload` as UTF-8 (`errors="strict"`); on `UnicodeDecodeError` raise
   `HTTPException(400, MALFORMED_BATCH_ERROR)`. Split on newlines.
4. `batch = parse_exchange(lines)` OUTSIDE the transaction — it validates before any DB
   touch by design. Catch `ValueError` and raise `HTTPException(400, MALFORMED_BATCH_ERROR)`.

5. `with session.begin(): report = apply_merge(session, batch, server_now=utcnow_iso())`.
   Never call `session.commit()` in the handler body — `apply_merge` deliberately does not
   commit so a poisoned record rolls the whole batch back.
6. Return a plain dict built from the `MergeReport` fields: `operations_inserted`,
   `operations_skipped`, `cash_inserted`, `cash_skipped`, `reference_inserted`,
   `reference_server_wins`, and `conflicts` as a list of `dataclasses.asdict(c)`.

Do NOT re-validate anything `parse_exchange` already checks, do NOT re-null `synced_at`
(the engine already forces it), and do NOT add any batch-id dedupe table — the origin-UUID
replay IS the idempotency mechanism (SYNC-02).

Wire it in `app/main.py`: add `sync` to the `from app.routes import (...)` block and add

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| internet → `/api/sync/*` | The app's first unauthenticated-by-cookie, internet-reachable POST surface |
| `auth_guard` bypass → HTML tree | An over-broad prefix here un-authenticates the entire browser application |
| client NDJSON payload → merge engine | Fully untrusted bytes, arbitrary size, arbitrary content |
| client-supplied `author_id`/`created_by` → ledger | Attribution is carried verbatim from the origin device |

## Accepted risk T-28-09 — client-supplied author attribution

**Recorded here in full so a later security audit evaluates the decision in place rather than re-raising it.**
This is the one threat in Phase 28 dispositioned `accept` on a security-relevant surface. Security enforcement
for this phase blocks on `high`; this threat is assessed **LOW** and the reasoning is below. Do not silently
re-rate it — if an auditor disagrees, the correct outcome is a Phase 29 design change, not a Phase 28 hotfix.

**The exposure.** `POST /api/sync/push` accepts NDJSON ledger records whose `author_id` and `created_by`
fields are carried into the database verbatim. A holder of a valid device token can therefore push an
operation attributed to any user — including one they are not and one that does not exist.

**Why it is accepted, not mitigated here:**

1. **Phase 28 does not own this contract.** Verbatim carriage of origin `author_id`/`created_by` is decision
   DD-6 of the Phase 27 merge engine, which is deliberately the ONE implementation shared by the online sync
   transport (Phase 29) and the offline self-uploading file (Phase 30). Re-attributing on the server inside
   this route would fork the engine's semantics for one of its two callers — precisely the divergence SYNC-04
   exists to prevent. The change belongs in the engine, in a phase that owns it.
2. **The attacker must already hold a valid device token.** There is no anonymous path: `require_device`
   rejects every request without one (T-28-02). So this is not an external attack — it is a privilege
   *misuse* by an already-trusted install.
3. **Tokens are admin-minted and individually revocable.** No self-service enrolment exists; an administrator
   creates each token at `/settings/devices` (Plan 05) and can revoke any one instantly. There is no shared
   or ambient credential to abuse.
4. **Attribution is not fully lost.** Every pushed row still carries the origin `device_id`, and the
   authenticating `DeviceToken` row independently records its own `device_id`, `user_id` and `last_used_at`.
   A forged `author_id` therefore leaves a mismatch that is visible after the fact rather than an untraceable
   write.
5. **Blast radius is a single-reseller deployment with 1-3 devices**, all operated by the same business, with
   no multi-tenant boundary and no financial authority attached to `author_id` — it drives display and
   reporting filters (RPT-01/USER-06), not permissions or money movement. Nothing authorises an action based
   on a ledger row's stored author.
6. **The append-only guarantee is unaffected.** A forged row can be inserted but never altered or deleted
   afterwards (Plan 01 keeps every immutable column, `author_id` and `created_by` included, under the trigger).
   The audit trail is therefore append-only and complete even when a value in it is untrue.

**Compensating controls already in this phase:** per-device token authentication (T-28-02), instant
revocation and a `last_used_at` staleness signal (T-28-01), TLS-only transport (T-28-05), and the untouched
append-only triggers (T-28-08).

**Revisit trigger — not deferred indefinitely.** Re-evaluate if any of these becomes true: a second
organisation or tenant shares a server; device tokens become self-service or long-lived shared credentials;
or `author_id` starts driving an authorisation or payout decision rather than a display filter. The concrete
tightening, when it is wanted, is server-side re-attribution from `token.user_id` inside the Phase 27 engine
with an explicit trusted-origin exemption for the offline path — a Phase 29/30 design decision.
