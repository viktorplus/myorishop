<objective>
Make the app deployable and safe on an internet-facing PostgreSQL server, and write the
runbook — without picking the operator's VPS provider, plan size or domain for them.

Purpose: SRV-04 says the central server HOSTS both UIs. The desktop and mobile trees already
ship (Plan 04 proves one app object serves both); what is missing is the hosting story, plus
two code-level landmines that only fire on a PostgreSQL deployment — the SQLite backup
routine running in `lifespan`, and a session cookie with no `Secure` flag on a public domain.
Output: a dialect-guarded startup backup, a `session_https_only` setting, and a `deploy/`
directory containing a systemd unit, a reverse-proxy config, a `pg_dump` backup timer and
`DEPLOY.md`.

**Locked scope boundary:** the VPS provider, plan size and domain name are an OPEN USER
DECISION and must NOT be chosen here. Every artifact is provider-agnostic and works on any
Ubuntu 24.04 box. Do NOT perform a live deploy to a specific host, do NOT register a domain,
and do NOT make any success criterion depend on a live server existing. All three ROADMAP
success criteria are provable locally and in CI.
</objective>

  <action>
**Guard 1 — the backup dialect gate.** In `app/services/backup.py::startup_backup()`, add an
explicit non-sqlite early return returning None when `engine.dialect.name != "sqlite"`. The
function currently resolves its engine lazily only after the file-missing and empty-DB
checks; move or mirror that resolution so the dialect can be inspected BEFORE any
SQLite-specific work happens.

Comment it with the reason, not just the rule: `create_backup` issues `VACUUM INTO`, which is
SQLite-only. Today a PostgreSQL deployment survives boot only because `settings.db_path`
happens to name a file that does not exist on the server. If that file ever exists — a stray
copy, a shared volume, a developer's `.env` — `create_backup` would run `VACUUM INTO` against
the PostgreSQL engine and raise inside `lifespan`, taking the whole server down on boot. The
dialect is the real condition; the missing file is an accident. Reference OQ-6 and Plan 06.

Do NOT remove the existing file-missing or empty-DB skip conditions — they still serve the
SQLite client. Smallest safe change: one added guard.

**Guard 2 — the session cookie Secure flag.** Add to `app/config.py::Settings`:
`session_https_only: bool = False` (pydantic-settings maps it to env var `SESSION_HTTPS_ONLY`).
Comment: the session cookie must carry `Secure` on an internet-facing HTTPS deployment
(ASVS V3), but the default MUST stay `False` or every local `run.bat` client and the entire
test suite would break — a `Secure` cookie is not sent over plain HTTP. It is set true in the
server's environment file only. Do NOT hardcode `True`.

In `app/main.py`, replace the hardcoded `https_only=False` on `SessionMiddleware` with
`https_only=config_settings.session_https_only`. Update the adjacent comment: `same_site=lax`
stays; `https_only` is now environment-driven — `False` for localhost, `True` on the VPS.
Change nothing else about the middleware.

**Tests** — append to `tests/test_backup.py`:

- `test_startup_backup_skips_non_sqlite_dialect` — build a non-sqlite engine via
  `create_engine("postgresql+psycopg://user:pw@localhost:5432/none")`. SQLAlchemy resolves the
  dialect WITHOUT connecting, so this test needs no PostgreSQL server and stays green in the
  default SQLite CI run. Pass it as `startup_backup(engine=...)` with `backup_on_startup`
  monkeypatched True and `settings.db_path` monkeypatched to a file that DOES exist (use
  `tmp_path`), so the accidental file-missing skip cannot be what saves it. Assert the return
  is None and that no file was written to the backup directory. This is the regression test
  that proves the GUARD — not the accident — is doing the work.
- `test_session_cookie_secure_flag_follows_setting` — assert the default `False` behaviour:
  a response that sets the session cookie carries no `Secure` attribute in its `set-cookie`
  header. If the flag is not observable after app construction (the middleware reads it at
  registration time), instead assert directly that `app/main.py` passes the setting through —
  e.g. that the constructed middleware's option equals `settings.session_https_only` — and
  state that limitation in the test docstring. Do not write a test that cannot fail.
