<objective>
Give the administrator a surface for the SYNC-09 token lifecycle: mint a token for a device,
see which tokens exist and when each was last used, and revoke a lost device instantly —
without ever storing or re-displaying a plaintext token.

Purpose: a token nobody can issue or revoke is not an access-control mechanism. Revocation
is the ONLY mitigation for a stolen token (there is no expiry, by decision), so the revoke
control is load-bearing security, not convenience.
Output: `/settings/devices` page + routes, a settings-index link, and tests.

Per the locked decision for this phase there is **no UI-SPEC**: this page reuses the existing
`/settings/users` patterns and the Phase 25 design system verbatim. Introduce NO new design
tokens, NO new CSS classes and NO new interaction model.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator-role session → admin token surface | A non-administrator must not mint or revoke device credentials |
| minted plaintext → rendered HTML | The only moment a device credential exists in cleartext outside the client |
| browser → mint/revoke POST | State-changing, cookie-authenticated, now internet-facing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-01 | Spoofing | Stolen device token with no expiry | mitigate | This page IS the mitigation: instant admin revocation on every token, plus a «Последнее использование» column so an unexpected or stale device is visible — Tasks 1-3. Residual risk accepted: revocation is manual and reactive |
| T-28-24 | Elevation of Privilege | An operator minting themselves a device token | mitigate | `include_router(devices.router, dependencies=[Depends(require_role("administrator"))])`, the same server-side boundary as `/settings/users`; handlers never re-check, so the gate cannot be forgotten per-route — Task 1; `test_operator_cannot_reach_devices_page` asserts 403 on ALL THREE routes — Task 3 |
| T-28-07 | Information Disclosure | Plaintext token persisting in session, logs, URL or a later render | mitigate | The plaintext travels only in the single creation response context — never into the session, a redirect target, a query string or a log line (grep-asserted); `test_plaintext_not_shown_on_reload` proves it is unrecoverable afterwards — Tasks 1-3 |
| T-28-25 | Information Disclosure | Stored credential material leaking through the UI | mitigate | Templates never render `token_hash` or `token_prefix` (grep-asserted) — Task 2; `test_stored_hash_never_rendered` — Task 3 |
| T-28-06 | Tampering | CSRF on the internet-facing mint/revoke POSTs | mitigate | Both POSTs go through the untouched `auth_guard` CSRF path — they are NOT under `/api/sync/` and therefore not bypassed; the `base.html` `hx-headers` chrome line supplies `X-CSRF-Token` and the create form also carries a hidden `csrf_token` for the non-HTMX fallback — Tasks 1-2 |
| T-28-26 | Tampering | XSS via an operator-supplied device label | mitigate | Jinja autoescape only; `|safe` grep-asserted at 0 in the rows partial, mirroring the T-05-18 precedent — Task 2 |
| T-28-18 | Repudiation | Destroying evidence by deleting a compromised token | mitigate | The UI exposes revoke only — no delete control exists in either template; revoked rows stay visible in the listing — Tasks 1-3 |
| T-28-SC | Tampering | Supply chain (npm/pip/cargo installs) | accept | Zero new packages; no JS beyond the already-vendored htmx 2.0.10, no CDN reference. If any task proposes a package, STOP and run the legitimacy gate first |
</threat_model>
