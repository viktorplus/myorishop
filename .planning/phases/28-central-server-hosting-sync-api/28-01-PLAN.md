<objective>
Relax the two append-only UPDATE triggers from "block every UPDATE" to "block any UPDATE
that changes an immutable column", so the `synced_at` sync cursor can be stamped on a
ledger row without reopening the ledger to tampering — on BOTH SQLite and PostgreSQL.

This is ROADMAP Success Criterion 3 for Phase 28 and the database prerequisite for the
Phase 29 client cursor (SYNC-01). It amends *how* the Phase 26 SRV-02 guarantee is
enforced without weakening it: DELETE stays unconditionally blocked and every immutable
column stays immutable.

Purpose: without this, a client that pushes successfully cannot record that it pushed —
`UPDATE operations SET synced_at = ...` is rejected by the current trigger.
Output: migration `0018`, the matching `app/db.py::APPEND_ONLY_TRIGGERS` edit, a new
SQLite test module, and new PostgreSQL parity cases in the existing pg-parity CI job.
</objective>

PostgreSQL: `CREATE TRIGGER <name> BEFORE UPDATE ON <table> FOR EACH ROW WHEN ( <guard> ) EXECUTE FUNCTION <table>_append_only()`

`operations` immutable columns (14, exactly these, in this order): `id`, `type`,

CRITICAL PG-only detail (Pitfall 1, reproduced by execution): `Operation.payload` is
`sa.JSON`, which maps to PostgreSQL's `json` type, which has NO equality operator. The
PostgreSQL guard term for that one column MUST be
`NEW.payload::text IS DISTINCT FROM OLD.payload::text`. Without the `::text` cast,
`CREATE TRIGGER` or every subsequent UPDATE fails with `operator does not exist: json = json`.
SQLite needs no cast (it stores JSON as TEXT). `cash_movements` has no JSON column.

with the dialect-correct grammar, then re-create them without any `WHEN` clause (the exact
byte-for-behaviour form frozen in `0001` and `0013`). Hold these in `_SQLITE_DOWNGRADE_DDL`
and `_PG_DOWNGRADE_DDL`. Again no `DROP FUNCTION`.
