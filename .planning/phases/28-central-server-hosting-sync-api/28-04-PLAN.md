<objective>
Complete the sync surface with `GET /api/sync/pull` — server-authoritative reference data,
paged by a timestamp cursor, serialized through the same Phase 27 NDJSON writer the client
already knows how to parse. Then prove ROADMAP Success Criterion 1 (SRV-04) with a test:
one app object serves both the desktop UI and the mobile UI.

Purpose: Phase 29's client sync pulls reference data down; this is the endpoint it calls.
SRV-04's hosting claim needs a standing assertion, not just a deployment doc.
Output: `app/services/sync.py`, the pull route, pull + SRV-04 tests.

**Locked semantic for Phase 29 (write it down here so it is not rediscovered):** the pull
endpoint returns reference data ONLY, never ledger rows, and the SERVER never writes
`synced_at`. On the server `synced_at` stays NULL forever; it means "this row has never been
pushed FROM here". Only the client stamps its own cursor, in Phase 29. The trigger relaxation
from Plan 01 is required on both engines by Success Criterion 3 regardless.

**Locked pagination contract for Phase 29 — the client MUST send BOTH cursor parts.** The
cursor is COMPOSITE: `(cursor_timestamp, id)`. The server returns two response headers,
`X-Sync-Next-Since` and `X-Sync-Next-After-Id`, and the client echoes them back as the
`since` and `after_id` query parameters on the next request, repeating until it receives
fewer than `limit` records. A client that sends only `since` and ignores `after_id` will
loop forever on any page where more rows share one timestamp than `limit` — which is the
documented common case, because a bulk edit writes many rows with an identical stamp.
These values travel as HTTP RESPONSE HEADERS, not inside the NDJSON header envelope: the
envelope emitted by `merge.serialize_exchange` has a fixed field set and this phase does
not modify the Phase 27 engine.
</objective>

`collect_reference_records(session, *, since: str | None, after_id: str | None, limit: int) -> PullPage`

**The cursor is COMPOSITE: `(cursor_column, id)`.** A single-column timestamp cursor cannot
terminate. `next_since` is the MAXIMUM cursor value on the page, and the comparison must
stay inclusive so no boundary row is dropped — so if more rows share one identical
timestamp than `limit`, the next request would compute `since == next_since` again and
return the identical page forever. That is not a corner case: RESEARCH cites "a bulk edit
writes many rows with an identical stamp" as the very reason the comparison is inclusive.
Tie-break on the primary key to guarantee forward progress.

- Clamp `limit` into `1..MAX_PULL_LIMIT`, defaulting to `DEFAULT_PULL_LIMIT` when None or <= 0.
- For each kind in `PULL_KINDS` order (FK-dependency order, so a client applying the file
  top-to-bottom never hits a missing parent), build
  `select(Model).order_by(cursor_col, Model.id).limit(remaining)`.
- Cursor predicate, expressed with portable ORM constructs:
  - when `since` is None → no `where` clause (first page);
  - when `since` is set and `after_id` is None → `where(cursor_col >= since)`. Inclusive, so
    a row sharing the boundary timestamp is never silently dropped; the client's reference
    upsert is idempotent (Phase 27), so the re-delivery costs nothing;
  - when BOTH are set → the row-value comparison `(cursor_col, id) > (since, after_id)`,
    written portably as
    `or_(cursor_col > since, and_(cursor_col == since, Model.id > after_id))`.
    Use this two-branch form, NOT SQLAlchemy's `tuple_()` row-value syntax — `tuple_` IN/
    comparison support is uneven across dialects and this project is portable-ORM-only.
  This is strictly greater on the COMPOSITE key while remaining inclusive on the timestamp
  alone, which is what makes paging both lossless and terminating.
- Include soft-deleted rows (do NOT filter `deleted_at IS NULL`) — a tombstone is exactly
  what the client needs in order to learn about a deletion; the Phase 27 upsert handles
  inline `deleted_at` tombstones already.
- Accumulate records across kinds up to the total `limit`, tracking `remaining`.
- Return a `PullPage` whose `next_since` and `next_after_id` are the cursor value and `id`
  of the LAST record on the page (not the maximum of the whole set — the last row in sort
  order IS the resume point). Both are None when the page is empty. The client re-requests
  with `since=next_since&after_id=next_after_id` until it receives fewer than `limit`
  records.
- Because `after_id` is per-kind ambiguous once the page spans kinds, resume from the last
  record's OWN kind: carry the kind forward too, or — simpler and sufficient here — make
  the `after_id` predicate apply only to the kind that the last record belonged to and let
  earlier kinds in `PULL_KINDS` order be re-scanned with the plain inclusive `>= since`
  predicate. Re-delivery is idempotent, so a small overlap is correct; a skipped row is not.
  Document whichever form is implemented in the function docstring.
- Use portable ORM constructs only: `select()`, `or_`, `and_`, and
  `getattr(Model, column_name)`. No `sqlalchemy.dialects`, no raw SQL, no `strftime`, no
  `func.now()`, no `tuple_()`.

`current_schema_version(session) -> str` — read the live Alembic revision via
`alembic.migration.MigrationContext.configure(session.connection()).get_current_revision()`
and return it, falling back to the empty string when it is None (the test fixtures build the
schema with `Base.metadata.create_all` and therefore have no `alembic_version` table).
`parse_exchange` accepts an empty `schema_version`, so the fallback is safe. Derive it — do
NOT hardcode a revision string; Phase 30's OFF-07 schema-version gate depends on this value
being truthful.
