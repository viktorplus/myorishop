Create `app/services/sync.py` ŌĆö pure functions, no HTTP, no file I/O, no `Response` objects,
mirroring the `app/services/merge.py` contract. Module docstring names SYNC-09/SRV-03 support

`PULL_KINDS: tuple[str, ...]` = `("warehouse", "product", "customer", "dictionary", "batch", "sale")`
ŌĆö deliberately the reference kinds only. Comment: `operation` and `cash_movement` are
EXCLUDED because the offline path is upload-only and SYNC-01 specifies pulling
server-authoritative REFERENCE data down. Derive the model for each kind from
`merge.KIND_TO_MODEL` rather than re-importing models, so the two modules cannot drift.

`CURSOR_COLUMN: dict[str, str]` mapping each kind to its cursor column name: `updated_at`
for warehouse, product, customer, dictionary and batch; **`created_at` for `sale`** ŌĆö `Sale`
has no `updated_at` and a uniform query would raise (Pitfall 8). Comment that line.

`DEFAULT_PULL_LIMIT = 500`, `MAX_PULL_LIMIT = 2000`.

`_row_to_record(kind: str, row) -> ExchangeRecord` ŌĆö build the data dict from
`merge.KIND_TO_FIELDS[kind]` (schema-derived, so a new column ships automatically) by reading
each attribute off the ORM row. Return `ExchangeRecord(kind=kind, data=data)`.

`collect_reference_records(session, *, since: str | None, limit: int) -> tuple[list[ExchangeRecord], str | None]`
- Clamp `limit` into `1..MAX_PULL_LIMIT`, defaulting to `DEFAULT_PULL_LIMIT` when None or <= 0.
- For each kind in `PULL_KINDS` order (FK-dependency order, so a client applying the file
  top-to-bottom never hits a missing parent), build
  `select(Model).order_by(cursor_col, Model.id).limit(remaining)` and, when `since` is not
  None, add `where(cursor_col >= since)`. **Use `>=`, never `>`** ŌĆö ISO-8601 text is
  second/microsecond granular and a bulk edit writes many rows with an identical stamp; a
  strict `>` silently and permanently drops every row sharing the page boundary. The
  receiving side's reference upsert is idempotent (Phase 27), so the overlap costs nothing.
  Order by the cursor column then `id` as a stable tiebreak.
- Include soft-deleted rows (do NOT filter `deleted_at IS NULL`) ŌĆö a tombstone is exactly
  what the client needs in order to learn about a deletion; the Phase 27 upsert handles
  inline `deleted_at` tombstones already.
- Accumulate records across kinds up to the total `limit`, tracking `remaining`.
- Return `(records, next_since)` where `next_since` is the maximum cursor value seen across
  the returned records, or None when nothing was returned. The client re-requests with
  `since=next_since` until it receives fewer than `limit` records.
- Use portable ORM constructs only: `select()` and `getattr(Model, column_name)`. No
  `sqlalchemy.dialects`, no raw SQL, no `strftime`, no `func.now()`.

`current_schema_version(session) -> str` ŌĆö read the live Alembic revision via
`alembic.migration.MigrationContext.configure(session.connection()).get_current_revision()`
and return it, falling back to the empty string when it is None (the test fixtures build the
schema with `Base.metadata.create_all` and therefore have no `alembic_version` table).
`parse_exchange` accepts an empty `schema_version`, so the fallback is safe. Derive it ŌĆö do
NOT hardcode a revision string; Phase 30's OFF-07 schema-version gate depends on this value
being truthful.
  </action>
