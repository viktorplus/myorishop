# Phase 14 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase type:** cross-cutting infrastructure + fix-and-consistency. This phase adds **zero new colors and zero new type sizes/weights** — it applies the design system already shipped in Phase 2 (`app/static/style.css`) to a new interaction shape (paged/filtered/sorted tables + quick-delete) across six list pages: products, warehouses, customers, dictionary, catalogs, history. A small number of new **structural CSS classes** are introduced (pagination bar, header-row filter cells) but every property inside them (color, spacing, font) is a citation of an existing token, not a new one.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — server-rendered Jinja2 + HTMX, no component framework, no build step (FastAPI/Jinja2/htmx per `CLAUDE.md`; not React/Next/Vite, so the shadcn gate does not apply — same finding as Phase 13's UI-SPEC) |
| Preset | not applicable |
| Component library | none — hand-written semantic HTML (`<table>`, `<select>`, `<input>`) + `app/static/style.css` utility classes |
| Icon library | none — text-only project (confirmed Phase 13); pagination controls use text labels/digits only, never chevron icon glyphs |
| Font | `system-ui, -apple-system, "Segoe UI", sans-serif` (`app/static/style.css` line 5) |

## Spacing Scale

Declared values (must be multiples of 4) — unchanged from the existing scale:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon/inline gaps (`label` margin-bottom) |
| sm | 8px | Compact spacing — pagination button gap, filter-row `<th>` padding |
| md | 16px | Default element spacing — `.filter-bar`/`.sort-bar` gap, pagination bar top margin |
| lg | 24px | Section padding, `.filter-bar` bottom margin |
| xl | 32px | Layout gaps (`.container` bottom padding) |
| 2xl | 48px | Major section breaks (`.empty-state` padding) |
| 3xl | 64px | Page-level spacing (unused here) |

Exceptions: none. The new pagination bar and header-row filter cells reuse `sm`(8)/`md`(16) exactly as `.filter-bar` and `.preset-bar` already do — no new spacing value is introduced.

---

## Typography

Unchanged — 4 sizes, 2 weights, no new role:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Display (h1) | 24px | 600 | 1.2 |
| Heading (h2, year headings in catalogs) | 20px | 600 | 1.2 |
| Body (table cells, buttons, filter/sort inputs) | 16px | 400 | 1.5 |
| Label / muted (`label`, `th`, `.muted`, page-of-N text) | 14px | 400 | 1.5 |

Pagination page-number buttons and "Страница X из Y" text MUST use **Label size (14px/400)** — matching the existing `th` font-size, since pagination sits directly under/near table chrome, not body content. The **current page indicator** uses the same 14px/400 but is visually distinguished by background tint, not by a third weight (see Color below) — never bold the current-page number.

## Color

Unchanged palette (`app/static/style.css`: "accent #2563eb, destructive #b91c1c"):

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#fafafa` (body background) | Page background |
| Secondary (30%) | `#ffffff` with `#d9d9d9` 1px border | Table surface, `th` background `#f0f0f0`, filter-row `<th>` inputs, pagination buttons (`.secondary`) |
| Accent (10%) | `#2563eb` (buttons/links/focus); tint `#e8effd` (selection/match) | See "Accent reserved for" below |
| Destructive | `#b91c1c` | `button.danger` (existing product/warehouse delete buttons — reused, not new), guard-blocked error text/`.error-block` |

Accent reserved for:
- Primary button background/border (unchanged — no new primary buttons this phase; pagination/sort controls are NOT primary actions, see below)
- Links (`a`)
- `:focus-visible` outline (2px solid `#2563eb`, offset 1px) — every new filter input, sort `<select>`, and pagination button inherits this automatically via the existing `input:focus, select:focus, button:focus-visible` rule; no new focus treatment needed
- **New reuse (this phase, no new color):** the current-page pagination indicator uses the existing selection tint `#e8effd` (already used by `mark`, `.batch-picker tr.selected-batch`, `.mobile-card.selected`) — semantically "this is the active/selected one", exactly the same meaning as those three existing usages

Explicitly NOT accent: pagination prev/next/page-number buttons and the "Сортировать по…" `<select>` are **secondary** controls (`class="secondary"` / plain `<select>`, white surface + `#d9d9d9` border) — they are navigation/refinement chrome, not the page's primary action. Do not style them as accent buttons.

## Phase-Specific Interaction Contracts

Five contracts restate `14-CONTEXT.md` decisions in prescriptive, copy-paste-ready form so planner/executor do not have to re-derive markup shape. All markup is illustrative HTMX/Jinja2 shape, not final variable names.

### Contract A — Page-number pagination (LIST-01, D-01/D-02/D-03)

One shared partial, e.g. `partials/pagination.html`, parameterized by `page` (0-based), `total_pages`, the swap target id, and the base `hx-get` URL (carrying forward any active filter/sort query params via `hx-include` or by re-serializing them into the URL server-side):

```html
<nav class="pagination" aria-label="Страницы">
  {% if page > 0 %}
  <button type="button" class="secondary"
          hx-get="{{ list_url }}?page={{ page - 1 }}{{ extra_qs }}"
          hx-target="#{{ rows_target_id }}" hx-swap="outerHTML" hx-push-url="true">‹ Пред.</button>
  {% endif %}
  {% for p in page_window %}
    {% if p == '…' %}
    <span class="muted">…</span>
    {% elif p == page %}
    <span class="current-page">{{ p + 1 }}</span>
    {% else %}
    <button type="button" class="secondary"
            hx-get="{{ list_url }}?page={{ p }}{{ extra_qs }}"
            hx-target="#{{ rows_target_id }}" hx-swap="outerHTML" hx-push-url="true">{{ p + 1 }}</button>
    {% endif %}
  {% endfor %}
  {% if page < total_pages - 1 %}
  <button type="button" class="secondary"
          hx-get="{{ list_url }}?page={{ page + 1 }}{{ extra_qs }}"
          hx-target="#{{ rows_target_id }}" hx-swap="outerHTML" hx-push-url="true">След. ›</button>
  {% endif %}
  <span class="muted">Страница {{ page + 1 }} из {{ total_pages }}</span>
</nav>
```

New CSS (structural only, all values existing tokens):
```css
.pagination { display: flex; gap: 8px; align-items: center; margin-top: 16px; flex-wrap: wrap; }
.pagination .current-page { padding: 8px; background: #e8effd; font-size: 14px; }
```

- **Page size: 20**, one Python constant reused by every list's query (D-03) — e.g. `LIST_PAGE_SIZE = 20` in a shared module, not six separate literals.
- Applies uniformly to products, warehouses, customers, dictionary, catalogs, AND history (D-02 retires `history_load_more.html`'s "Показать ещё" `<tfoot>` button in favor of this same partial — the total-count query this requires is a **new** addition to `app/services/operations.py`, which today only computes a `has_next` sentinel, not a total).
- Every list needs its own total-count query added alongside its existing/new paginated row query (an implementation task, not a design-contract concern).

### Contract B — Per-column header-row filters (LIST-02, D-04/D-05)

Filters live in a **second `<tr>` inside `<thead>`**, one filter control per filterable column, aligned directly under that column's label:

```html
<thead>
  <tr>
    <th>Код</th>
    <th>Название</th>
    <th>...</th>
  </tr>
  <tr class="filter-row">
    <th><input type="text" name="code" placeholder="Фильтр…"
               hx-get="{{ list_url }}" hx-trigger="input changed delay:300ms"
               hx-include="closest table" hx-target="#{{ rows_target_id }}"
               hx-swap="outerHTML" hx-push-url="true"></th>
    <th><input type="text" name="name" placeholder="Фильтр…"
               hx-get="{{ list_url }}" hx-trigger="input changed delay:300ms"
               hx-include="closest table" hx-target="#{{ rows_target_id }}"
               hx-swap="outerHTML" hx-push-url="true"></th>
    <th></th>
  </tr>
</thead>
```

New CSS (structural only, reuses existing input/select/th rules — do not restyle `input`/`select` themselves):
```css
.filter-row th { background: #ffffff; padding: 8px; }
.filter-row input, .filter-row select { width: 100%; box-sizing: border-box; }
```

- Debounce: `hx-trigger="input changed delay:300ms"` for text inputs, `hx-trigger="change"` for select filters — identical timing to the existing product/customer search box and `history_filters.html` (D-05, no new timing value).
- `hx-include="closest table"` (or equivalent) so every active column filter, the sort `<select>` (Contract C), and pagination state combine in one request — mirrors `history_filters.html`'s existing `hx-include="[name='product']"` pattern, generalized.
- A filter change always resets to page 0 (implicit — no `page` param sent), matching the existing `history_filters.html` comment convention.
- **Column selection per list** (which columns get a filter) is Claude's Discretion / planning's call per `14-CONTEXT.md` — candidates based on each list's existing search/lookup fields: products (code, name, category), customers (name, surname, consultant number), dictionary (code, name), warehouses (name, address, active/deleted — new select), catalogs (year — select, label), history (already has type + product selects — move them from `.filter-bar` into this header-row shape for consistency, per D-04's explicit rejection of the standalone filter-bar pattern).
- The existing standalone `q` search inputs on `/products` and `/customers` and the standalone `.filter-bar` on `/history` are **superseded** by this header-row pattern for consistency (D-04 explicitly rejects "a single search box above the table") — retiring them is in scope for this phase, not a parallel addition.

### Contract C — Sort dropdown (LIST-03, D-06/D-07)

One `<select name="sort">` positioned directly **above** the table (not inside `<thead>`), reusing the existing `.filter-bar` wrapper class verbatim (no new wrapper class needed):

```html
<div class="filter-bar">
  <div class="field">
    <label for="sort">Сортировать по</label>
    <select id="sort" name="sort"
            hx-get="{{ list_url }}" hx-trigger="change"
            hx-include="closest table, previous .filter-bar"
            hx-target="#{{ rows_target_id }}" hx-swap="outerHTML" hx-push-url="true">
      <option value="">{{ default_option_label }} (по умолчанию)</option>
      <option value="{{ key }}">{{ label }}</option>
      ...
    </select>
  </div>
</div>
```

- Default order per list (no `sort` param / empty value) matches CURRENT behavior exactly (D-07): products/customers by name, warehouses active-first, dictionary by code, history newest-first (`created_at desc, seq desc`), catalogs newest-first by year — changing none of these is itself a requirement, not just a fallback.
- Exact non-default sort options offered per list (e.g. "Название (Я→А)", "Остаток (по убыванию)") are Claude's Discretion — left to planning per `14-CONTEXT.md`.
- A sort change resets to page 0, same as a filter change.

### Contract D — Product quick delete (LIST-05, D-08/D-09/D-10)

Add a delete action to `partials/product_rows.html`'s existing `<td>Действия</td>` cell (currently only "Изменить"), reusing the exact confirm copy from `product_form.html`:

```html
<td>
  <a href="/products/{{ product.id }}/edit">Изменить</a>
  <button type="button" class="danger"
          hx-post="/products/{{ product.id }}/quick-delete"
          hx-confirm="Удалить товар „{{ product.name }}“? Он будет скрыт из каталога и поиска, история операций сохранится."
          hx-target="#product-rows" hx-swap="outerHTML">Удалить</button>
</td>
```

- **New guard (D-08):** if the product has stock on hand > 0, the POST is rejected and the re-rendered `#product-rows` includes an inline `.error` message next to the blocked row: `Нельзя удалить: на остатке {{ qty }} шт. Спишите остаток, чтобы удалить товар.` — no override path (hard block, unlike the warehouse last-active guard).
- On success, the row disappears entirely (D-10 — matches current full-page behavior, no visual change to the "gone" outcome, only the swap mechanism moves from `HX-Redirect` full-page reload to a partial `#product-rows` re-render, per `14-CONTEXT.md`'s Integration Points note — an implementation detail for planning, not a visual change).
- Button placement/order: "Изменить" first (existing), "Удалить" second — same left-to-right convention as `warehouse_rows.html`'s existing "Сохранить склад" → "Удалить"/"Восстановить" ordering.

### Contract E — Warehouse quick delete (LIST-04, D-11/D-12/D-13/D-14)

`warehouse_rows.html` already has a delete button with `hx-confirm` (D-13 — no change needed to that copy or mechanism). This phase adds a **new stock guard** that must pass BEFORE the existing last-active guard is evaluated:

1. **Stock guard (new, D-11, hard block, no override):** if the warehouse has any stock on hand, reject and render inline `.error`: `Нельзя удалить: на складе есть остаток. Переместите или спишите товар, чтобы удалить склад.` — no `confirm=1` override button, unlike step 2.
2. **Last-active guard (existing, D-12, unchanged mechanism):** only reached if step 1 passes. Behaves exactly as `soft_delete_warehouse` does today — renders the existing `warning_id == w.id` block with "Удалить всё равно" / "Отмена" (verbatim existing copy, existing markup, `error-block`/`form-actions` classes).
3. **On success (D-14, behavior change):** the deleted warehouse's row is removed from `#warehouse-rows` entirely — it must NOT remain visible muted with a "Восстановить" button, which is the CURRENT behavior for every other trigger of `soft_delete_warehouse`. This means `warehouse_rows.html`'s row-rendering loop needs a condition change: rows where `w.deleted_at` is set are no longer rendered in this list's `<tbody>` at all (the `{% if w.deleted_at %} class="muted"{% endif %}` branch and its sibling `{% else %}` restore-button branch go away for THIS list).

**Open question flagged, not resolved here** (per `14-CONTEXT.md`'s "Claude's Discretion" section): once deleted warehouses no longer appear inline in this list, is there ANY remaining path to restore one (e.g. a future "Показать удалённые" toggle), or is `restore_warehouse` effectively orphaned code after this phase? This is a product decision for planning, not a visual/interaction contract question — do not silently drop the restore capability without an explicit decision recorded in the plan.
