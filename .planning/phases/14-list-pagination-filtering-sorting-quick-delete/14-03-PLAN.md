<objective>
Turn `/dictionary`'s unbounded `list_entries(session) -> list[Dictionary]` (currently returns all 6,856 rows in one query, ordered by code, no filter, no pagination) into a filtered/sorted/paginated SQL query using LIMIT/OFFSET + a matching COUNT query — the only one of the six lists, besides history, large enough to need SQL-side (not Python-side) pagination — and add the header-row code/name filters plus a sort dropdown, consuming the `Dictionary.name_lc` shadow column and shared `pagination.py`/`pagination.html` built in Plan 14-01.

Purpose: dictionary is by far the largest list in the app (6,856 rows live today per RESEARCH.md's row-count query) — an unbounded `SELECT * ORDER BY code` with no LIMIT is the single most impactful unbounded-list problem this phase fixes.
Output: `list_entries()` returns a dict with `entries`/`total`/`total_pages` reflecting the active `code`/`name`/`sort`/`page` query; `/dictionary` renders one swappable `#dictionary-rows` block with header-row filters, a sort dropdown, and numbered pagination.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser query params (`page`, `sort`, `code`, `name`) -> `list_entries` | Untrusted input arrives via GET query string on every request, including bookmarked/hand-edited URLs, against the largest table in the app (6,856 rows). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-14-06 | Tampering | `sort` query param | mitigate | Resolved through the fixed `_SORT_MAP` dict via `.get(sort, Dictionary.code)`, never string-interpolated into `order_by()`. |
| T-14-07 | Tampering | `code`/`name` filter query params | mitigate | `code` filtered via `func.lower(Dictionary.code).contains(...)` (ASCII-only, safe); `name` filtered via `Dictionary.name_lc.contains(..., autoescape=True)` — both parameterized ORM `.where()` calls, never raw SQL string formatting. |
| T-14-08 | Tampering (confused deputy) | `page` query param | mitigate | Clamped server-side into `[0, total_pages - 1]` before use in `.offset()`. |
| T-14-SC-dict | Denial of Service | Unbounded `LIKE`-style `.contains()` scan on 6,856 rows per keystroke | accept | Debounced 300ms client-side trigger (existing app-wide convention) bounds request rate; SQLite handles a 6.8K-row indexed scan in low single-digit milliseconds locally — no new mitigation needed for a single local operator. |
</threat_model>
