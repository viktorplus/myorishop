<objective>
Replace `/history`'s "Показать ещё" load-more pagination with page-number pagination (D-02/D-03), move its existing type/product filters from the standalone `.filter-bar` into a per-column header-row filter (D-04), and add a "Сортировать по" dropdown (D-06/D-07) — while keeping the existing filter semantics, row markup, and RU copy completely unchanged.

Output: `history_view()` returns a real total-count-based page (not a `has_next` sentinel); `/history` renders one swappable `#history-rows` block containing the sort dropdown, the filter-row table, and the shared pagination partial; `history_load_more.html`, `history_response.html`, and `history_filters.html` are retired (deleted).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser query params (`page`, `sort`, `type`, `product`) -> `history_view` | Untrusted input arrives via GET query string on every request, including bookmarked/hand-edited URLs. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-14-03 | Tampering | `sort` query param | mitigate | Resolved through the fixed `_SORT_MAP` dict via `.get(sort, default)` — an unknown value silently falls back to the default order, never string-interpolated into `order_by()`. |
| T-14-04 | Tampering (confused deputy) | `page` query param | mitigate | Clamped server-side into `[0, total_pages - 1]` before use in `.offset()` — a bookmarked `?page=-1` or `?page=99999` can no longer produce a negative OFFSET or a permanently-empty page with no way back. |
| T-14-05 | Information Disclosure | `type`/`product` filter echoed into `&lt;option selected&gt;` | accept | Both already validated against `OPERATION_TYPES`/existing product ids before this phase; Jinja2 autoescape covers the render path regardless. |
</threat_model>
