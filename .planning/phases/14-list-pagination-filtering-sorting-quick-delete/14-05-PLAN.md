<objective>
Give `/warehouses` filter/sort/pagination for the first time (today it returns ALL rows, active and deleted, unconditionally — RESEARCH.md's D-09 "must never filter deleted_at" convention), add a NEW per-warehouse stock guard to `soft_delete_warehouse` that runs BEFORE the existing last-active-warehouse warn-then-confirm guard (D-11/D-12 — neither guard replaces the other), and change the default list view so a quick-deleted warehouse disappears entirely (D-14) — while resolving CONTEXT.md's explicitly-flagged open question (a restore path must still exist) via a `status` filter (`Активные` default / `Удалённые` / `Все склады`) that reuses the EXISTING, untouched `restore_warehouse`/`POST /warehouses/{id}/restore` machinery.

Purpose: this is the most behaviorally-invasive of the six list plans — it changes a DEFAULT query result shape (deleted rows no longer show by default) that four existing route handlers (add/update/delete/restore) all rely on, and stacks a brand-new hard-block guard in front of an existing soft-block guard on the same write path.
Output: `list_warehouses()` accepts `name`/`address`/`status`/`sort`/`page` and returns a paginated dict; `soft_delete_warehouse()` rejects any warehouse with stock on hand before it ever reaches the last-active check; `/warehouses` renders one swappable `#warehouse-rows` block with header-row filters (including the status/restore-path select), a sort dropdown, and numbered pagination.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser query params (`page`, `sort`, `name`, `address`, `status`) -> `list_warehouses` | Untrusted input via GET query string, including bookmarked/hand-edited URLs. |
| Browser POST -> `soft_delete_warehouse` (`confirm` form field) | The existing `confirm=1` override is a soft-guard bypass ONLY — it must never affect the new hard stock guard. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-14-14 | Tampering | `sort`/`status` query params | mitigate | Both resolved through fixed if/elif branches over a small closed value set, never string-interpolated into a query. An unrecognized `status` value falls through to the safe default (active-only). |
| T-14-15 | Tampering (confused deputy) | `page` query param | mitigate | Clamped inside `pagination.paginate()` into `[0, total_pages - 1]`. |
| T-14-16 | Elevation of Privilege | D-11 stock guard bypass via the existing `confirm=1` override | mitigate | The stock check is evaluated and returns BEFORE the `if not confirm:` block is ever reached — `confirm=1` can only skip the last-active check, never the stock check (D-12: both guards apply together, neither replaces the other). |
| T-14-17 | Information Disclosure | Blocked-delete error echoes `warehouse.name`/stock quantity | accept | Jinja2 autoescape covers the render path; `warehouse.name` is already-validated operator-entered text. |
</threat_model>
