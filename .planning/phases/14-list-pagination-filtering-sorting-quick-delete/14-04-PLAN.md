<objective>
Give `/products` its own dedicated filter/sort/page list function — separate from the existing `search_products`/`search_view`/`split_match` trio, which stay completely UNTOUCHED because `app/routes/mobile_search.py` and `app/routes/sales.py` (Phase 12 SAL-06 autocomplete) depend on them — and add a one-click "Удалить" quick-delete action to each product row, guarded by a NEW stock-on-hand check that does not exist anywhere in the codebase today (D-08).

Purpose: `list_products` today silently caps at 20 with zero filter/sort (D-26 shape), and the standalone `/products/search` route + search box duplicate what the new header-row filters do (Pitfall 6) — retiring it removes a second, now-redundant filtering mechanism. `soft_delete_product` today has NO stock guard at all — a product with batches/stock still referencing it can currently be hard-hidden from the catalog while stock records still point at it; quick-delete introduces the guard the LIST action needs without changing the edit page's existing (unchanged) delete behavior.
Output: `catalog.list_products_view()` (new) + `catalog.quick_delete_product()` (new); `GET /products` gains filter/sort/page query params and an is_hx dual response; `POST /products/{id}/quick-delete` (new route); `/products/search` (route only — NOT the underlying `search_products`/`search_view` functions) is deleted.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser query params (`page`, `sort`, `code`, `name`, `category`) -> `list_products_view` | Untrusted input via GET query string, including bookmarked/hand-edited URLs. |
| Browser POST -> `quick_delete_product` | Untrusted trigger for a destructive write; only a `product_id` path param, no body fields to validate. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-14-09 | Tampering | `sort` query param | mitigate | Resolved through the fixed `_SORT_MAP` dict, never string-interpolated into a sort key. |
| T-14-10 | Tampering (confused deputy) | `page` query param | mitigate | Clamped inside `pagination.paginate()` into `[0, total_pages - 1]`. |
| T-14-11 | Elevation of Privilege | D-08 stock guard bypass | mitigate | The guard is a hard block with NO `confirm=1`-style override param (unlike the warehouse last-active guard) — a product with `quantity > 0` can never be quick-deleted from the list regardless of any request parameter. |
| T-14-12 | Information Disclosure | Blocked-delete error echoes `product.name` (via `hx-confirm`) and `blocked_qty` | accept | Jinja2 autoescape covers the render path; `product.name` is already-validated operator-entered text, not new untrusted input. |
| T-14-13 | Tampering | Bypassing the browser `hx-confirm` dialog via a direct POST to `/products/{id}/quick-delete` | accept | Matches the existing accepted risk for every other `hx-confirm`-gated destructive action in this app (e.g. warehouse delete) — single local operator, no auth boundary, CLAUDE.md's explicit "no auth complexity needed in v1" constraint. |
</threat_model>
