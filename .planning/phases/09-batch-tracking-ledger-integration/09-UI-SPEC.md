# Phase 9 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (server-rendered Jinja2 + HTMX 2.0.10 vendored; plain CSS, no framework, no CDN) |
| Preset | not applicable |
| Component library | none — reuse existing `app/static/style.css` (scale established in Phase 2, carried through Phases 3–8) |
| Icon library | none — text labels only (offline-first, no icon dependency) |
| Font | system-ui stack (existing): `system-ui, -apple-system, "Segoe UI", sans-serif` |

**shadcn gate:** not applicable — stack is FastAPI + Jinja2 + HTMX (not React/Next.js/Vite); no `components.json` in the repo. Registry safety gate: not applicable.

**Locked carry-over from Phases 2–8 (source: 02/07/08-UI-SPEC.md, `app/static/style.css`):** every token below is IDENTICAL to prior phases. This phase adds NO new pages and NO new nav entries — it extends five existing forms (receipt, sale, write-off, correction, return) and the /history rows with two new partials (`batch_picker.html`, `receipt_batch_chooser.html`). All markup reuses existing classes verbatim: `.field`, `.form-actions`, `.error`, `.error-block`, `.muted`, `.num`, `table`/`th`/`td`, `button`/`button.secondary`/`button.danger`. UI language: Russian. Batch `comment` and `location` are untrusted stored input (like product names): Jinja autoescape only, never `|safe`.

**Exactly two new CSS rules are permitted this phase** (added to `style.css`, reusing existing token values only — no new colors, no new sizes):

```css
/* Batch picker (Phase 9): nested batch table under a form field / basket line. */
.batch-picker { margin: 8px 0 0; }                         /* sm gap instead of the default 16px table margin-top */
.batch-picker tr.selected-batch td { background: #e8effd; } /* selection highlight — existing mark/search tint, NOT a new color */
```

No other CSS changes. The radio inputs and `<input type="date">` are native controls; the existing global `:focus` outline rule already covers them.

## Spacing Scale

Declared values (must be multiples of 4). Identical to Phases 2–8 — already implemented in `style.css`.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Label→input gap; `.error` top margin; gap between a picker note and its table |
| sm | 8px | Table cell padding (8px 12px); button vertical padding; `.batch-picker` top margin (new rule above); gap between chooser radios |

Exceptions: 12px — table cell horizontal padding only (existing `th, td { padding: 8px 12px }`, carried from Phase 2; the nested `.batch-picker` table inherits it unchanged). Control heights stay 36px.

## Typography

Exactly 4 sizes, 2 weights — identical to Phases 2–8. No other values permitted. The nested batch-picker table uses the SAME table ramp as every other table (no smaller "dense" size is introduced).

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body (inputs, table cells incl. picker cells, radio labels, paragraphs) | 16px | 400 | 1.5 |
| Label (form labels, table headers incl. picker `th`, muted hints/notes) | 14px | 400 (labels/hints) / 600 (table `th`, `.error-block` headings via `<strong>`) | 1.5 |
| Heading (h2) | 20px | 600 | 1.2 — not used this phase |
| Display (h1, existing page titles) | 24px | 600 | 1.2 — unchanged, no new pages |

- Numeric picker columns («Цена», «Остаток») use existing `.num` (right-aligned, tabular-nums).
- Money renders through the existing `| cents` filter (comma decimal). Expiry dates display as `dd.mm.yyyy` (new small Jinja filter, e.g. `ru_date`, applied to the stored ISO `yyyy-mm-dd`); the `<input type="date">` value stays ISO.
- No italics, no uppercase, no color-coded text (unchanged convention).

## Interaction Contract

- **Selection is server-driven and always visible (discretion resolved):** picking a radio hits `/sales/batch-pick` (or the form-scoped equivalent), which re-renders the whole wrapper — radio checked, row `class="selected-batch"` (highlight `#e8effd`), hidden `batch_id[]` synced, fresh remaining quantity (RESEARCH Pattern 3). The picker table does NOT collapse after selection — collapsing would hide the LOT-02 attributes; the highlighted row keeps the choice visible and changeable (D-06 spirit).
- **Structural array alignment (Pitfall 2):** the hidden `batch_id[]` input is ALWAYS rendered in every basket line's wrapper row, empty until picked; «Удалить строку» removes BOTH `<tr>`s of its line. Never render the hidden input only inside a lookup response.
- **Auto-select single batch (D-06):** exactly one open batch → lookup response renders the picker pre-checked, row highlighted, hidden input pre-set, muted note «Партия выбрана автоматически — единственная», price filled by the same rules. Still a plain radio — one click changes nothing about how to re-pick.
- **Price fill precedence (D-05/D-14, Pitfall 4):** product has ≥1 open batch → `/sales/lookup` does NOT fill the card price; the batch pick is the sole fill source («Цена подставлена из партии — можно изменить.»). Picked batch has NULL price (legacy) → card `sale_cents` fallback with the existing card hint. The existing typed-value `oob-before-swap` guard applies unchanged — a batch pick never clobbers an operator-typed price.
- **422 / warn re-echo (D-04, Pitfall 3):** every re-render goes through the line-echo path carrying `batch_id[]`; the wrapper row re-renders the hidden input plus the selected-batch summary (same partial, selected-only mode, still highlighted). An unresolvable echoed batch_id renders as "no batch picked" with the «Выберите партию.» error path as backstop.
- **Warn-but-allow unchanged (D-09):** per-batch oversell/over-removal warnings are zero-write; the danger button re-POSTs the intact form with `hx-vals='{"confirm": "1"}'` via `form=` association; the secondary button is a client-side dismiss (`.remove()`), no server call. Same batch across multiple basket lines is aggregated before comparison.
- **Receipt chooser (D-01):** warehouse `<select>` change AND code lookup both refresh `#batch-chooser`; «Новая партия» reveals its fields via the hidden+disabled radio idiom (disabled inputs never submit). Note A1 (RESEARCH): when touching radio toggles, use single-colon `hx-on:change` for DOM events and verify the existing `correction_form.html` double-colon toggle in-browser.
- **Return flow (D-08):** no picker, no batch inputs — the origin batch is displayed read-only; the server resolves it (origin op's batch_id, legacy fallback).
- **Keyboard-first:** radios reachable by Tab, changeable by arrow keys (native); `<input type="date">` native keyboard entry; visible 2px accent focus outline everywhere (existing global rule); no focus traps, picker insertion never moves focus.
- **No new JS beyond vendored HTMX idioms already in the repo** (hx-get/hx-swap-oob/hx-vals/hx-on one-liners). No modals, no toasts.
- **Behavior boundary (flag for planner, D-12 consequence):** a product with zero open batches cannot be sold/written-off/corrected until a receipt creates a batch — the picker shows «Нет партий с остатком.» and submit fails with «Выберите партию.». The old "oversell from zero product stock" path no longer exists for batch-less products; this follows from D-03 (batches leave pickers at qty 0) + D-12 (batch required) and is not a UI defect.
- **Manual UAT gates:** (1) receipt with existing batches forces an explicit top-up/new choice; (2) sale of a 2-batch product shows 4 columns in D-07 order (earliest expiry first, «без срока» last) and the picked batch's price lands in the line; (3) single-batch product auto-selects with the locked muted note; (4) overselling one batch while another has stock triggers the per-batch warning, not the product-total one; (5) 422 on qty keeps the picked batch highlighted; (6) deleting a middle basket row keeps the other rows' picks attributed correctly; (7) /history shows «До внедрения партий» on pre-migration stock ops and nothing extra on price-change rows; (8) return of a legacy sale shows «Возврат в партию: Остаток до внедрения партий».
