<objective>
Make the goods-receipt flow the birth path for batches (D-01/D-02): a required warehouse select, an HTMX resolve-or-create chooser ("top up an existing batch" vs "new batch"), and the three new-batch fields (expiry LOT-03, location WH-02, comment LOT-04). `register_receipt` resolves or creates the `Batch` in the same transaction as the product, snapshots the sale price into `Batch.price_cents`, and threads `batch_id` into `record_operation`.

Purpose: Deliver LOT-01 stock creation with per-batch warehouse/expiry/price/location/comment, and give the pickers in Plans 03-04 real batches to select. This plan makes receipts batch-aware and updates its tests to supply the chooser inputs.
Output: A receipt form that forces an explicit batch choice, creates/tops-up batches, and defends against tampered/stale batch ids and zero-warehouse states.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| receipt form → register_receipt | `warehouse_id`, `batch_choice`, `expiry`, `location`, `comment` are all untrusted; a stale/crafted POST may name a foreign batch or a deleted warehouse |
| stored batch text → chooser HTML | `location`/`comment` are untrusted-at-rest, rendered back into HTML |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-04 | Tampering | receipt top-up `batch_choice` | mitigate | `register_receipt` re-validates `batch.product_id == product.id and batch.warehouse_id == warehouse_id` before writing (Pitfall 10); `batch_choice` allow-list ("new" or a resolvable id) |
| T-09-05 | Tampering / Input validation | receipt `warehouse_id` + `expiry` | mitigate | server-side active-warehouse re-check (zero-warehouse blocking state re-enforced on POST); `date.fromisoformat` validation of expiry (browser ISO is not trusted) |
| T-09-06 | Tampering (XSS) | location/comment in chooser HTML | mitigate | Jinja autoescape only; never `|safe` on stored batch text (repo rule) |
| T-09-SC | Tampering | npm/pip/cargo installs | accept | zero packages installed this phase (vacuous gate) |
</threat_model>
