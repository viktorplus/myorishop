<objective>
Fix Phase 9 UAT test 1, symptoms 1 and 2 (major). Symptom 1: on receipt-form load the batch chooser shows a single, already-checked «Новая партия» radio with no context — "непонятно к чему относится и уже выбрана". Symptom 2: entering an existing product's code does not autofill «Название».

Root causes already diagnosed (do NOT re-diagnose — see .planning/debug/receipt-batch-chooser-ux.md):
- Symptom 1: receipt_batch_chooser.html sets `new_selected = not batches`, and on every fresh GET /receipts/new `batches` is unconditionally `[]` (no code typed yet), so a lone pre-checked radio renders with no `<fieldset>/<legend>` to tie it to the top-up/new decision and no way to tell "nothing chosen yet" from "no open batches".
- Symptom 2: both the client `hx-on::before-swap` guard (receipt_form.html) and the server 204 guard (routes/receipts.py) use "is #name non-empty" as a proxy for "operator typed a name". A stale value left by an earlier autofill (nothing resets #name on a code change) or by native browser autofill (name_input.html has no `autocomplete="off"`) permanently suppresses further autofill.

Symptom 3 (batches need a name) is a schema change handled separately in plan 09-09.

IMPORTANT SCOPE NOTE — keep the new-product receipt flow working: an unknown/new product code returns 204 from /receipts/lookup and does NOT refresh the embedded chooser, so the embedded chooser's default must still allow creating a new batch (expiry/location/comment reachable). Therefore this plan does NOT remove the auto-select-new default for the empty-batches state; it removes the AMBIGUITY (fieldset/legend + state-adaptive helper text) and fixes name autofill via a client-side autofilled-vs-typed dirty flag. This is a deliberate, safe deviation from the gap's "do not auto-check on bare load" sub-bullet: fully removing the pre-check would break new-product receipts (the dominant case) because unknown codes never re-render the chooser.

Purpose: make the chooser self-explanatory and restore name autofill without regressing new-product receipts.
Output: clearer chooser markup, a small route flag, an autofilled-dirty-flag on the name field, and regression tests.
</objective>

Goal: distinguish a name the operator deliberately typed (must never be overwritten) from a stale/autofilled name (safe to refresh on a new code lookup), WITHOUT changing the server 204 contract or its test. Implement the dirty flag client-side.

In app/templates/partials/name_input.html (the single source for the name field):
  1. Add `autocomplete="off"` to the `<input id="name" name="name">` so the browser's native autofill can no longer plant a stale value (the missing item from the diagnosis).
  2. When the fragment renders as an autofill result, mark the input: add `{% if autofilled %}data-autofilled="true"{% endif %}` to the input tag (the `/receipts/lookup` path includes this fragment with `autofilled=True`, so its response carries the marker; the plain form include renders it without the marker).
  3. Add an `oninput` handler that clears the marker as soon as the operator edits the name, converting it to "typed/dirty": `oninput="this.removeAttribute('data-autofilled')"`.

In app/templates/partials/receipt_form.html:
  4. On the `#code` input, add a small handler that, whenever the operator edits the code, clears an AUTOFILLED name so the next debounced lookup sends an empty name and the server refills it (a typed name — no `data-autofilled` — is left untouched and the server 204 still protects it). Add to the `<input id="code">`: `hx-on:input="var n=document.getElementById('name'); if (n && n.dataset.autofilled==='true') { n.value=''; n.removeAttribute('data-autofilled'); }"`. This fires before the existing `hx-trigger="input changed delay:300ms"` request is sent, so `hx-include="[name='name']…"` serializes an empty name for a previously-autofilled field. Leave the existing `hx-on::before-swap` name guard and the code input's other htmx attributes unchanged.

Do NOT change routes/receipts.py's `if name.strip(): return Response(status_code=204)` guard — with the reset above, a stale autofilled name is cleared to empty before the request, so the server sees empty and refills; a genuinely typed name is preserved and the 204 still fires.
