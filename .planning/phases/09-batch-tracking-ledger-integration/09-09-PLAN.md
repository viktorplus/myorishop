<objective>
Fix Phase 9 UAT test 1, symptom 3 (major): batches have no name — "партия должна иметь название при создании имя и текущая дата". Give every new batch an auto-generated name «{product name} — {creation date}».

Root cause already diagnosed (do NOT re-diagnose — see .planning/debug/receipt-batch-chooser-ux.md): genuine schema + write-path gap. The Batch model has no name/label column, register_receipt never sets one, and the chooser identifies a batch only by expiry/price/qty. This is the one symptom in test 1 that touches the schema, so it is split from the UX fixes in plan 09-08.

RECOMMENDATION (made per the planning guidance's "stored column vs auto-generated" question): add a stored, nullable `batches.name` column and AUTO-GENERATE its value server-side at batch creation as «{product.name} — {dd.mm.yyyy}». No manual name input for the operator (the operator asked for an automatic name = product name + current date, not a field to type). A STORED column is chosen over a purely read-time derived label so the name is snapshotted at creation and survives later product renames — consistent with this project's snapshot/append-only philosophy. Existing and legacy batches keep NULL name and fall back to the current expiry/price description.

FROZEN ALEMBIC RULES (CLAUDE.md + migrations 0007/0008): use a NATIVE `op.add_column` on `batches` — SQLite supports ADD COLUMN natively. NEVER use an Alembic batch / move-and-copy rebuild here (that pattern is what would drop the append-only `operations_no_update`/`operations_no_delete` triggers — and it is unnecessary for a simple add-column on `batches`, which has no triggers anyway). Keep the migration portable (no SQLite-specific SQL) and free of app-module imports (WR-06 immutability rule).

Purpose: batches carry a human-readable, snapshotted name surfaced in the chooser.
Output: migration 0009, model column, auto-name write path, chooser label, tests.
</objective>
