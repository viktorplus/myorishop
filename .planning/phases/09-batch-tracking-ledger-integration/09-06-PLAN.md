<objective>
Fix the Phase 9 UAT blocker (tests 4 and 5): the sale-line inline batch picker submits an unusable form — a batch that was visibly picked is rejected on submit with «Выберите партию.», the «Выберите партию:» label appears twice, and the quantity input looks unlabeled.

Root cause is already diagnosed (do NOT re-diagnose — see .planning/debug/sale-batch-picker-not-submitted.md): partials/sale_lookup.html emits a bare `<td id="name-wrap">` as the FIRST top-level tag, then a LATER sibling `<tr id="batch-wrap-{row}" hx-swap-oob="outerHTML">`. htmx 2.0.10 picks its fragment-parsing context from that first `<td>`, so the later OOB `<tr>` is parsed as if already inside a row and the browser folds its content into the open basket line as a bogus `<td colspan="5">`. That leaves TWO `batch_id[]` hidden inputs per line (a stale empty one first in DOM order, the correctly-picked one second); FastAPI's positional zip pairs each `code[]` with the FIRST (empty) value, so register_sale rejects every line.

The fix is htmx's documented pattern for OOB table elements: wrap each `hx-swap-oob` table-row / table-cell fragment in a `<template>` tag (verified against https://htmx.org/attributes/hx-swap-oob/ — "Troublesome Tables and lists": a `<tr>`/`<td>` that cannot stand alone in the DOM must be enclosed in a `<template>` for an OOB swap). A `<template>` is a script-supporting element, valid inside `<tr>`/`<tbody>` parse contexts, so htmx preserves it and processes the OOB element inside it instead of the browser folding it into the open row.

Purpose: make the sale-with-batches flow usable — the single highest-value defect in this gap set.
Output: templated OOB fragments in two partials + a regression test that locks the fix in.
</objective>
