<objective>
Deliver the core sale slice: a multi-line basket (D-01) where the operator adds product lines (code → name/price auto-fill, quantity, editable sale price), then one «Оформить продажу» writes all lines as `sale` ops (qty_delta<0) + a `sales` header in ONE transaction (D-02/D-03). Prices are snapshotted per line — entered price → `unit_price_cents` (SAL-02/D-10), card cost frozen → `unit_cost_cents` (SAL-05/D-11), NULL cost allowed / empty price rejected (D-12). Walk-in only in this plan (customer picker is 04-05; the header stores `customer_id=None`).

Purpose: this is the phase's central user-facing capability (SAL-01/02/05) and the fast-entry core value. Oversell handling is refined in 04-03.
Output: sales service, sales routes, five sale templates + nav wiring, `.basket` CSS.
</objective>

| Browser form → POST /sales | Untrusted line codes/qty/price + customer_id cross here |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-4-04 | Tampering | line price/qty parsing in register_sale | mitigate | Integer cents via `to_cents` (ROUND_HALF_UP, rejects inf/nan); empty/garbage price → RU error, 0 writes; qty strict positive int (D-12) |
| T-4-05 | Tampering | POST /sales form binding | mitigate | Explicit `Form(...)` params only (code[]/qty[]/price[]/customer_id/confirm); service reads named fields, ignores extras |
| T-4-01 | Tampering/Info | product names in recent_sales / sale templates | mitigate | Jinja autoescape ON, never `|safe` on stored names |
| T-4-02 | Tampering | GET /sales/lookup code query | mitigate | SQLAlchemy ORM parametrized `Product.code == code`; no raw SQL |
| T-4-SC | Tampering | package installs | accept | No package installs this phase — not applicable |
</threat_model>
