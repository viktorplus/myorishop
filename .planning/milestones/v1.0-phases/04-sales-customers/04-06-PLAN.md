<objective>
Close the Phase 4 UAT gap: the debounced per-line code lookup on the sales basket (`GET /sales/lookup`) never autofills «Название», in every case, not just when a price is typed first.

Root cause (fully diagnosed in `.planning/debug/sale-lookup-name-not-filling.md`): `sale_lookup()` in `app/routes/sales.py` declares bare query params `code`/`name`/`price`, but the basket row inputs (`app/templates/partials/sale_row.html`) are array-form fields named `code[]`/`name[]`/`price[]`, and the lookup's `hx-include="closest tr"` serializes the request using those bracketed names. FastAPI never binds `code[]=...` to a param named `code`, so `code` always arrives as `""`, `lookup_prefill()` returns `None`, and the route always answers `204` — htmx performs no swap, and «Название» is never populated, unconditionally (SAL-01).

The existing test `test_web_sale_lookup_prefills_price` sends bare `code`/`name`/`price` params — which do NOT reflect what the real browser sends — so it passed despite the route being broken for real usage. This plan fixes the route to bind the bracketed keys AND fixes the test to send the request shape the DOM actually produces, plus adds a regression test for the exact UAT reproduction (price typed before code).

Purpose: Restore SAL-01's code-lookup autofill on the sales basket, which is a core interaction of registering a sale by product code.
Output: `GET /sales/lookup` correctly binds `code[]`/`name[]`/`price[]` query keys; `tests/test_sales.py` covers both the fill-when-empty and no-clobber-when-typed paths using the real bracketed request shape.
</objective>
