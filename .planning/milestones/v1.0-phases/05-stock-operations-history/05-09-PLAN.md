<objective>
Close the one remaining blocking gap from `05-VERIFICATION.md` (new CR-01, requirement OPS-04): the "Показать ещё" pagination control on `/history` is destroyed by the first filter-select interaction on any page, because `app/templates/partials/history_rows.html:44`'s `<tr id="load-more" hx-swap-oob="true">` lives INSIDE `<tbody id="history-tbody">` — the exact element both type/product `<select>`s in `app/templates/partials/history_filters.html` target with htmx's default (unset, therefore `innerHTML`) swap. Per htmx's documented oob-before-main-swap ordering: the oob-extracted `#load-more` row is swapped back into its current DOM position inside `#history-tbody` FIRST, then the main swap immediately runs `#history-tbody.innerHTML = <new rows>`, wiping every child of `#history-tbody` including the row the oob step just placed. After the first filter interaction on a page, `#load-more` no longer exists anywhere in the DOM for the rest of that page session — `has_next` can remain `True` server-side with no control left to reach page 2+ of a filtered view. The same nesting mistake also causes the "Показать ещё" button's own `beforeend` click to reposition the control between row batches instead of leaving it at the true end (05-REVIEW.md's WR-01, now folded into this fix since it shares the identical root cause).

This is the fourth gap-closure round for Phase 5. Plans 05-01 through 05-08 are untouched by this plan and stay in scope as-is: 05-08 already closed a DIFFERENT, now-fully-resolved defect (the old CR-01 chrome-decision bug, reconfirmed still fixed in `05-VERIFICATION.md`'s third pass). Do NOT re-open OPS-01/02/03 — this plan touches OPS-04 (`/history`) only. Explicitly OUT of scope (per `05-VERIFICATION.md`'s carried-over Warning/Info items — do not touch): `/corrections` missing a persistent nav entry, the product filter dropdown only listing active products, and the various Info-level items in `05-REVIEW.md` (IN-01 through IN-08).

Root-cause fix (per `05-VERIFICATION.md`'s "Fix scope" and `05-REVIEW.md`'s CR-01 recommendation): move the persistent `#load-more` control OUT of `<tbody id="history-tbody">` into its own `<tfoot>` row — a DOM sibling, never touched by the tbody's innerHTML or beforeend swaps — while keeping the existing `hx-swap-oob="true"` mechanism to update that now-isolated row in place after every filter change or pagination click.

Purpose: Make `/history`'s pagination reliably reachable beyond the first 50 matching operations under every filter combination, so the operator can browse the complete operation trail (OPS-04) and ROADMAP.md Success Criterion 4 holds for datasets that grow past one page — the exact kind of growth this ledger-based app is designed to accumulate over time.
Output: `partials/history_rows.html` reduced to data rows only; a new standalone `partials/history_load_more.html` control rendered in a `<tfoot>`; a new `partials/history_response.html` combining both for HX responses; `pages/history.html` and `app/routes/history.py` updated to match; a regression test proving the fix against a >50-row filtered result set.
</objective>
