<objective>
Close the one remaining blocking gap from `05-VERIFICATION.md` (CR-01, requirement OPS-04): `/history`'s chrome-decision branch in `app/routes/history.py` currently reads `if is_hx or is_filtered:`, which incorrectly serves the chrome-less rows-only partial to ANY plain (non-htmx) top-level navigation that carries a `type`/`product` filter param — exactly what happens when an operator reloads, bookmarks, or shares a URL that htmx's `hx-push-url` just wrote into the address bar after picking a filter. A real browser drops the resulting bare `<tr>` markup per HTML5 tree-construction rules (a `<tr>` outside a table context in the "in body" insertion mode is a parse error and the token is discarded), rendering an essentially blank, unnavigable page. This was independently reproduced in `05-VERIFICATION.md` via TestClient (`GET /history?type=writeoff`, no `HX-Request` header → 200, but `<html`/`<table`/`<nav` all absent from the body).

This is the third and final gap-closure round for Phase 5. Plans 05-01 through 05-07 are untouched by this plan. Two prior gap-closure rounds already landed (05-06: OPS-01 nav link; 05-07: CR-02/CR-03/WR-03 session-rollback reliability fixes) — both reconfirmed green in `05-VERIFICATION.md`'s re-verification. Only CR-01 remains blocking.

Explicitly OUT of scope for this plan (per `05-VERIFICATION.md`'s "Non-blocking items" list — do not touch): WR-01 (pagination control stranding), WR-02 (`/corrections` missing persistent nav entry), WR-04 (product filter dropdown only lists active products), IN-01/02/03/05/06/07 from `05-REVIEW.md`, and the REQUIREMENTS.md traceability table's stale "In Progress" wording.

Purpose: Make `/history` reliably deliver the full page chrome (nav + filter bar + table) to every plain top-level navigation, filtered or not, so an operator can always browse the full operation history (OPS-04) — including after a reload, a bookmark, or sharing a filtered link — while genuine htmx-driven filter/pagination interactions keep receiving the lightweight rows-only partial they need.
Output: `app/routes/history.py`'s chrome decision keyed solely on the real `HX-Request` header; a new regression test in `tests/test_history.py` proving the fix and guarding against future regression.
</objective>
