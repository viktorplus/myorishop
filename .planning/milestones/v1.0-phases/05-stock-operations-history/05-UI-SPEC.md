# Phase 5 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (server-rendered Jinja2 + HTMX 2.0.10 vendored; plain CSS, no framework, no CDN) |
| Preset | not applicable |
| Component library | none — reuse existing `app/static/style.css` (spec scale established in Phase 2, carried through Phases 3–4) |
| Icon library | none — text labels only (offline-first, no icon dependency) |
| Font | system-ui stack (existing): `system-ui, -apple-system, "Segoe UI", sans-serif` |

**shadcn gate:** not applicable — stack is FastAPI + Jinja2 + HTMX (not React/Next.js/Vite); no `components.json`. Registry safety gate: not applicable.

**Locked carry-over from Phases 2–4 (source: 02/03/04-UI-SPEC.md, `app/static/style.css`):** every token below is the SAME as Phases 2–4 — this phase adds screens (write-off form, return-from-sale flow, correction form, `/history` audit page), NOT new tokens. All styling lives in `app/static/style.css`; new CSS rules are added ONLY where existing classes cannot express a layout, and stay strictly on the scale/palette below. UI language: Russian (`base.html lang="ru"`); stored operation `type`/`reason_code` values stay Latin (D-02). Jinja autoescape ON — product names, notes, and `created_by` render untrusted in the history table and forms; **never `|safe`** (T-3-02 / T-4-01 precedent). Money is integer cents rendered via the `cents` filter; timestamps via the `local_dt` filter.

## Spacing Scale

Declared values (must be multiples of 4). Identical to Phases 2–4 — already implemented in `style.css`.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Gap between label and input; error/hint text top margin; current-qty hint gap |
| sm | 8px | Gaps inside form rows; table cell padding (8px 12px); button vertical padding; nav padding; filter-bar inner gap |
| md | 16px | Default gap between form fields (`.field`); button horizontal padding; error/warning-block padding; gap between mode toggle and inputs |
| lg | 24px | Gap between page sections (history filters → table; correction form → recent history); `.form-actions` top margin |
| xl | 32px | Page top padding under nav (existing `.container`/nav margins) |
| 2xl | 48px | Empty-state block vertical padding (`.empty-state`) |
| 3xl | 64px | not used this phase |

Exceptions: 12px — dense table cell horizontal padding only (existing `th, td { padding: 8px 12px }`; multiple of 4, carried from Phase 2). Control heights derived from scale: inputs/buttons/selects 36px (8px vertical padding + 16px text + borders). The `/history` filter `<select>` controls reuse the same 36px control height.

## Typography

Exactly 4 sizes, 2 weights — identical to Phases 2–4. No other values permitted.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body (inputs, selects, table cells, paragraphs) | 16px | 400 | 1.5 |
| Label (form labels, filter labels, table headers, muted meta: dates/hints/«(необязательно)»/current-qty) | 14px | 400 (labels/hints) / 600 (table `th`) | 1.5 |
| Heading (h2 section titles: «История операций», «Последние операции», «Возврат») | 20px | 600 | 1.2 |
| Display (h1 page titles: «Списание», «Корректировка остатка», «История операций») | 24px | 600 | 1.2 |

- Weights: 400 (regular) and 600 (semibold) only.
- Numeric columns (signed quantity, unit price, unit cost): 16px, 400, right-aligned, `font-variant-numeric: tabular-nums` (existing `.num` class).
- Signed quantity in the history table renders with an explicit sign (`+3`, `−2`) — no color coding (semantic budget stays at one; sign is the signal, not color).
- No italics, no uppercase-tracking treatments.

## Color

Identical to Phases 2–4 — no new colors introduced. Semantic color budget stays at exactly one (destructive).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#fafafa` | Page background (existing); text `#222` |
| Secondary (30%) | `#ffffff` | Tables, form surfaces, nav bar, filter bar; borders `#d9d9d9`; table header background `#f0f0f0` |
| Accent (10%) | `#2563eb` | See reserved list below; hover/active shade `#1d4ed8` |
| Destructive | `#b91c1c` | Inline validation errors, error blocks (bg `#fdecec`), the write-off oversell warning block, and the «Списать всё равно» confirm button (`button.danger`, existing) |

Accent reserved for: primary submit buttons («Списать», «Оформить возврат», «Сохранить корректировку»), the per-sale-line «Вернуть» action (rendered as a neutral text link, NOT accent-filled — see below), text links (nav links; active nav link = accent + 600 weight), input/select `:focus` outline (2px, existing rule), and the `<mark>` match highlight if a product filter uses search. Nothing else. Neutral/secondary actions («Отмена», «Вернуться к списку», the correction mode toggle, «Показать ещё») use the neutral `button.secondary` style (white, `#d9d9d9` border, `#222` text) or a plain text link.

**The «Вернуть» line action is a neutral text link,** not an accent-filled button: it appears once per sale line in recent-sales and purchase-history tables, and filling every row with an accent button would break the 10% accent budget. It uses the default `a` link treatment (accent text, no fill).

**Write-off oversell warning uses destructive, not a new "warning" color:** listing a write-off that would drive stock below zero is a blocking, must-confirm condition — it reuses the existing `.error-block` treatment (`#b91c1c` on `#fdecec`) and the existing `button.danger` for «Списать всё равно». This keeps the semantic budget at exactly one and needs no CSS additions (mirrors Phase 4 oversell, D-04 / SAL-04).

Muted text (`.muted`, `#6b7280`): operation timestamps, «(необязательно)» markers, field hints, the current-quantity hint on the correction form, the empty-payload «—» dash in history cells, empty-state body text, `created_by` when it is the default operator. Tint of the dominant neutral, not a fifth color role.

Success feedback: neutral — plain `#222` text on the page surface («Списание сохранено…», «Возврат оформлен…», «Корректировка сохранена…»), no green (Phase 2–4 convention).

## Interaction Contract

- **Write-off save-and-next loop (mirrors receipt D-02, core value — fast entry):** `hx-post="/writeoff"` targeting the form partial (`hx-target="#writeoff-form-wrap" hx-swap="outerHTML"`). Success → 200 with a fresh empty form + neutral success line + `hx-swap-oob` recent-operations refresh; validation failure → 422 with the form re-rendered, entered values preserved, inline errors (422 swap already whitelisted in `base.html` htmx-config). Focus: explicit `hx-on::htmx:load` on the fresh form calling `document.getElementById('code').focus()` — do NOT rely on `autofocus` inside swapped content (Pitfall 6). Keep `autofocus` on Код for initial page load.
- **Write-off code lookup (reuses receipt/dictionary 204 pattern):** `hx-get="/writeoff/lookup"` on the Код input, `hx-trigger="input changed delay:300ms"`, `hx-sync="this:replace"`, `hx-include` the name field. Product/dictionary match → fragment filling Название; nothing / operator already typed → 204 no swap. Swap-time guard (`hx-on::htmx:before-swap` → `shouldSwap = false` when name already typed) replicated from `receipt_form.html` — a lookup response never clobbers typed input (Pitfall 7).
- **Write-off oversell two-step (reuses Phase 4 SAL-04 / D-04):** save runs a server-side stock check BEFORE any write. If the write-off drives stock below zero and `confirm != "1"`, return `writeoff_oversell.html` with ZERO writes. «Списать всё равно» re-POSTs the same form via `hx-post="/writeoff" hx-vals='{"confirm":"1"}'` (form association, no re-serialization). On `confirm=1` the service skips the block and writes; stock may go negative (a later correction fixes counts, D-04). «Вернуться к форме» dismisses the warning and restores the editable form (client-side swap back, no write).
- **Return launch (D-05):** the «Вернуть» link on each `sale` line uses `hx-get="/returns?sale_id={id}&product_id={pid}"` targeting an in-place slot (or navigates to a small return view), rendering `return_form.html` with the returnable remaining computed server-side (aggregate prior `return` ops for `sale_id`+`product_id`, D-08). The quantity input defaults to and is capped at `remaining`.
- **Return submit (D-06/D-07/D-08):** `hx-post="/returns"` with hidden `sale_id` + `product_id`; the service copies `unit_price_cents`/`unit_cost_cents` from the original sale line snapshot (never current card prices — preserves SAL-05) and writes a `return` op with `qty_delta > 0`. Returned qty > remaining → 422 with «Нельзя вернуть больше, чем доступно ({remaining}).», zero writes. Success → neutral «Возврат оформлен…» + `hx-swap-oob` refresh of the originating recent-sales / purchase-history list so the remaining-returnable count updates.
- **Correction mode toggle (D-09/D-11):** two radio buttons switch which quantity input is active; default = «Пересчёт (фактический остаток)» (D-11). Toggling is pure client-side visibility (`hx-on::change` toggling a hidden attribute, or both inputs present with the inactive one disabled so only the active value posts). The current cached quantity is shown beside the counted input («Текущий остаток: {qty}») so the operator sees what they adjust from (D-09).
- **Correction lookup (reuses receipt pattern):** `hx-get="/corrections/lookup"` on the Код input, 300ms debounce, `hx-sync="this:replace"`; product match → fragment filling Название + the current-quantity hint; 204 otherwise. Typed-value guard as above.
- **Correction submit (D-10/D-12):** `hx-post="/corrections"` targeting the form partial. Counted mode → server computes `qty_delta = counted − current`; delta mode → writes the entered delta. Zero net delta → 422 «Остаток не изменился — нечего сохранять.», zero writes (D-10). Always a `correction` op via `record_operation` (single write path, one commit) — never a direct `products.quantity` edit. This endpoint supersedes `POST /ops` (D-12); the old route is migrated, not left as a second correction path.
- **History filters (D-14):** the «Тип операции» and «Товар» `<select>` controls use `hx-get="/history"`, `hx-trigger="change"`, `hx-include` both selects, `hx-target` the results table body, `hx-push-url="true"` so the filtered view is linkable/back-navigable. Filtering resets pagination to page 1. Date-range filtering is explicitly NOT in this phase (deferred to Phase 6, D-14).
- **History pagination (D-15):** «Показать ещё» button uses `hx-get="/history?page={next}&...filters"`, `hx-target` the tbody, `hx-swap="beforeend"`, appending the next 50 rows and re-rendering (or removing) itself via oob when the last page is reached. The ledger is never loaded unbounded.
- **Keyboard-first (core value: fast entry):** every form submits on Enter; tab order = visual order (write-off: Код → Название → Количество → Причина → Примечание → «Списать»); visible 2px accent focus outline on all interactive elements incl. `<select>` and radios — never `outline:none` without replacement.
- **Buttons during request:** `hx-disabled-elt` on submit/confirm buttons; existing `button:disabled { opacity:.6 }`.
- **Money display:** unit price / unit cost in history via the `cents`/`format_cents` filter (comma decimal separator). No money INPUTS this phase (write-off has none; return copies frozen snapshot; correction is quantity-only). Quantity inputs `type="text" inputmode="numeric"`; correction delta accepts a leading «−».
- **Signed quantity:** the history «Кол-во» column renders `op.qty_delta` with an explicit sign; positive returns/receipts/corrections show `+`, negative sales/write-offs show `−`, right-aligned `.num`. No color, no arrows.
- **No client-side JS beyond vendored HTMX.** No modals, no toasts; htmx 2.x `hx-on::` double-colon syntax only. Native `hx-confirm` available if a destructive action is added (not required this phase).
- **Manual UAT gates:** after a write-off save, focus lands on «Код» without a click; the lookup never overwrites a typed name; a below-zero write-off shows the warning with zero committed ops until «Списать всё равно»; a return copies the FROZEN sale-line price/cost (not the current card price) and cannot exceed the returnable remaining; a correction is recorded as a `correction` op (append-only), never a direct `products.quantity` edit, and a zero net delta is rejected; `/history` renders newest-first with type/product filters and «Показать ещё» without loading the whole ledger.
