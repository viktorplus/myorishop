## Phase Goal

**As a** shop operator, **I want to** maintain product cards (code, name, category, three prices) and find any product in seconds by code or name, **so that** daily receipt and sale entry stays fast and prices are never lost.

*(Same assembled story as Plan 02-01 — ROADMAP Goal line is outcome prose; flagged to orchestrator.)*

<objective>
Vertical slice 2: operator can EDIT a product card, see its PRICE HISTORY, and soft-DELETE/RESTORE it. Completes CAT-01 (edit path) and delivers CAT-04 (price changes kept as history) end-to-end: every price change becomes an immutable price_change ledger op rendered on the card.

Purpose: after this plan the operator can correct prices without losing the old values — the CAT-04 promise — and retire products safely (D-20).
Output: catalog service update/delete/restore/history functions, four new /products/{id} routes, edit mode of product_form.html with «История цен», price_history partial, «Изменить» column in the product table.

**Plan decision PD-3 (from RESEARCH Finding 5, accepted tradeoff):** multiple changed price fields in one save emit one record_operation call per field; each call commits separately (payload shape is singular per D-28). The FIRST record_operation commit also persists all staged product-row mutations. Do NOT refactor record_operation to defer commits.

**Plan decision PD-4:** delete/restore buttons are htmx POSTs; routes answer 200 with an `HX-Redirect` response header (core htmx header, present in vendored 2.0.10 — verified by grep in Task 3) so the browser navigates after the native hx-confirm dialog. Soft-delete/restore themselves write no ledger op (RESEARCH assumption A3 — CONTEXT lists ops only for create/edit/price).
</objective>
