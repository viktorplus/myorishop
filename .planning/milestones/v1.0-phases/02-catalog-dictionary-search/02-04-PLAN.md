## Phase Goal

**As a** shop operator, **I want to** maintain product cards (code, name, category, three prices) and find any product in seconds by code or name, **so that** daily receipt and sale entry stays fast and prices are never lost.

*(Same assembled story as Plan 02-01 — ROADMAP Goal line is outcome prose; flagged to orchestrator.)*

<objective>
Vertical slice 4: the reference dictionary. Operator maintains code→name pairs at /dictionary, and the product form auto-fills the name when a known code is typed — CAT-02 complete. Runs in wave 3 in parallel with Plan 02-03 (zero file overlap).

Purpose: the operator pre-loads Oriflame codes once and then enters products (and, in Phase 3, receipts) at maximum speed — names appear by themselves.
Output: dictionary service + routes + /dictionary page with inline add/edit, GET /dictionary/lookup autofill endpoint (204 pattern), name_input partial shared by the product form, «Справочник» nav link.

**Plan decision PD-5 (route placement):** the lookup endpoint lives in app/routes/dictionary.py at `GET /dictionary/lookup` (RESEARCH sketched it under /products/lookup). Reason: file-ownership — Plan 02-03 owns app/routes/products.py in the same wave; the dictionary router owning all dictionary reads keeps wave 3 conflict-free. Same thin-route semantics.

**Plan decision PD-6 (single source for the name field):** product_form.html includes partials/name_input.html for the name field in ALL modes, so the static form markup and the HTMX-swapped fragment can never drift apart. Autofill attributes stay on the code input in both create and edit modes — the name-non-empty → 204 guard makes edit mode safe (Pitfall 5).

**Note (STATE.md blocker, RESEARCH Open Question 1):** there is no verified bulk source for the Oriflame dictionary; manual row-by-row entry with paste-friendly inputs IS the locked scope (Excel import explicitly deferred). The optional "paste many lines" textarea from RESEARCH is NOT planned — flagged by the researcher as optional, not required by CAT-02.
