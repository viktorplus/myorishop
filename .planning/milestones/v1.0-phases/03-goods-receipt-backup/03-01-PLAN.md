## Phase Goal

**As a** shop operator, **I want to** register goods receipts by product code — name auto-filled, prices captured, every receipt an immutable ledger entry — while the database backs itself up automatically with a proven restore path, **so that** stock on the shelf is always correct and a day of data entry can never be lost.

*(Assembled from the ROADMAP Phase 3 Goal line, which is outcome prose — same convention as Phase 2 plans; flagged to orchestrator.)*

<objective>
Vertical slice 1 of 3: the receipt entry loop. Operator opens /receipts/new, types code + name + quantity (+ optional prices), hits «Сохранить приход» — stock goes up through the ledger, the form clears for the next item, and the last 10 receipts are visible below. RCP-01 complete except the lookup pre-fill and card price sync (Plan 03-02).

Purpose: fast repeat entry is the core value — one box of goods = a rapid save-and-next loop with minimal clicks (D-01, D-02).
Output: receipts service (single-transaction write composing Phase 1–2 machinery), receipts routes, receipt form page + partials, recent-receipts partial with hx-swap-oob, «Приход» nav link.

### Decision Traceability (from 03-CONTEXT.md)

| ID | Locked decision | This plan |
|----|-----------------|-----------|
| D-01 | One receipt entry = one product line = one ledger `receipt` op, qty_delta > 0 | implements |
| D-02 | /receipts/new form; after save the form clears and focus returns to code | implements |
| D-04 | Recent receipts visible on the receipts page (last N partial) | implements (last 10, partial under the form — RESEARCH discretion) |
| D-05 | Unknown code auto-creates the product (+ product_created op) in the same transaction | implements |
| D-06 | Receipt op snapshots unit_cost_cents and prices (payload carries catalog price) | implements |
| D-03, D-07 | Lookup pre-fill / card price sync | Plan 03-02 (wave 2) |

**Plan decision PD-8 (prices optional — RESEARCH Open Question 2 / A3):** quantity is required (strict positive integer); the three prices are optional. Empty price string → NULL in the receipt op columns/payload. Empty prices NEVER modify the product card (card sync itself lands in 03-02). Matches CAT-01 optionality and keeps fast entry fast.

**Note (RESEARCH Open Question 1 / PD-9 preview):** for an EXISTING product the name field is informational — a typed name never renames the product (renames go through /products/{id}/edit). This plan's service simply ignores the name for existing products; 03-02 keeps that contract.
</objective>
