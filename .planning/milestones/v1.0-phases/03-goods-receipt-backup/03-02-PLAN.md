## Phase Goal

**As a** shop operator, **I want to** register goods receipts by product code — name auto-filled, prices captured, every receipt an immutable ledger entry — while the database backs itself up automatically with a proven restore path, **so that** stock on the shelf is always correct and a day of data entry can never be lost.

*(Assembled from the ROADMAP Phase 3 Goal line, which is outcome prose — same convention as Phase 2 plans; flagged to orchestrator.)*

<objective>
Vertical slice 2 of 3: the receipt form learns the catalog. Typing a code pre-fills the name (dictionary) or name + prices (existing product card), and saving a receipt for an existing product updates the card prices with full CAT-04 history. RCP-02 complete; D-03 and D-07 complete. Runs in wave 2 in parallel with Plan 03-03 (zero file overlap).

Purpose: the operator types a code and the rest appears by itself — the last manual friction in the fast-entry loop disappears, and the card always reflects the latest intake prices without losing history.
Output: GET /receipts/lookup endpoint, lookup_prefill service read, receipt_lookup fragment (name + per-field oob price fills), price sync in register_receipt, parametrized hint in name_input.html.

### Decision Traceability (from 03-CONTEXT.md)

| ID | Locked decision | This plan |
|----|-----------------|-----------|
| D-03 | Name auto-fills via existing HTMX lookup pattern (204, ~300ms debounce); existing product also pre-fills current name/prices | implements |
| D-07 | Entered prices update the product card via price_change ops in the same transaction (CAT-04 reuse) | implements |

**Plan decision PD-7 (endpoint placement):** the receipt lookup lives at `GET /receipts/lookup` in app/routes/receipts.py (RESEARCH Pattern 2 recommendation) — the dictionary router keeps its own /dictionary/lookup untouched; file ownership stays clean for wave-2 parallelism.

**Plan decision PD-9 (name is read-only for existing products — RESEARCH Open Question 1):** a name typed over the pre-fill never renames an existing product and writes no product_edited op; renames go through /products/{id}/edit. Prevents surprise renames from autofill races.

**Plan decision PD-10 (per-field oob fill instead of one wrapper fragment):** UI-SPEC locks the field order Код → Название → Количество → prices, so a single wrapper around name+prices would have to swallow «Количество» and risk clobbering it mid-entry. Instead the lookup response main-swaps #name-wrap (same as Phase 2) and carries hx-swap-oob fragments ONLY for price fields that arrived empty in the request — rendered from the same receipt_price_inputs.html partial the form uses (single-source rule PD-6).
</objective>
