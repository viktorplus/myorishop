## Accomplishments

## Decisions Made
- `Dictionary.rubric` filter is an exact `==` match rather than a substring `.contains()` (unlike the code/name filters) — `RUBRICS` is a closed, fixed vocabulary, so partial matches would be misleading.
- The category `<select>` on `/dictionary` was placed in the table's `filter-row` (matching the existing code/name filter idiom), which only renders when `entries` is non-empty. This is consistent with the plan's explicit action instructions and with how the code/name filters already behave — the plan's own test-naming spec didn't call out seeding an entry, so the test was written to seed one for a deterministic, non-flaky assertion.

## Deviations from Plan

None - plan executed exactly as written. One test refinement (seeding an entry in `test_web_dictionary_renders_category_select` before asserting the select renders) was made during the GREEN step to keep the assertion deterministic, matching the pre-existing architecture where filter-row inputs only render alongside table rows — not a behavior change, just a test-correctness fix within the TDD cycle.
